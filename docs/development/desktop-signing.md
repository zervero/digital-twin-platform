# Desktop Code-Signing & Updater Secrets

> Operator handoff for V3.2 (Track H) desktop release
> pipeline. Covers the three secret sets the
> `desktop-build.yml` workflow needs, where to get them,
> how to store them in GitHub Actions, and the
> rotation / maintenance story.

## What ships in V3.2

The Tauri desktop app ships three platform builds plus an
in-app auto-update channel:

| Platform | Artifact | Code-signed | Notarized |
| --- | --- | --- | --- |
| macOS (universal2) | `.dmg` | yes (Developer ID) | yes (Apple notarization) |
| Windows | `.msi` | yes (Authenticode) | n/a |
| Linux | `.AppImage` + `.deb` + `.rpm` | no (V3.2.x follow-up adds GPG) | n/a |

The in-app updater downloads `latest.json` + a `.sig`
file from the GitHub Release for the new version, then
verifies the signature against the public key committed
to `apps/desktop/src-tauri/tauri.conf.json`. The signing
side of that flow uses an ed25519 (minisign-format)
keypair generated by `tauri signer generate`.

## The three secret sets

### macOS (used by T4)

Required secrets in the repo's GitHub Actions
`Settings -> Secrets and variables -> Actions`:

| Secret | Source | Format |
| --- | --- | --- |
| `APPLE_CERTIFICATE` | Apple Developer portal: Certificates -> "+" -> Developer ID Application, export as `.p12` | base64-encoded `.p12` file content |
| `APPLE_CERTIFICATE_PASSWORD` | Password you set when exporting the `.p12` | plain string |
| `APPLE_ID` | Apple Developer account email | plain string |
| `APPLE_PASSWORD` | App-specific password (appleid.apple.com -> Sign-In and Security -> App-Specific Passwords) | plain string |
| `APPLE_TEAM_ID` | Apple Developer portal -> Membership -> Team ID | plain string (10 chars) |

The `APPLE_*` secrets are consumed by `tauri-action`
during the `macos` job's "Build + sign + notarize"
step. If any are missing, the job fails fast with a
clear error - V3.2 does not have an "unsigned macOS"
fallback because unsigned `.dmg` artifacts trigger
Gatekeeper warnings that would block end users.

### Windows (used by T5)

| Secret | Source | Format |
| --- | --- | --- |
| `WINDOWS_CERTIFICATE` | Code-signing cert from a public CA (Sectigo, DigiCert), exported as `.pfx` | base64-encoded `.pfx` file content |
| `WINDOWS_CERTIFICATE_PASSWORD` | Password you set when exporting the `.pfx` | plain string |

`tauri-action` calls `signtool` (preinstalled on
`windows-latest` runners) using these secrets. If the
secrets are missing, the `.msi` is built unsigned - this
is intentionally allowed so a contributor can run a
dev release without a code-signing cert, with the caveat
that SmartScreen will warn end users on first launch.

### Updater keypair (used by T4-T6)

| Secret / file | Source | Format |
| --- | --- | --- |
| `apps/desktop/src-tauri/tauri.conf.json` `plugins.updater.pubkey` | `tauri signer generate` (this section) | base64-encoded public key (committed to repo) |
| `TAURI_SIGNING_PRIVATE_KEY` | the `.key` file produced by `tauri signer generate` | entire `.key` file content, as a single string |
| `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` | password you set when running `tauri signer generate` (omit the secret if you generated without a password) | plain string |

Generation, run once on the operator's machine:

```bash
# From the repo root or apps/desktop, with the
# @tauri-apps/cli workspace package installed.
pnpm --filter @dt/desktop exec tauri signer generate \
  -w ~/.tauri/dtp.key -p <your-password>
```

Flags:

- `-w <path>` writes the private key to a file. If
  omitted, the private key is printed to stdout and the
  public key only is written to `<path>.pub`.
- `-p <password>` sets the password. **Strongly
  recommended** - the CLI warns when run without one.
  The macOS-only keychain lookup can silently shadow
  the explicit empty-password case (see Known quirks
  below), so always set a password in production.
- `--ci` skips interactive prompts. Useful in scripted
  regeneration, not for the operator one-time setup.

After generation:

1. Commit the public key: copy the contents of
   `~/.tauri/dtp.key.pub` (the entire base64 string,
   comment + key data) into
   `apps/desktop/src-tauri/tauri.conf.json`
   `plugins.updater.pubkey`. V3.2 T1 ships a
   placeholder string (`REPLACE_WITH_OUTPUT_OF_TAURI_SIGNER_GENERATE`)
   for local CI compile; T3 fills it with a dev
   keypair generated by the contributor. **The operator
   must regenerate with a real password before the
   first tagged release** - the T3 keypair is for
   local compile + smoke only.
2. Store the private key as
   `TAURI_SIGNING_PRIVATE_KEY` in GH Actions secrets.
   Paste the entire content of `~/.tauri/dtp.key` (a
   single base64 line) - `tauri-action` parses it
   directly.
3. Store the password as
   `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`. If the key has
   no password, omit this secret and either leave the
   env var unset or set it to an empty string.

The repo's `.gitignore` excludes `*.key` and
`*.key.pub` so an operator who runs `generate` inside
the repo cannot accidentally commit the private key.

## Per-release flow

Nothing changes between releases. The three secret sets
are long-lived; `tauri-action` reads them on each
tag-triggered run.

## Rotation procedure (sketch; V3.2.x follow-up)

**V3.2 does not ship a rotation story.** The following is
the shape of the follow-up - not the V3.2 contract.

1. Generate a new keypair (Section "Updater keypair").
2. Edit `tauri.conf.json` `plugins.updater.pubkey` to
   be a JSON array of two entries: the old public key
   first, the new public key second. This requires a
   Tauri-side change to accept an array of pubkeys for
   one release cycle.
3. Cut a release with the dual-pubkey `tauri.conf.json`.
4. Wait one release cycle (typically 2-4 weeks) for
   active clients to update to the dual-pubkey version.
5. Cut another release with the new pubkey only.
6. Rotate the `TAURI_SIGNING_PRIVATE_KEY` GH secret.

The macOS and Windows code-signing certs follow their
own CA-managed renewal timelines; rotation is "renew
the cert, replace the GH secret, next release picks it
up".

## Known quirks

- **macOS keychain lookup can shadow no-password keys.**
  On macOS, `tauri signer sign` without an explicit
  `-p` flag tries to read the password from the system
  keychain. For a key generated with `--ci` (no
  password), this returns `Device not configured (os
  error 6)`. Workaround: pass `-p ""` explicitly, or
  always generate with a password (preferred).
- **`.key.pub` is the entire pubkey, not just the
  base64 body.** The minisign-format pubkey file is a
  single base64 blob that encodes both the "untrusted
  comment" header and the actual key data. Paste the
  entire file content into `tauri.conf.json`
  `plugins.updater.pubkey` - do not strip the header.
- **`signer verify` is not a CLI subcommand.** Tauri
  CLI 2.x exposes `signer generate` and `signer sign`
  only; signature verification is performed by the
  runtime in the user's installed app, not by the CLI.

## What's not in V3.2

- Automatic key rotation (sketched above).
- Linux GPG signing for `.AppImage` / `.deb` / `.rpm`.
  Trust relies on GitHub Release provenance.
- Staged rollouts or release channels - V3.2 ships one
  `latest` channel only.
- Mobile code signing (iOS App Store, Google Play).
  Mobile is V4+ per the dev spec.
- Notarization auto-retry. If Apple notarization times
  out, the operator reruns the workflow manually.
- Delta updates. V3.2 ships full bundles per release.
