# Internationalization (`@dt/i18n`)

The `@dt/app-shell` chrome used to ship with a mixed
Chinese / English UI: `设备` next to `Install`,
`加载中…` next to `Loading installed plugins…`. V3.5
(Track K, ADR 0018) ships a self-hosted i18n package that
ends the mix.

## What `@dt/i18n` is

A self-hosted, key-based localization layer for
`@dt/app-shell`. Two locales ship: `en` (default) and
`zh-CN`. Locale choice is `system` / `en` / `zh-CN`. The
choice persists in `localStorage` under `dt.locale.v1`.

The package does **not** depend on `vue-i18n`. Decision
record and alternatives considered are in
[`docs/adr/0018-v3.5-i18n.md`](../adr/0018-v3.5-i18n.md).

```ts
import { useI18n } from '@dt/i18n';

const { t, locale, setLocale } = useI18n();
t('device.title');                                 // -> "Devices" or "设备"
t('marketplace.installHint', { permission: 'plugin:publish' });
```

## How to add a new key

1. Pick the namespace (`common`, `device`, `auth`,
   `marketplace`, `scene`). Add a new key to
   `packages/i18n/src/locales/en/<ns>.json`.
2. Mirror the same key in
   `packages/i18n/src/locales/zh-CN/<ns>.json`. The
   dictionary-completeness test fails until both files
   match.
3. Use the key in the component:

   ```vue
   <script setup lang="ts">
   import { useI18n } from '@dt/i18n';
   const { t } = useI18n();
   </script>

   <template>
     <button>{{ t('marketplace.install') }}</button>
   </template>
   ```

4. If the key needs interpolation, use `{name}` placeholders
   and pass them via the second argument:

   ```ts
   t('marketplace.installHint', { permission: 'plugin:publish' });
   // en: "Publishing requires the plugin:publish permission."
   // zh-CN: "发布插件需要 plugin:publish 权限。"
   ```

   Unknown placeholders are left as `{name}` in the
   rendered string — visible to the eye during development
   rather than silently empty.

5. Run `pnpm --filter @dt/i18n test`. The completeness test
   covers every leaf key in both directions.

## Pluralization

The package does not bundle an ICU pluralizer. For
singular / plural pairs, declare two keys:

```jsonc
// en
"marketplace": {
  "version": "version",
  "versions": "versions"
}
// zh-CN
"marketplace": {
  "version": "版本",
  "versions": "版本"
}
```

In the template, switch on the count:

```vue
{{ count }} {{ count === 1 ? t('marketplace.version') : t('marketplace.versions') }}
```

This pattern is intentional: Chinese does not inflect
plurals, so the two-key shape is the minimum that keeps
the Chinese copy grammatical. Promoting to ICU is a V3.5.x
follow-up — see ADR 0018 "Revisit when".

## Do not translate

Some strings stay literal in every locale. Codifying this
list keeps future contributors from over-translating.

| String | Why literal |
| --- | --- |
| `Digital Twin` (brand mark) | Trademark / product name. |
| `4.4.0` (version chip) | Build-time version. Not UI copy. |
| `plugin:publish`, `plugin:install`, `plugin:read` | API contract permission identifiers. They appear inside `<code>` blocks; translating them would break API parity for an operator reading logs. |
| `live` / `connecting` / `offline` (TopToolbar status) | Status enum values that also appear in `useDeviceStream`'s connection state machine and in the realtime stream event names. The toolbar chip mirrors the underlying value so the user can correlate UI to API. |
| `active` (marketplace version badge) | Mirrors the `InstallRecord.active` boolean. The active badge shows this to match the install API response shape; translating it would diverge from the contract. |
| `hello-plugin`, `1.0.0` (input placeholders) | Teaching examples — replacing with translations would break the example for non-English readers. |

If you find yourself wanting to translate one of these,
the right answer is usually a new i18n key alongside the
literal, not a translation of the literal.

## Dictionary-completeness test

`packages/i18n/src/__tests__/dictionary-completeness.test.ts`
walks every leaf key in `en` and asserts the same key
resolves to a non-empty string in `zh-CN`, and vice versa.
The test is the gate that catches the original "page mixes
Chinese and English" symptom at CI time. When the test
fails, the message names the missing key(s) and tells you
which side is missing it. Add the missing translation, no
other code change required.

The test runs on every `pnpm test` invocation and on the
pre-push lefthook gate.

## Locale persistence

The store persists the **choice** (`system` / `en` /
`zh-CN`), not the resolved locale, in `localStorage` under
`dt.locale.v1`. The `v1` suffix lets us rename the storage
shape later (e.g. when adding typed JSON) without
colliding with an old value.

The **resolved** locale is computed from `choice` plus
`navigator.language` when `choice === 'system'`. The
fallback chain is:

```
localStorage('dt.locale.v1')  →  navigator.language  →  'en'
```

If `localStorage` is missing (SSR, tests without
happy-dom) or `navigator` is missing (rare in modern
environments), the store defaults to `en`.
