# ADR 0019 - V3.5 Closure (Track K: Internationalization + CORS Unblocker)

## Status

Accepted (V3.5 implementation complete). Records the
moment V3.5's 8 implementation tasks land plus the
T8.1 CORS unblocker that surfaced during T8
verification, so the V3 roadmap (ADR 0012) has a
single auditable artifact to point at when asking
"what was V3.5?" and the post-V3 era (V4 +) has a
clean handoff.

## Context

ADR 0012 defined the V3 roadmap: 5 tracks (F real auth,
G production platform, H Tauri releases, I multi-tenant,
J plugin marketplace) across 5 releases (V3.0 -> V3.4).
Each of those 5 shipped and is captured in a closure
ADR (0013 through 0017). V3.5 is a **sixth V3-era
track** added after the V3.4 closure because the v4-prep
visual refresh (V4.4.0) made the V3-era English-only
copy gap visible at the same time as the new toolbar
design.

Track K is small in surface area but high in
operator-impact: every user-facing string in the
chrome is now either localized or explicitly marked
"do not translate", and the operator can flip the UI
language at runtime via a segmented control in the
top toolbar.

Track K layers on top:

- A new `@dt/i18n` package in `packages/i18n/` that
  exports a Pinia `useLocaleStore` and a `useI18n()`
  composable. The package is pure types + a tiny
  reactive store + key-based dictionary lookup; it
  does not depend on `vue-i18n`. The decision to
  self-host rather than adopt `vue-i18n` is in
  [ADR 0018](./0018-v3.5-i18n.md) (the "zero-build
  library surface" precedent from V4-prep). Two
  locales ship in V3.5: `en` (default) and `zh-CN`.
  Locale resolution is `localStorage > navigator.language
  > 'en'`, persisted under `dt.locale.v1`.
- 5 namespaces × 2 locales (`common`, `device`,
  `auth`, `marketplace`, `scene`), stored as JSON in
  `packages/i18n/src/locales/{en,zh-CN}/`. The
  `dictionary-completeness.test.ts` test walks every
  leaf key in both directions and fails CI if the two
  dictionaries drift, so asymmetry is caught at PR
  time rather than at runtime when a key is missing.
- A segmented `System / EN / 中文` switcher in
  `TopToolbar.vue`. Switching is reactive (no page
  reload) and persists across reloads via the
  `localStorage` write. The "System" option follows
  the browser's `navigator.language` at boot time and
  re-resolves if the user later changes their OS
  language setting.
- An explicit "do not translate" list, codified in
  [ADR 0018](./0018-v3.5-i18n.md) and
  [docs/development/i18n.md](../development/i18n.md):
  the brand `Digital Twin`, the version chip, every
  permission identifier (`plugin:publish`, `scene:read`,
  etc.), every status enum mirror (`live` / `connecting` /
  `offline`, `active`), and every input example
  placeholder (`hello-plugin`, `1.0.0`). Pluralization
  uses a two-key pattern (e.g. `marketplace.version` /
  `marketplace.versions`); Chinese reuses the same
  word in both branches because Chinese does not
  inflect plurals.
- Migration of the 18 user-visible strings across
  `AppShell`, `TopToolbar`, `LoginButton`, `DevicePanel`,
  `MarketplacePanel`, and `SceneViewport` to `t('xxx')`
  lookups. The `forbidden-template-grep.test.ts` test
  greps every SFC in `packages/app-shell/src/components/`
  for bare user-visible text and fails if a hardcoded
  string appears outside the dictionary.
- The `apps/web` + `apps/desktop` entries both call
  `createI18n(app)` at boot. The desktop Tauri shell
  picks up the same locale state as web because both
  share the same web bundle inside the WebView, so a
  single wire-up point covers both surfaces.

## CORS unblocker (T8.1, scope addition)

While exercising the V3.5 work in the actual browser
(Vite at `http://localhost:5173` reaching the BFF at
`http://localhost:3001`), every fetch failed with:

  Access to fetch at 'http://localhost:3001/api/scene'
  from origin 'http://localhost:5173' has been blocked
  by CORS policy: Response to preflight request doesn't
  pass access control check: No 'Access-Control-Allow-
  Origin' header is present on the requested resource.

The BFF had **no CORS middleware at all** -- a
pre-existing gap from V3.0 (Track F) that the smoke
scripts never caught because they hit the BFF on
`localhost:3001` directly without an `Origin` header.
CI runs the smoke scripts on each PR, so this
regression-shaped gap survived every release from V3.0
through V3.4. The V3.5 track was the first time the
web app and the BFF were exercised together through a
real browser, and the preflight failed on the first
`fetch('/api/scene')`.

**Fix**:

- `apps/bff/src/server.ts` -- mounts Hono's built-in
  `cors` middleware (`hono/cors`) before all routes,
  with `origin` resolved from `CORS_ALLOWED_ORIGINS`
  (env var, comma-separated), defaulting to
  `['http://localhost:5173', 'http://localhost:1420']`
  in development and an empty array (deny all) in
  production, and `credentials: true` so the OIDC
  session cookie rides along on cross-origin XHRs.
- `apps/bff/src/server.ts` -- a new `buildApp` helper
  exported alongside `createServer`. The helper returns
  the assembled Hono app without binding to a port, so
  the new test suite can exercise preflight + simple-
  response CORS paths in-process via `app.request`.
  `createServer` still owns the `serve()` + drain path;
  the public API of `createServer` is unchanged.
- `apps/bff/src/__tests__/cors.test.ts` -- 12 tests
  covering the `resolveCorsOrigins` policy matrix
  (dev defaults, prod deny, comma parse, trim,
  whitespace-only treated as unset, env overrides dev)
  and the `buildApp` preflight + simple-response
  behavior (allowed origin, non-allowlisted origin,
  no-Origin same-origin path, production deny,
  env-driven production allowlist).
- `docs/development/local-dev.md` -- documents
  `CORS_ALLOWED_ORIGINS` in the BFF env-var table and
  adds a "No 'Access-Control-Allow-Origin' header"
  troubleshooting entry pointing back at ADR 0018.

**Why this scope belongs on the V3.5 PR, not a hotfix
PR**: the user's established pattern is to fold
follow-ups into the active branch and let the PR
absorb them. PR #18 in V4-prep carried `feat(ui-kit)`
+ `feat(app-shell)` together; the V3.4 closure did
the same. A separate CORS-unblocker hotfix PR would
have created a release that sits between V3.4 and
V3.5 in the changelog, which is worse history than
folding the unblocker into the i18n release that
revealed it. The v3.5 plan was amended in place to
record the addition as T8.1.

**Re-test triggers for the future**: if the BFF ever
adds a new app origin (mobile webview, third-party
embed), the `buildApp CORS preflight (V3.5 Track K)`
test group in `apps/bff/src/__tests__/cors.test.ts` is
the single point that needs a new case.

## Decision

V3.5 ships 8 implementation tasks (T1-T8) + the T8.1
CORS unblocker as a single auditable release. Two
artifacts capture the work: the i18n architecture and
"do not translate" rules (ADR 0018), and this closure
ADR (0019).

## T8.2 - WebSocket subprotocol auth tunnel (scope addition, also tracked in the V3.5 plan)

After T8.1 landed and the browser finally reached the BFF,
the WebSocket stream still failed:

  WebSocket connection to 'ws://localhost:3001/api/stream'
  failed: ...

Root cause: `/api/stream` was gated on `requiresTenantScope`
(V3.3 added the tenant isolation), but the browser WebSocket
API does not let the client set arbitrary request headers,
so the upgrade went out without the `Authorization: Bearer`
header. The server-side gate has no token to look up; the
upgrade is rejected with 401.

The standard escape hatch is the `Sec-WebSocket-Protocol`
subprotocols list -- the browser DOES let the client pass an
array of subprotocol names, and the server can read them off
the upgrade request. The client tunnels the token as a
subprotocol value, the BFF reads it back and injects it into
`Authorization`, and the existing gate handles the rest. No
new token format, no new session store, no new auth model --
just a translation step between two transports.

**Changes**:

- `packages/realtime/src/index.ts` --
  `createWebSocketStream` accepts an optional `getToken:`
  and passes `['bearer', token]` as the subprotocols list
  when a token is present. `WebSocketRealtimeStream` gains
  a `reconnect()` method (the app shell uses it after
  successful login to attach with the new token).
- `packages/api-client/src/create-api-client.ts` -- exposes
  `getAuthToken(): string | null` so the realtime stream
  reads the same source of truth that `setAuthToken`
  mutates.
- `packages/app-shell/src/stores/auth-store.ts` -- adds a
  `token` computed ref derived from
  `state.value.session.token`.
- `packages/app-shell/src/composables/useDeviceStream.ts`
  -- reads `api.getAuthToken()` on every connect, watches
  the auth store's `token` ref, and calls
  `stream.reconnect()` on transitions from `null` to a real
  token.
- `apps/bff/src/server.ts` -- adds a `subprotocolAuth`
  middleware (global, right after CORS) that reads
  `Sec-WebSocket-Protocol`, finds the `bearer` value, and
  writes the trailing subprotocol into `Authorization` on
  the underlying Web Request.
- `apps/bff/src/server.ts` -- `buildApp` now returns the
  `AuthStore` it wired so tests can mint a real session and
  exercise the end-to-end path.
- `apps/bff/src/__tests__/cors.test.ts` -- adds 9 cases
  (6 pipeline + 3 end-to-end) to the existing CORS test
  file.
- `packages/realtime/src/__tests__/ws-stream.test.ts` --
  adds 5 cases for `getToken` and `reconnect()`.
- `packages/api-client/src/__tests__/api-client.test.ts` --
  adds 4 cases for `getAuthToken()`.
- `docs/development/local-dev.md` -- WebSocket auth note in
  the BFF env-var section + a troubleshooting entry for the
  exact browser error message.

**Re-test triggers for the future**: any new WebSocket
endpoint on the BFF inherits the auth path automatically
(the middleware is global). If the BFF ever switches to
cookie-based auth, both this subprotocol tunnel and the
tenant scope gate can be removed in a single follow-up.

## Deliberate non-goals (codified, not scope-crept later)

These are restated from
[docs/plans/v3.5-implementation-plan.md](../plans/v3.5-implementation-plan.md)
so the closure record captures them in one place:

- **No `vue-i18n`**. The decision is in ADR 0018. If
  plural forms or date formatting become necessary
  later, promote `@dt/i18n` to a `vue-i18n` facade in
  a follow-up ADR.
- **No locale lazy-loading**. Two locales × ~25 keys
  each ships in the initial bundle. Future growth
  pushes for lazy-loading per ADR 0018 "Revisit when".
- **No RTL layout**. Not on the roadmap.
- **No `@dt/ui-kit` i18n wiring**. ui-kit stays
  copy-free by convention. Any future component with
  text is a separate ADR.
- **No renaming of API contract fields**. `plugin:publish`
  stays `plugin:publish` in every locale. The
  marketplace panel renders it inside `<code>...</code>`
  literally.

## What's next

The V3 roadmap (ADR 0012) is closed across V3.0 through
V3.5. V3.4.x follow-ups (the items the V3.4 plan records
as "Known caveats": file-based storage -> SQLite/Postgres,
per-instance HMAC -> GPG/Sigstore/cosign, local-only
marketplace -> remote registry, stub activation -> real
activation, single-replica fan-out -> shared storage)
are the natural parking lot for the next V3-era cleanup
work, but they are V3.4.x, not V3.5.x.

The V4 era is a separate ADR (still in the V4-prep
visual refresh) and a separate roadmap. The V3.5 closure
hands off a fully-localized chrome to whatever the V4
redesign lands.
