# ADR 0017 - V3.4 Closure (Track J: Plugin Marketplace + Persistence)

## Status

Accepted (V3.4 implementation complete). Records the
moment V3.4's 8 implementation tasks land, so the
V3 roadmap (ADR 0012) has a single auditable artifact
to point at when asking "what was V3.4?" and the
post-V3 era (V4 +) has a clean handoff.

## Context

ADR 0012 defined the V3 roadmap: 5 tracks (F real auth,
G production platform, H Tauri releases, I multi-tenant,
J plugin marketplace) across 5 releases (V3.0 -> V3.4).
V3.0 shipped Track F (real auth / OIDC) per ADR 0013;
V3.1 shipped Track G (production platform / Helm chart)
per ADR 0014; V3.2 shipped Track H (Tauri release
pipeline) per ADR 0015; V3.3 shipped Track I
(multi-tenant data model) per ADR 0016. V3.4 is Track J
and the **last V3 track**; the V3 roadmap closes with
this ADR.

Track J graduates the V2.2 plugin contract (manifest,
registry, extension types) into a real marketplace
with persisted install state, signed artifacts, a
publish / install / activate / uninstall API, and a
tenant-scoped authorization model that reuses V3.3's
`requiresTenantScope` middleware. V3.4 is the V3-era
last major surface change; the post-V3 era (V4) is a
separate ADR and a separate roadmap.

V3.4 layers on top:

- New persistence types in `@dt/plugin-runtime`
  (`PluginStore`, `InstallRecord`, `InstalledPluginVersion`,
  `compareSemver`) so the marketplace has a single
  async interface a storage backend implements. The
  runtime itself stays pure types + validation; no
  filesystem coupling.
- A new `@dt/plugin-registry` package that owns the
  marketplace data model (`Plugin`, `PluginVersion`,
  `RegistryIndex`) and ships the
  `createInMemoryPluginIndex` factory the BFF tests
  use. Pure types + an in-memory factory; no
  filesystem, no Vue, no BFF coupling.
- A file-based `FilePluginStore` in
  `apps/bff/src/plugins/storage.ts` with the on-disk
  shape `<root>/<tenantId>/<pluginId>/<version>/...`
  for install-time artifacts and
  `<root>/_registry/<pluginId>/<version>/...` for
  publish-time artifacts. Single-replica only; a
  V3.4.x follow-up moves the store to SQLite or
  Postgres when multi-replica or higher volume
  warrants it. The `PluginStore` interface is the
  single point that changes.
- HMAC-SHA256 artifact signing in
  `apps/bff/src/plugins/signing.ts`. The BFF refuses
  to boot in production with a missing or short
  `PLUGIN_SIGNING_SECRET`; in dev a per-checkout
  random secret is auto-generated and written to
  `apps/bff/.data/dev-signing-secret` so the dev
  loop works without manual setup. The signature is
  per-instance (not third-party-verifiable); a
  V3.4.x follow-up moves to GPG / Sigstore / cosign
  when the operator wants third-party-verifiable
  provenance.
- The marketplace API surface in
  `apps/bff/src/routes/marketplace.ts`:
  `GET /api/plugins`, `POST /api/plugins`,
  `GET /api/plugins/:id`, `POST /api/plugins/:id/install`,
  `GET /api/plugins/:id/installed`,
  `PUT /api/plugins/:id/activate`,
  `DELETE /api/plugins/:id/installed/:ver`. Every
  route is gated on `plugin:read`; the explicit
  policy gate (`canInstallForTenant` /
  `canPublish` in `apps/bff/src/plugins/policy.ts`)
  layers the marketplace-specific admin-only rule on
  top so install / publish / activate / uninstall
  are admin-only.
- The `Permission` union in `@dt/contracts` grows
  by three: `plugin:read` (granted to every role),
  `plugin:install` (admin only),
  `plugin:publish` (admin only). The extension is
  additive; the V3.0 stable contract is preserved.
  Consumers that exhaustively switch on the union
  need a compile-time bump to add the new cases;
  the V3.4 test suite covers this.
- The app-shell's `useMarketplaceInstall` composable
  - `MarketplacePanel` UI render marketplace-installed
    plugins as placeholder panels (manifest name +
    version). Real dynamic activation is a V3.4.x
    follow-up; V3.4 proves the install + activate
    plumbing without committing to a dynamic-import
    shape.
- Operator handoff at
  `docs/development/marketplace.md`: the model, the
  signing key contract, the dev loop, the API surface
  table, and a troubleshooting tree for every
  documented error code.

The V3.4 plan is at
[`docs/plans/v3.4-implementation-plan.md`](../plans/v3.4-implementation-plan.md);
the per-task detail, the acceptance matrix, and the
known caveats live there. This ADR records the
moment the plan became code on `main`.

## Decision

V3.4 is **implementation complete** as of commit
`19568d0` ("fix(bff): unblock windows bootstrap tests
(fileURLToPath + secret)", the T8 closure-time fix).
The 8 implementation tasks are shipped; 11 of the 12
acceptance items in the plan's table are verified.
The remaining 1 acceptance item (item 12) requires
the operator to read the first post-merge
`smoke:marketplace` CI run in full per `AGENTS.md` § 5;
it cannot be verified on `main` itself because the CI
run that needs to be inspected is the first one that
fires against the merged V3.4 surface.

The plan's effective target version is
`digital-twin-platform@6.1.0` (V3-era minor bump per
ADR 0012's V3-era policy; closing a V3 track is the
trigger, but V3.4's `Permission` union extension is
additive so the bump is a minor, not a major).
However, per the release-please artifact caveat
recorded in ADR 0014 and re-confirmed in the V3.3
closure (ADR 0016), release-please computes a
`4.x` / `5.x` minor bump from the conventional
commits because no V3.4 commit carries a
`BREAKING CHANGE:` footer. The release-please bot
opened PR #17 shortly after the T8 closure-time
fix landed; the operator chose Option A (merge
as-is, matching the V3.1 / V3.2 / V3.3 precedent)
and the release was published as
[`digital-twin-platform-v4.3.0`](https://github.com/zervero/digital-twin-platform/releases/tag/digital-twin-platform-v4.3.0)
at commit `b02616e` (release-please PR #17,
merged 2026-07-08T13:33:07Z). The V3-era intended
version `6.1.0` is preserved as a follow-up ADR
concern: a future V3.4.x closure ADR can either
retroactively renumber `v4.3.0` to `v6.1.0` via a
tag / release edit, or accept the artifact and
move on. The Options B and C paths (manual bump to
`6.1.0`, or add a `BREAKING CHANGE:` footer) are no
longer available because PR #17 is merged.

### Deviations from the plan

Six items in the plan were deliberately adjusted
during implementation. The first three are scope
adjustments the plan itself called out; the last
three are closure-time findings (T5 bootstrap path,
T5 production test, T8 smoke assertion).

1. **In-memory `MemoryPluginStore` for the T3 unit
   tests, file-based `FilePluginStore` in T4** (per
   plan §T3 step 3.0). T3 ships the routes with a
   thin in-memory `PluginStore` so the route unit
   tests stay fast and free of filesystem state; T4
   ships the file-based implementation the BFF's
   server actually wires. Both implementations
   satisfy the same 7-method contract, so the route
   handlers cannot tell them apart. The
   `marketplace.test.ts` (T3) uses `MemoryPluginStore`
   via `mkdtempSync` + `process.env.PLUGIN_STORAGE_ROOT`
   for filesystem isolation; `storage.test.ts` (T4)
   exercises the `FilePluginStore` end-to-end against
   a per-test tmp dir.
2. **`FilePluginStore` is single-replica only; a real
   database is a V3.4.x follow-up** (per plan
   §"Known caveats"). The on-disk JSON layout works
   for the V3.4 install volume (a few hundred plugins
   per tenant at most, each with <10 versions); the
   multi-replica story needs shared storage (NFS, S3,
   or a database). The `PluginStore` interface is the
   single point that changes when the swap lands.
3. **HMAC-SHA256 per-instance secret, not
   third-party-verifiable; a real signing chain
   (GPG / Sigstore / cosign) is a V3.4.x follow-up**
   (per plan §"Known caveats"). The signature is
   bound to the per-instance secret; an attacker with
   the secret can sign malicious artifacts. The
   secret is the trust boundary. Moving to a
   third-party-verifiable provenance chain is a
   V3.4.x follow-up when the operator wants it.
4. **T5 bootstrap path fix for Windows
   (`fileURLToPath` instead of `new URL().pathname`)**.
   The T5 commit
   (`f0e4fe8 feat(bff): add v3.4 plugin artifact
signing + dev secret bootstrap`) used
   `new URL('.', import.meta.url).pathname` to
   compute `DEV_SIGNING_SECRET_PATH`. On Windows
   the URL pathname has a leading slash
   (`/D:/a/.../src/`), and `path.win32.resolve`
   then turned it into a doubled
   `D:\D:\a\...\.data\dev-signing-secret` — every
   `ensureSigningSecret` call in dev blew up with
   `ENOENT: mkdir 'D:\D:\a\...\.data'`. The Linux
   CI never caught this; the Windows CI caught it
   on the first post-T8 run. Fixed in
   `19568d0` by switching to
   `fileURLToPath(new URL('.', import.meta.url))`,
   which returns a native path on both platforms.
   This is the kind of bug AGENTS.md § 5 explicitly
   warns about: a green Linux run can hide a
   Windows path bug, and a green log can hide a
   missing-env-var bug. The Windows CI job is the
   floor for the marketplace smoke, and it is the
   reason the T5 fix landed before this ADR.
5. **T5 production test missing
   `PLUGIN_SIGNING_SECRET`** (also fixed in
   `19568d0`). The V3.1 T2 "reports
   environment=production" test stubs
   `readAppEnv` to return `production: true` but
   never sets `PLUGIN_SIGNING_SECRET`, so the
   T5 `ensureSigningSecret` guard throws and
   `bootstrap` fails before the test can observe
   the env-reporting call to `startOtel`. The test
   is asserting the env-reporting path, not the
   secret validation, so it gives a 64-byte secret
   and lets `ensureSigningSecret` short-circuit on
   the existing-and-long branch. Pre-existing test
   bug, surfaced by the T5 commit; fixed in the
   same closure commit as deviation #4.
6. **T8 smoke email format uses `admin@example.com`
   instead of the plan's `admin@acme-corp`**. The
   plan's example used `admin@acme-corp` to keep
   the email domain aligned with the tenant id, but
   the `MockAuthStore` email regex
   (`^[^\s@]+@[^\s@]+\.[^\s@]+$`) requires a dot in
   the domain, so `acme-corp` is rejected. The smoke
   uses `admin@example.com` and `op@example.com`
   (matching the V2.1 / V2.3 smoke convention) and
   relies on the mock store's hardcoded
   `tenantId: 'acme-corp'` to scope the session to
   the right tenant. The semantic is unchanged; the
   email is just a mock-store handle, not a tenant
   id.
7. **T8 smoke `jq -r` assertion fix**. The plan's
   example compared the active version with the
   quoted literal `'"1.1.0"'`, but `jq -r` outputs
   raw strings without JSON quotes, so the
   comparison always failed. Fixed by dropping the
   quotes (`'1.1.0'`). The pre-existing V2.1 /
   V2.3 smokes sidestep this by using
   `grep -q` rather than `jq -r | test`; the T8
   smoke uses `jq` directly because it asserts on
   structured fields (`.active`, `.version`,
   `.tenantId`) where `grep` would be brittle.

## Consequences

- **The V3 roadmap is closed.** All 5 V3 tracks (F,
  G, H, I, J) shipped across 5 releases (V3.0 ->
  V3.4). The V3-era intended versions (`3.0.0` for
  V3.0, `4.0.0` for V3.1, `5.0.0` for V3.2,
  `6.0.0` for V3.3, `6.1.0` for V3.4) are recorded
  in their respective closure ADRs; the actual
  released versions are footnoted in the CHANGELOG
  per the release-please artifact pattern from
  ADR 0014. A V4 ADR is a separate concern and a
  separate roadmap.
- **The marketplace has a real persistence layer.**
  The V2.2 plugin contract is now backed by a
  file-based `PluginStore` with HMAC-SHA256 signing,
  a tenant-scoped install model, and a
  marketplace API surface. A V3.4.x follow-up moves
  the store to SQLite or Postgres when the volume
  or the multi-replica story warrants it; the
  `PluginStore` interface is the single point that
  changes.
- **Operators are the install / publish
  authorization gate.** The `Permission` union
  extension (`plugin:read` for every role,
  `plugin:install` and `plugin:publish` for admin
  only) is the contract; the
  `canInstallForTenant` / `canPublish` policy gate
  is the enforcement point. Tenants can browse the
  registry (any role with `plugin:read`); only
  admins can mutate it.
- **The marketplace is local + signed, not remote +
  third-party-verifiable.** A V3.4.x follow-up
  moves the publish surface to a remote registry
  (GitHub Releases, npm, or a custom
  `@dt/plugin-index`) and the signing chain to
  GPG / Sigstore / cosign when the operator wants
  third-party-verifiable provenance. The V3.4
  surface is the install envelope, not the load
  envelope; the V2.2 trust model (same-origin
  static import, contract-based) still applies to
  marketplace plugins.
- **The activation envelope is a stub.** The
  `useMarketplaceInstall` composable renders
  marketplace-installed plugins as placeholder
  panels (manifest name + version). Real dynamic
  activation (the plugin's `activate` function
  running its real logic) is a V3.4.x follow-up;
  V3.4 proves the install + activate plumbing
  without committing to a dynamic-import shape.
- **`@dt/contracts` `Permission` union is now 14
  values, not 11.** The extension is additive; the
  V3.0 stable contract is preserved. Consumers that
  exhaustively switch on the union need a
  compile-time bump to add the new cases; the V3.4
  test suite covers this. The next major ADR
  (V4 or a V3.4.x follow-up that breaks the union)
  is the place to formalize the new shape.

V3.4.x.
