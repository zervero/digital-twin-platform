# ADR 0013 - V3.0 Closure (Track F: Real Auth, OIDC)

## Status

Accepted (V3.0). Records the moment V3.0 ships, so V3.1
(Track G: production platform) has a single auditable
artifact to point to when asking "what was V3.0?".

## Context

ADR 0012 defined the V3 roadmap: 5 tracks (F real auth, G
production platform, H Tauri releases, I multi-tenant, J
plugin marketplace) across 5 releases (V3.0 → V3.4).
`docs/plans/v3-overview.md` ordered them `F → G → H → I → J`
so the smallest possible delta from V2.3 — real auth — lands
first and unlocks everything else (most importantly Track I,
which needs a real IdP for tenant identity).

V2.3 shipped as `digital-twin-platform@2.3.0` on 2026-07-06
(see [ADR 0011](./0011-v2.3-closure.md)). V2.3 carved out
the `AUTH_PROVIDER` env gate and the `AUTH_PROVIDER=oidc`
constant, but the OIDC code path was a stub. The V2.3 closure
ADR explicitly lists "Real auth provider (OAuth, SAML, JWT
signing) — V3" under "What's not in V2.3".

V3.0 is Track F: filling in that OIDC code path so
`AUTH_PROVIDER=oidc` is functional end-to-end (BFF verifies
JWTs, the V2.1 permission union is enforced from the token's
`scope` / `permissions` claims, the web + desktop redirect
to a real OIDC issuer, the `mock` provider keeps working for
dev). The V3.0 plan covers:

- A new `@dt/auth-oidc` package that owns JWT verification +
  JWKS fetching (no Vue, no Three, no BFF).
- An `OidcConfig` block in `@dt/config` validated at boot.
- An `OidcAuthStore` in the BFF that reads an HttpOnly cookie
  and verifies it against the issuer's JWKS.
- Two new BFF routes (`/api/auth/oidc/start`,
  `/api/auth/oidc/callback`) that drive the Authorization
  Code + PKCE flow.
- The V2.1 `requiresPermission` middleware actually wired
  into the existing `/devices`, `/scene`, `/commands` routes
  (it was a stub in V2.1; V3.0 makes it the gate).
- A `useOIDCStart` composable in `@dt/app-shell` plus a
  LoginButton that the web + desktop apps use to trigger the
  redirect.
- A dev OIDC IdP (`scripts/dev-oidc-idp.mjs`) that signs real
  RS256 JWTs so the OIDC code path is exercisable without a
  hosted provider.
- An e2e smoke (`scripts/smoke-oidc.sh`) that drives the full
  flow against the dev IdP.
- A new `docs/development/oidc.md` that captures the env-var
  contract, JWT claim shape, cookie attributes, and
  troubleshooting.

`docs/plans/v3.0-implementation-plan.md` decomposes V3.0 into
8 bite-sized tasks with exact files, exact code, exact tests,
and exact verification commands. The plan resolves 7 open
design questions (provider choice, JWKS caching, claim shape,
token transport, dev IdP, middleware semantics, dev BFF
toggle) up front so the implementation is mechanical, not
exploratory.

This ADR records that all 8 tasks shipped and that the 10
acceptance items are verified.

### Deviations from the plan

Two items in the plan were deliberately adjusted during
implementation:

1. **AuthStore interface was refactored to be headers-based.**
   The plan called for a token-based `getMe(token?)` / `logout(token?)`
   shape. During T3 we realized the BFF's auth route needs the
   raw request `Headers` to read either the `Authorization`
   header (mock) or the `dt_oidc_session` cookie (OIDC), and
   that abstracting the transport behind a `Headers` argument
   was cleaner than threading two different shapes. The new
   `getMe(headers)` / `logout(headers)` shape is a strict
   generalization — mock still reads the bearer header; OIDC
   reads the cookie. No public behavior changed.
2. **`AuthSession.permissions?` was made optional.** The plan
   implied permissions always come from `permissionsFor(roles)`.
   The OIDC store returns permissions directly from the JWT,
   so `permissions` is now `readonly Permission[] | undefined`.
   The `requiresPermission` middleware falls back to the
   role-based derivation when the field is absent (mock path)
   and uses the JWT's claim when present (OIDC path). The
   `mock` shape is unchanged.

Neither deviation changes the V3.0 public surface or the
acceptance matrix. Both are documented in code comments in
`apps/bff/src/auth/store.ts` and `apps/bff/src/auth/oidc-store.ts`.

### OIDC email-bug fix (T7 follow-up)

During T7, `OidcAuthStore.toAuthSession` synthesized the
email as `${subject}@${issuer.host}`. For the dev IdP (where
`sub` is already the user's email), this produced
`admin@example.com@localhost:9999` — clearly wrong. The fix
in commit `b6a5e4f` reads the standard OIDC claims
(`email`, `name`, `preferred_username`) with a fallback to
`subject`, and uses `VerifiedSession` from `@dt/auth-oidc` as
the parameter type so the claims object is reachable. The
fix is verified by `pnpm smoke:oidc` step 5, which asserts
`"email":"admin@example.com"` in the `/me` response.

## Consequences

- `@dt/auth-oidc` becomes a permanent dependency of the
  platform. Its public surface (`verifyJwt`,
  `verifyJwtWithResolver`, `extractPermissions`, `ALL_PERMISSIONS`)
  is the canonical place to evolve OIDC semantics in V3.x.
- The `AUTH_PROVIDER` gate from V2.3 stays. `mock` is always
  valid; `oidc` is valid only when the OIDC env vars are set.
- The `requiresPermission` middleware is now the single
  permission-gating surface. Route handlers stop using
  inline role checks; future routes should call it.
- The dev IdP is a development convenience, not a production
  component. Production deployments must point at a real
  OIDC issuer. The "What's not in V3.0" section of
  [`docs/development/oidc.md`](../../development/oidc.md)
  enumerates the V3.0 out-of-scope items (RP-initiated
  logout, refresh tokens, token introspection, group claim
  mapping, multi-tenant routing).
- The `mock` provider still works end-to-end in dev. The
  `LoginRequest.roles` field it accepts (added in V2.3) is
  preserved; the OIDC store ignores it.

## What's next

V3.1 is Track G (production platform): Kubernetes manifests,
Helm chart, TLS via cert-manager, OpenTelemetry metrics +
traces + logs. The V3.1 plan will land in
[`docs/plans/v3.1-implementation-plan.md`](../plans/) and
follow the same 8-task decomposition that V3.0 used. Track
G depends on V3.0's auth being functional (so the ingress +
TLS contract has a real `Authorization` flow to terminate
against) but does not require Track I (multi-tenancy) to
ship first.
