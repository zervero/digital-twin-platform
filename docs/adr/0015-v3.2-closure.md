# ADR 0015 - V3.2 Closure (Track H: Tauri Release Pipeline)

## Status

Accepted (V3.2 implementation complete). Records the
moment V3.2's 8 implementation tasks land, so the
remaining operator actions (provision GH secrets, regen
updater keypair with a real password, first tag-triggered
CI run) have a single auditable artifact to point at when
asking "what was V3.2?".

## Context

ADR 0012 defined the V3 roadmap: 5 tracks (F real auth,
G production platform, H Tauri releases, I multi-tenant,
J plugin marketplace) across 5 releases (V3.0 -> V3.4).
V3.0 shipped Track F (real auth / OIDC) per ADR 0013;
V3.1 shipped Track G (production platform / Helm chart)
per ADR 0014. V3.2 is Track H.

Track H graduates the V1 "Tauri runs locally" story into a
signed, auto-updating desktop distribution. V1 shipped a
Tauri 2 scaffold that boots the Vite webview against the
local BFF; V3.2 layers on top:

- A signed macOS `.dmg` (Developer ID + Apple notary).
- A signed Windows `.msi` (Authenticode; unsigned fallback
  allowed for dev releases, controlled by a workflow
  input).
- A Linux `.AppImage` + `.deb` + `.rpm` triple (unsigned in
  V3.2; GPG signing is a documented V3.2.x follow-up).
- An in-app update channel using `@tauri-apps/plugin-updater`
  that reads `releases/latest/download/latest.json` from
  GitHub Releases and verifies the artifact signature
  against the public key committed to
  `apps/desktop/src-tauri/tauri.conf.json`.
- A new `.github/workflows/desktop-build.yml` that
  orchestrates the matrix build + the release-consolidation
  job that flips the draft GitHub Release to "latest".
- Two new operator-facing docs: `desktop-releases.md`
  (end-to-end flow + manual fallback + troubleshooting)
  and `desktop-signing.md` (the three secret sets and
  the updater keypair ceremony).

`docs/plans/v3.2-implementation-plan.md` decomposes V3.2
into 8 bite-sized tasks with exact files, exact code,
exact tests, and exact verification commands. The plan
resolves 4 open design questions (release trigger model,
signing-secret model, Linux distribution shape, updater
keypair handling) up front so the implementation is
mechanical, not exploratory.

This ADR records that all 8 tasks shipped and that 9 of
the 12 acceptance items are verified locally. The
remaining 3 items are operator-only (see "Operator
follow-up" below) and unblock the first real release-please
tag.

## Decision

V3.2 is **implementation complete** as of commit `3950102`.
The 8 implementation tasks and 9 of the 12 acceptance
items are verified. The remaining 3 acceptance items
(items 8, 9, 10 in the plan's table) require operator
action and the first tag-triggered CI run; they cannot
be verified on `main` itself because GitHub Actions does
not expose a way to synthesize a `v5.0.0` tag locally.
The plan's effective target version is
`digital-twin-platform@5.0.0` (V3-era major bump per
ADR 0012; closing a V3 track is the trigger). No tag
has been cut yet. release-please will emit the tag on
its next pass over `main`, subject to the release-please
artifact caveat called out in ADR 0014.

### Deviations from the plan

Two items in the plan were deliberately adjusted during
implementation, both already noted in their respective
commit messages:

1. **T2 wired the boot scaffold in `src/index.ts` (not
   `src/main.ts` as the plan's `apps/desktop/src/main.ts`
   snippet suggested).** The current Tauri webview loads
   `apps/web/dist` directly (a Vite build, not a local
   dev server), so the boot call into `@dt/app-shell` is
   dead code today. The wrapper (`src/lib/updater.ts`) is
   still real and is what the in-app updater will call once
   the webview loads the shell bundle. The boot call will
   be wired up in V3.2.x when the webview starts shipping
   the shell bundle instead of the raw Vite output. Recorded
   in commit `78bb033`'s body.
2. **T7's release-consolidation job adds
   `gh release edit --draft=false`** on top of the plan's
   "verify artifacts" snippet, and skips the
   `pull-requests: write` permission (no PR to comment on
   for a tag-triggered release). The plan's "promote the
   draft to latest" intent is preserved; the implementation
   uses `gh release edit` because the GitHub-native "Latest"
   flag is what the in-app updater polls for. Recorded in
   commit `b1d6aae`'s body.

Plus one non-deviation worth recording:

3. **T3 ships a no-password dev updater keypair** as a
   placeholder so the local Rust toolchain can sign a test
   build. The committed pubkey is real and the
   `tauri.conf.json` `pubkey` field is wired to it. The
   **operator must regenerate the keypair with a real
   password** before the first real release; the
   `desktop-signing.md` doc walks through the
   `tauri signer generate -p <password>` ceremony. The
   generated key file is git-ignored (`*.key` is in
   `.gitignore`); only the `.pub` counterpart is committed.
   The `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` GH secret is
   a no-op until the keypair is regenerated; the workflow
   already handles "no password" gracefully.

## Consequences

- The V1 "Tauri runs locally" story is now a V3.2 "Tauri
  ships signed, auto-updates from a single GH Release"
  story. V1 contributors get the same `pnpm --filter
  @dt/desktop dev` loop they had before; V3.2 adds
  `pnpm --filter @dt/desktop build` for a local production
  bundle and a tag-triggered CI path for cross-platform
  release.
- `@tauri-apps/plugin-updater` becomes a permanent
  dependency of the desktop app. Its public surface
  (`check()`, `downloadAndInstall()`, the in-app
  update prompt UX) is the canonical place to evolve
  the update channel in V3.2.x. The wrapper at
  `apps/desktop/src/lib/updater.ts` is the single entry
  point; raw `@tauri-apps/plugin-updater` calls should
  not appear in the webview code.
- Code-signing and notarization are gated on operator-
  managed secrets. The workflow is non-functional for
  macOS / Windows until the operator provisions
  `APPLE_*` (5) and `WINDOWS_CERTIFICATE` / `_PASSWORD`
  (2) in the repo's Actions secrets. The Linux job has
  no secret requirement and is the path of least
  resistance for forks and dev VMs.
- GitHub Actions repo-level setting "Allow GitHub Actions
  to create and approve pull requests" must be ON for
  release-please to open its release PR. The same toggle
  is also what release-please uses to comment on the
  release PR with changelog links. This was the cause of
  the early V1 release-please failure recorded in
  `AGENTS.md` S5; V3.2 inherits the same requirement
  but does not introduce new ones.
- The Tauri 2 dependency is now part of the desktop app's
  VCS-stated surface. Pinning the version (currently
  `tauri = "2"` in `Cargo.toml` and `@tauri-apps/plugin-updater`
  `^2` in `package.json`) is the way to avoid silent
  breaking changes from upstream minor releases; the V1
  pre-existing `@tauri-apps/api` was already pinned.
- `docs/development/deployment.md` (the V2.3-era guide)
  is now slightly out of date: the desktop app has a
  release path that lives in `desktop-releases.md`, not
  `deployment.md`. A future V3.x ADR can consolidate
  these; for V3.2 we keep both docs to avoid scope creep.
- The V3.1-era release-please artifact caveat (ADR 0014,
  deviation #4) still applies. release-please may emit
  an extra `5.0.1` tag after the `5.0.0` cut; the V3.2
  CHANGELOG footnote (if needed) will mirror the V3.1
  pattern from `3237833`.

## Operator follow-up (the V3.2 "release-please unblockers")

These are the 3 actions that move V3.2 from "implementation
complete" to "actually shipped". None of them is a code
change; all of them are required before the next
release-please pass can produce a real, signed release.

1. **Regenerate the updater keypair with a real password.**
   `pnpm tauri signer generate -p <password>` in
   `apps/desktop/`. Replace the dev pubkey in
   `tauri.conf.json` with the new one. Add
   `TAURI_SIGNING_PRIVATE_KEY` (the `.key` file contents,
   base64-encoded) and `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`
   to the repo's GH secrets. The ceremony is in
   `docs/development/desktop-signing.md`.
2. **Provision the macOS + Windows code-signing secrets.**
   `APPLE_CERTIFICATE` + `_PASSWORD`, `APPLE_ID`,
   `APPLE_PASSWORD`, `APPLE_TEAM_ID` (5 total) and
   `WINDOWS_CERTIFICATE` + `_PASSWORD` (2 total). The
   `desktop-build.yml` workflow gates macOS and Windows
   jobs on these; the Linux job runs without them.
3. **Read the first tag-triggered `desktop-build.yml` run
   in full**, per `AGENTS.md` S5. The first real run is
   the one that catches the "green with a warning" failure
   mode (Node 20 deprecation, secret-name typos, tauri-action
   version drift, etc.). Do not rely on the
   `pull_request_target`-style test runs that fire on a
   regular PR; those don't have access to the signing
   secrets and exercise the unsigned fallback.

After these 3 actions land, the next release-please PR
(merging `main` -> `main` with the V3.2 commit set) will
cut the `v5.0.0` tag, which triggers `desktop-build.yml`,
which produces the first signed, auto-updatable release
artifacts. Items 8/9/10 in the acceptance matrix flip
green at that point and V3.2 ships for real.

## What's next

V3.3 is Track I (multi-tenant): tenant isolation in the
BFF, tenant-scoped device / scene data, tenant-aware
OIDC claims, and a per-tenant rate limiter. The V3.3 plan
will land in `docs/plans/v3.3-implementation-plan.md`
and follow the same 8-task decomposition that V3.0,
V3.1, and V3.2 used. Track I is largely independent of
F/G/H at the code level (it adds a `tenantId` column to
the BFF's data model) but ships after H so the desktop
release channel established by V3.2 is the distribution
path for any per-tenant desktop configuration that V3.3
needs.

The remaining V3 tracks per ADR 0012 are Track J
(plugin marketplace, V3.4). Tracks I and J are the last
two V3 deliverables; the V3 roadmap closes with V3.4.
