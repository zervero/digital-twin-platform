# ADR 0016 - V3.3 Closure (Track I: Multi-Tenant Data Model)

## Status

Accepted (V3.3 implementation complete). Records the
moment V3.3's 8 implementation tasks land, so V3.4
(Track J: plugin marketplace) has a single auditable
artifact to point at when asking "what was V3.3?".

## Context

ADR 0012 defined the V3 roadmap: 5 tracks (F real auth,
G production platform, H Tauri releases, I multi-tenant,
J plugin marketplace) across 5 releases (V3.0 -> V3.4).
V3.0 shipped Track F (real auth / OIDC) per ADR 0013;
V3.1 shipped Track G (production platform / Helm chart)
per ADR 0014; V3.2 shipped Track H (Tauri release
pipeline) per ADR 0015. V3.3 is Track I.

Track I graduates the V3.0 single-tenant data model
into a tenant-scoped one. V1 - V3.2 all share a single
implicit tenant (the dev loop and the OIDC `mock`
provider both mint sessions for "the platform"); V3.3
makes the tenant a first-class field on every scoped
DTO, a namespaced JWT claim sourced from a real OIDC
IdP, and an enforcement point at every BFF route.

V3.3 layers on top:

- A new `@dt/tenant` package that owns the `Tenant`
  shape, the canonical claim constant
  (`https://api.digital-twin-platform.local/tenant_id`,
  overridable via `OIDC_TENANT_CLAIM`), and the
  `getTenantIdFromClaims` helper. Pure types + one pure
  function; no I/O, no Vue, no BFF coupling.
- A `tenantId` field on every tenant-scoped DTO in
  `@dt/contracts` (device, scene, command, event). The
  field is required on inputs, present on outputs, and
  part of the JSDoc contract.
- `@dt/auth-oidc` surfaces `VerifiedSession.tenantId`
  (sourced from the namespaced claim) so the BFF can
  carry the tenant through `AuthSession.tenantId` into
  the new `requiresTenantScope` middleware.
- A BFF middleware
  (`apps/bff/src/middleware/requires-tenant.ts`) that
  resolves the JWT's tenant claim against a dev tenant
  registry (`apps/bff/src/tenants/resolve.ts`), puts the
  resolved `Tenant` on `c.var.tenant`, and returns
  401 `AUTH_NO_TENANT` for missing / unknown tenants.
- Tenant-scoped route handlers (`/api/devices`,
  `/api/scene`, `/api/commands`) that read
  `c.var.tenant.tenant.id` and filter at the data layer.
  Cross-tenant command bodies get 403 `TENANT_FORBIDDEN`
  with a diagnostic message naming both tenants.
- A per-tenant `RealtimeBroadcaster` filter: the
  `broadcast` signature gains a `tenantId` parameter;
  the body applies a 1-line predicate on
  `event.tenantId` so subscribers only see events for
  their tenant. Keepalives (`ping` / `pong`) carry
  `tenantId` too, stamped on the WebSocket route before
  the broadcaster call.
- Mock data split into three tenants
  (`acme-corp` + `globex-ind` + `initech-llc`) in
  `apps/bff/src/mock/demo-data.ts`, with a real
  per-tenant status distribution
  (acme 3 machines, globex 2, initech 4) so the smoke
  can assert isolation by counts, not just by IDs.
- A `mint` subcommand on `scripts/dev-oidc-idp.mjs`
  that signs per-tenant id_tokens with the same RSA
  keypair the server uses (persisted to `os.tmpdir()`
  with `DEV_OIDC_KEY_FILE` override) so the smoke can
  mint and verify in one process group.
- A new `pnpm smoke:tenant` end-to-end smoke
  (`scripts/smoke-tenant.sh`) wired into a new
  `smoke:tenant` CI job in `.github/workflows/ci.yml`
  that runs on every PR and every push to `main`.
- A new `docs/development/multi-tenant.md` operator
  handoff covering the model, the claim shape, the
  dev IdP flow, and a troubleshooting table
  (`AUTH_NO_TENANT` / `TENANT_FORBIDDEN` /
  `AUTH_FORBIDDEN` / missing-WS-event).

`docs/plans/v3.3-implementation-plan.md` decomposes
V3.3 into 8 bite-sized tasks with exact files, exact
code, exact tests, and exact verification commands.
The plan resolves 8 open design questions (tenancy
model, tenant identity source, tenant resolution at
request time, realtime fan-out, permission model,
`@dt/tenant` ownership, WebSocket auth, dev IdP
shape) up front so the implementation is mechanical,
not exploratory.

This ADR records that all 8 tasks shipped and that
11 of the 12 acceptance items are verified locally.
The remaining 1 item (item 12) is operator-only
(see "Operator follow-up" below) and unblocks the
first post-merge smoke:tenant CI run.

## Decision

V3.3 is **implementation complete** as of commit
`d4c9fa7` ("docs(v3.3): multi-tenant model +
cross-link updates", the T8 cross-link commit). The 8
implementation tasks and 11 of the 12 acceptance items
are verified. The remaining 1 acceptance item (item 12
in the plan's table) requires the operator to read the
first post-merge `smoke:tenant` CI run in full per
`AGENTS.md` S5; it cannot be verified on `main` itself
because the CI run that needs to be inspected is the
first one that fires against the merged V3.3 surface.

The plan's effective target version is
`digital-twin-platform@6.0.0` (V3-era major bump per
ADR 0012; closing a V3 track is the trigger). However,
as of the V3.3 closure commit, release-please has
already opened **PR #15** (`chore(main): release
digital-twin-platform 4.2.0`) that lumps V3.2 and
V3.3 commits together with a **minor** bump instead
of the intended major. The decision of which version
actually ships is the operator's call; see "Operator
follow-up" below and the "Deviations" section's
release-please artifact entry.

### Deviations from the plan

Five items in the plan were deliberately adjusted
during implementation, all already noted in their
respective commit messages or the source code
comments:

1. **Smoke uses the `Cookie: dt_oidc_session=...`
   header, not `Authorization: Bearer ...`.** The plan's
   smoke snippet in `docs/plans/v3.3-implementation-plan.md`
   Step 8 used `Authorization: Bearer $TOKEN_A`; the
   implementation in `scripts/smoke-tenant.sh:123-125`
   uses `Cookie: dt_oidc_session=$TOKEN_A` because the
   OIDC store reads the JWT from the session cookie
   (`apps/bff/src/auth/oidc-store.ts`) and the
   `Authorization` header is not consulted on the
   cookie-reading path. The plan's mental model
   conflated "session is established" with "header
   is verified"; the actual contract is that the
   session cookie is the only transport for the OIDC
   JWT. The end-to-end behavior (tenant A cannot see
   tenant B's devices) is the same; the transport
   detail differs.
2. **`issueIdToken` uses a destructured-without-default
   signature, not an ES6 default parameter.** The
   plan's snippet suggested
   `function issueIdToken({ ..., tenantId = DEFAULT_TENANT })`;
   the implementation at
   `scripts/dev-oidc-idp.mjs:191-203` destructures
   without a default and explicitly checks
   `tenantId === undefined || tenantId === null`. The
   reason: an ES6 default would replace an explicit
   `undefined` (the `--no-tenant` mint path) with
   `DEFAULT_TENANT`, silently re-introducing the
   tenant claim on a token that was supposed to be
   tenant-less. The explicit check ensures `--no-tenant`
   tokens actually omit the claim, which is what the
   smoke's `AUTH_NO_TENANT` assertion depends on.
3. **The dev IdP keypair is persisted to
   `os.tmpdir()/dev-oidc-keypair.json`.** The plan
   treated the CLI `mint` subcommand and the server
   listener as the same process, so a fresh keypair
   per smoke run was fine. The implementation at
   `scripts/dev-oidc-idp.mjs:69-78` writes the
   keypair to a file (path overridable via the
   `DEV_OIDC_KEY_FILE` env var, default
   `os.tmpdir() + '/dev-oidc-keypair.json'`) so the
   `mint` CLI and the server listener (two separate
   processes in the smoke) share the same key.
   Without the file, every `mint` invocation would
   generate a fresh keypair that the BFF could not
   verify (the BFF fetches JWKS from the running IdP,
   which still has the previous key). The trade-off
   is that a crashed smoke can leave the keypair file
   behind; this is acceptable for a dev-only script
   and the next smoke just overwrites it.
4. **`withTimestamp` doesn't enforce `tenantId`.** The
   `RealtimeEvent` type from V3.2 had a `withTimestamp`
   helper that stamped `timestamp` but did not stamp
   `tenantId`; the WebSocket route's `ping` keepalive
   was working under V3.2 because the broadcaster
   didn't filter. After T7, the broadcaster filters
   on `event.tenantId`, so a `ping` without
   `tenantId` would be silently dropped (T7 wired
   the explicit stamp on the route side; see
   `apps/bff/src/realtime/broadcaster.ts:11-21` for
   the JSDoc). This is a known gap in
   `@dt/contracts`; a future V3.x ADR can either
   tighten `withTimestamp`'s signature or remove the
   helper in favor of explicit construction at the
   call site.
5. **Mock tenant registry in production code path.**
   The plan called this out as a known caveat; the
   implementation at
   `apps/bff/src/tenants/resolve.ts` is a thin
   `DEMO_TENANTS.find(...)` lookup. The function's
   JSDoc is explicit that production would replace
   this with a database lookup and that the call
   site (`requiresTenantScope`) does not change
   because the return type is the only contract.
   This is recorded here for the audit trail, not
   as a deviation from intent.

Plus one release-please artifact worth recording
explicitly in this ADR (it is the V3-era pattern,
documented separately in ADR 0014 deviation #4 and
ADR 0015's "Operator follow-up"):

6. **Release-please opened PR #15
   (`chore(main): release digital-twin-platform 4.2.0`)
   as a minor bump, lumping V3.2 + V3.3 commits
   together.** No V3.2 or V3.3 commit carries a
   `BREAKING CHANGE:` footer; release-please only
   reads commit messages, not ADRs. The V3-era policy
   in ADR 0012 intends a major bump per V3 track
   (V3.3 should be `6.0.0`), but release-please
   computed `4.2.0` because all 11 unreleased
   features (3 from V3.2 + 8 from V3.3) are
   conventionally `feat:`. PR #15 is OPEN at the
   time of this ADR; the operator's call to merge
   as-is (4.2.0) or edit to 6.0.0 is recorded under
   "Operator follow-up" below. The effective
   V3.3 version is **either** `6.0.0` (per ADR 0012
   policy) **or** whatever release-please emits
   (per release-please artifact caveat, same as V3.1
   / V3.2); the actual tag will be set when the
   release PR merges. The pre-merge PR is at
   <https://github.com/zervero/digital-twin-platform/pull/15>.

None of items 1-5 change the public surface or the
acceptance matrix. All five are documented in code
comments in the relevant files:

- Cookie transport: `scripts/smoke-tenant.sh:14-19`
  and `apps/bff/src/auth/oidc-store.ts` session-cookie
  contract
- `--no-tenant` default: `scripts/dev-oidc-idp.mjs:191-203`
- Keypair persistence: `scripts/dev-oidc-idp.mjs:69-78`
- `withTimestamp` gap: `apps/bff/src/realtime/broadcaster.ts:11-21`
- Mock tenant registry: `apps/bff/src/tenants/resolve.ts`

## Consequences

- The V1-V3.2 "single implicit tenant" story is now
  a V3.3 "tenant-scoped, claim-derived, middleware-
  enforced" story. Existing contributors who only
  touch the `mock` auth provider see no behavior change
  (the mock store mints a default dev `tenantId`);
  contributors who use the OIDC provider now see a
  required `tenant_id` claim and an `AUTH_NO_TENANT`
  error if the claim is missing.
- `@dt/tenant` becomes a permanent shared package. Its
  public surface (`Tenant`, `TenantPlan`, `TenantContext`,
  `TENANT_ID_CLAIM`, `getTenantIdFromClaims`) is the
  canonical place to evolve tenancy semantics in V3.x.
  The package has a single dependency on `@dt/contracts`,
  which preserves the AGENTS.md "no new package may
  import a non-shared package" boundary.
- The BFF's `requiresTenantScope` middleware is the
  canonical place to add new tenant-scoped routes in
  V3.x. Route handlers should read
  `c.var.tenant.tenant.id` and filter at the data
  layer; cross-resource references that need a different
  tenant should produce a 403 `TENANT_FORBIDDEN` with
  a diagnostic message naming both tenants.
- The `RealtimeBroadcaster` signature now requires
  `tenantId`; per-tenant filtering is enforced at the
  broadcast step so subscribers only see events for
  their tenant. Multi-replica deployments will need a
  cross-process pub/sub layer (V3.3.x follow-up); the
  V3.3 filter is per-process only.
- The dev IdP gains a `mint` subcommand that signs
  per-tenant id_tokens; the keypair is persisted to
  `os.tmpdir()` so the CLI and the server listener
  share the same key. The `DEV_OIDC_KEY_FILE` env var
  overrides the path for tests and CI. The mint path
  is dev-only; production uses a real OIDC provider
  and the BFF's JWKS-based verification.
- A new `pnpm smoke:tenant` smoke is part of the CI
  surface. The new `smoke:tenant` job in
  `.github/workflows/ci.yml:129` follows the V3.3
  plan's AGENTS.md S5 checklist: pinned action versions
  (`@v5`), minimum permissions (the job needs no
  write scope), `concurrency:` to cancel duplicates,
  `timeout-minutes: 5` (the smoke itself runs in ~10s).
  The first merged run is the one that needs the
  operator's full read (item 12 in the acceptance
  matrix).
- `docs/development/multi-tenant.md` is the operator
  handoff. The troubleshooting table covers
  `AUTH_NO_TENANT` (claim missing or unknown tenant),
  `TENANT_FORBIDDEN` (command body tenant mismatch),
  `AUTH_FORBIDDEN` (permission check, unchanged from
  V3.0), and "WS event not arriving" (event published
  for a different tenant, broadcaster filter dropped it).
- `docs/development/deployment.md` (V2.3-era) is
  unchanged; multi-tenancy is a runtime / BFF concern,
  not a deployment concern. The V3.1-era
  `production-platform.md` and `oidc.md` are unchanged;
  the V3.3 multi-tenant doc cross-references both.
- The V3.2-era release-please artifact caveat (ADR
  0015's "Operator follow-up") still applies. The
  V3.3 closure PR (#15) is currently open with a
  minor bump (4.2.0) instead of the intended major
  (6.0.0); the operator decides which version
  actually ships. Same as V3.1, the effective
  V3.3 version is "whatever release-please emits,
  footnoted in the CHANGELOG if the intended major
  doesn't match".

## Operator follow-up (the V3.3 "closure unblockers")

These are the 2 actions that move V3.3 from
"implementation complete" to "actually shipped" and
audit-clean. The first is the standard V3-era
post-merge verification; the second is the V3.3-
specific version question.

1. **Read the first post-merge `smoke:tenant` CI run
   in full**, per `AGENTS.md` S5. The first real run
   is the one that catches the "green with a warning"
   failure mode (Node 20 deprecation in older action
   versions, secret-name typos in the OIDC env vars,
   smoke timeouts under load, etc.). Do not rely on
   the PR-level runs that fire on the V3.3 PRs
   themselves; those run against PR branches and
   don't exercise the merged `main` surface. The CI
   job URL is
   <https://github.com/zervero/digital-twin-platform/actions/workflows/ci.yml>
   filtered to the `smoke:tenant` job. Item 12 in
   the acceptance matrix flips green at that point.
2. **Decide the version question for PR #15.** The
   V3-era policy in ADR 0012 intends a major bump
   (V3.3 = `6.0.0`); release-please computed a
   minor bump (`4.2.0`) because no V3.2 or V3.3
   commit carries a `BREAKING CHANGE:` footer.
   Three options:
   - **A. Merge PR #15 as-is** -> ships `4.2.0`,
     lumps V3.2 + V3.3 into a minor bump. Pragmatic;
     follows what release-please computed. Document
     the artifact in the CHANGELOG footnote (same
     pattern as V3.1's `4.1.0` per ADR 0014
     deviation #4).
   - **B. Edit PR #15 to bump `package.json` +
     `.release-please-manifest.json` to `6.0.0`**
     manually -> forces the V3-era major version.
     Requires editing the PR branch or closing +
     reopening. Higher-friction, but matches the
     ADR 0012 intent.
   - **C. Add a `BREAKING CHANGE:` footer to a
     docs-only commit on `main`**, push, wait for
     release-please to re-emit `6.0.0`. Lowest
     friction at the cost of a tiny narrative
     mismatch (the footer would point at a doc
     change, not a real breaking change).
   **Recommendation: Option A**, matching the
   V3.1 / V3.2 precedent. Document the artifact in
   the CHANGELOG footnote; the V3.3 closure is
   "implementation complete + V3-era major bump
   intent preserved as a follow-up ADR" rather
   than "this exact tag was cut". The V3.3
   effective version will be `4.2.0` (or
   `5.0.0` / `6.0.0` if the operator picks B or
   C); the closure ADR is valid for all three.

## What's next

V3.4 is Track J (plugin marketplace + persistence):
persisted plugin storage, signed plugin artifacts,
marketplace install / upgrade API, with V2.2's
plugin contract becoming the activation envelope.
The V3.4 plan will land in
`docs/plans/v3.4-implementation-plan.md` and follow
the same 8-task decomposition that V3.0 - V3.3 used.
Track J is the last V3 deliverable; the V3 roadmap
closes with V3.4 per ADR 0012.

Track J depends on I (multi-tenancy) for the
marketplace's "who can install what" question (the
marketplace needs to know which tenant is asking
and which plugins that tenant is allowed to install)
and on F (auth) for the install / upgrade API
authentication. J is largely independent of G and H
at the code level but ships after I so the
multi-tenant context is available for the marketplace
authorization model.
