# ADR 0009 - V2.1 Closure

## Status

Accepted (V2.1). Records the moment V2.1 (auth contracts) is
shipped, so V2.2 (plugin runtime) has a single auditable artifact
to point to when asking "what was V2.1?".

## Context

ADR 0007 defined the V2 roadmap: 5 tracks (realtime, plugin
runtime, observability, auth, production deploy) split across 4
releases (V2.0 - V2.3). `docs/plans/v2-overview.md` ordered them
A+C → D → B → E so realtime and auth land before plugin runtime,
and the plugin runtime ships before production deployment.

V2.0 (Tracks A+C, realtime + observability) shipped on 2026-07-06
as `digital-twin-platform@2.0.0` (see [ADR 0008](./0008-v2-closure.md)).

V2.1 is Track D: a stable auth and permission model in
`@dt/contracts`, plumbed through `@dt/api-client` and the BFF,
and consumed from `@dt/app-shell` via composables. The motivation
in `docs/plans/v2-overview.md` is that V3 multi-tenancy needs
the model in place before the plugin runtime (V2.2) ships, so
plugin UI can ask "does the current user have permission X?"
without inventing the question and forcing V3 to redo the shape.

`docs/plans/v2.1-implementation-plan.md` decomposed V2.1 into 6
bite-sized tasks with exact files, exact code, exact tests, and
exact verification commands.

This ADR records that those 6 tasks shipped and that the 7
acceptance items are verified.

## Decision

V2.1 is **shipped** as `digital-twin-platform@2.1.0`. All 7
acceptance items below are verified on the host and on CI as of
the release commit (`96f0939`).

## What V2.1 actually contains

- **Contracts** (`@dt/contracts`): a stable auth shape that
  consumers and the BFF both read:
    - `Role` union: `'admin' | 'operator' | 'viewer'`.
    - `Permission` union: `'device:read' | 'device:write' |
      'scene:read' | 'scene:write' | 'command:send' |
      'auth:login'`.
    - `ROLE_PERMISSIONS: Readonly<Record<Role, readonly Permission[]>>`:
      the single source of truth for what each role grants. The
      BFF and the app shell both consult it; adding a permission
      means adding it here and to every role that should grant it.
    - `permissionsFor(roles)` flattens and dedupes the map.
    - `User`, `AuthSession`, `AuthState` (discriminated union
      `{kind:'anonymous'} | {kind:'authenticated', session} |
      {kind:'expired', user}`), `LoginRequest`, `LoginResponse`,
      `MeResponse` (anonymous → `{session:null}`, authenticated →
      `{session: AuthSession}`), `AuthErrorCode`.

- **API client** (`@dt/api-client`): four new methods on
  `ApiClient`:
    - `getMe(): Promise<MeResponse>` — `GET /api/auth/me`, sends
      `Authorization: Bearer <token>` when a token has been set.
    - `login(req: LoginRequest): Promise<LoginResponse>` —
      `POST /api/auth/login`, stashes the returned token in the
      client closure.
    - `logout(): Promise<void>` — `POST /api/auth/logout`,
      clears the internal token.
    - `setAuthToken(token: string | null): void` — manual control
      for rehydration from `sessionStorage` (or future cookies).
    - `request<T>` now reads the closure-held token and the
      `Authorization` header; reads the body as text first so a
      204 / empty response doesn't trip `response.json()`.

- **BFF** (`@dt/bff`):
    - New `MockAuthStore` (`apps/bff/src/auth/mock-store.ts`).
      Deterministic, in-memory: any well-formed email returns a
      user with the `viewer` role and a `mock-<uuid>` token with
      a 1h `expiresAt`. Implements an `AuthStore` interface so V3
      can swap in a real provider without touching the route
      module.
    - New `authRoute(store)` (`apps/bff/src/routes/auth.ts`)
      exposing `GET /me`, `POST /login`, `POST /logout`, and the
      `GET /_protected` demo target. Bearer extraction is
      case-insensitive; missing or malformed bodies return 400
      `AUTH_INVALID_CREDENTIALS`; `AuthError` from the store
      becomes a 401 with the same `error` / `message` shape the
      rest of the BFF uses.
    - New `requiresPermission(store, perm)` middleware
      (`apps/bff/src/middleware/requires-permission.ts`). Reads
      the bearer token, calls `store.getMe`, computes
      `permissionsFor(roles)`, and returns 401 `AUTH_SESSION_EXPIRED`
      / 403 `AUTH_FORBIDDEN` before setting
      `c.var.user` / `c.var.permissions`. The middleware
      `declare module 'hono'` block augments
      `ContextVariableMap` so downstream handlers get typed
      `c.get('user')` / `c.get('permissions')`.
    - vitest introduced: `apps/bff/vitest.config.ts` (mirrors
      `@dt/observability`'s node env, picks up
      `src/**/*.{test,spec}.ts`); `pnpm test` is now
      `vitest run` instead of the V1 echo placeholder.

- **App shell** (`@dt/app-shell`):
    - New `useAuthStore` Pinia setup store
      (`src/stores/auth-store.ts`) with the state machine
      `anonymous ⇄ authenticated → expired`. The token is
      persisted to `sessionStorage` under `dt:auth:token` so a
      page reload keeps the session within the tab; closing the
      tab clears it. `login(email)` calls
      `client.login({ email })`, stores the token, sets
      `state.kind = 'authenticated'`. `logout()` calls
      `client.logout()`, clears the storage entry and the
      client-side token, returns to `anonymous`. `refresh()`
      rehydrates from `sessionStorage`: with a stored token it
      calls `getMe()`; on a hit, `state` becomes `authenticated`;
      on a miss, it drops the storage entry and downgrades to
      `expired` (preserving the prior `user` when available) so
      the UI can prompt for a fresh login.
    - A `permissions: Ref<Permission[]>` is derived from `state`
      via `watch(state, ..., { immediate: true })`; consumers get
      a flat reactive list without recomputing the join on every
      read.
    - `useCurrentUser(): ComputedRef<User | null>` returns the
      user when `authenticated`, the last-known user when
      `expired`, and `null` when `anonymous`.
    - `usePermission(perm: Permission): ComputedRef<boolean>`
      reads `store.permissions` so it's reactive to `login`,
      `logout`, and `refresh()`.
    - `bootstrapAppShell` was updated to call `authStore.refresh()`
      on mount. The plan's draft referenced
      `app._context.config.globalProperties.$pinia`; the
      implementation instead captures the `createPinia()` instance
      explicitly and passes it to `useAuthStore(pinia)` so the
      hydration runs against a known, fresh store regardless of
      how the host app has configured itself.

- **Smoke** (`scripts/smoke-v2.sh`): inserts a 3-call auth flow
  before the existing `requestId → log-line` assertion. Uses a
  small Node one-liner to pull the token out of the login JSON
  without pulling in `jq` as a build-time dependency. The
  post-logout `/me` reuses the now-invalidated token and asserts
  the response contains `"session":null`.

- **Build / CI**: no new GitHub Action. The V2.0 `ci.yml` covers
  the new test work via `pnpm test` (now `vitest run` in the BFF
  as well) and the smoke step runs the extended script as-is.
  The "Run V2 smoke" step is unchanged.

## What V2.1 explicitly does not contain

- Real auth provider (OAuth, SAML, JWT signing/verification).
  `MockAuthStore` is the only implementation; the `AuthStore`
  interface exists for V3 to swap in a real one.
- Multi-tenant workspace model (V3, spec #7 in
  `digital-twin-platform-codex-dev-doc.md`).
- Audit log events (V3, spec #8).
- **Permission enforcement on existing routes.** `/api/devices`,
  `/api/scene`, `/api/commands`, and `/api/stream` keep their
  V2.0 open access. The middleware ships and is exercised by
  the new `/api/auth/*` routes (login → me) plus the
  `/api/auth/_protected` demo, but it is not wired into the
  pre-existing surfaces. A follow-up ADR (likely V2.2) should
  decide whether plugin runtime enforces permissions at all,
  and if so on which routes.
- Session persistence beyond `sessionStorage`. The token is
  returned but storage is local to the tab; cross-tab sharing,
  cookies, and refresh tokens are V3 territory.
- Login form UX, password handling, MFA, account recovery —
  the mock store accepts any well-formed email and assigns
  `viewer` so the demo "just works".

## Known limitations at V2.1 closure

- **Existing routes stay open.** As called out above,
  `requiresPermission` is mounted in front of `/api/auth/_protected`
  only. A consumer who only ever calls `/api/auth/*` is fully
  covered; a consumer who calls `/api/devices` directly can do so
  without any auth check. V2.2 will decide whether to gate the
  existing surfaces and on what cadence.

- **Mock store only assigns `viewer`.** Admins and operators can
  be created by direct mutation in tests, but no production path
  promotes a user. Anything beyond a smoke test needs an admin
  override for now. A V2.2 follow-up can add a header- or
  env-driven role override for the dev BFF.

- **`usePermission` is a local cache of the in-memory permissions
  list.** It is reactive against `login` / `logout` / `refresh()`
  but it does not re-check the server on every navigation. If a
  server-side role change matters for a specific flow, the
  caller should `await authStore.refresh()` first. V2.2 will
  revisit when plugin UI starts gating whole panels.

- **OPTIONS preflight on BFF routes still returns 404.** Carried
  over from the V2.0 known limitations; the auth work didn't
  touch the CORS shape. Still a V2.2 / V2.3 follow-up. Desktop
  webview's GETs still succeed; the noise in the logs is
  cosmetic.

- **Smoke assumes `NODE_ENV=development`.** Same constraint as
  V2.0 — the mock source and the mock auth store are
  always available in dev, and the smoke forces
  `NODE_ENV=development` to keep it that way. Production
  behavior is exercised by `pnpm build`, not the smoke.

- **No two-browser realtime UI test.** Still unaddressed; the
  auth work doesn't change this. Belongs to a later V2.x.

## Consequences

- `@dt/contracts` grew a new public surface (the auth types and
  `permissionsFor`). Consumers that take a `Role` from a server
  payload can now narrow it via the same union the BFF and
  plugins use, with no duplicated string literals.
- `@dt/api-client` grew 4 methods. The token model is a single
  `string` (no refresh token); V3 can extend `AuthSession`
  without breaking V2.1 callers because `token: string` is
  already the field they reach for.
- Existing V1/V2 routes are unchanged. A V2.0 consumer that
  calls `client.getDevices()` continues to work with no auth
  header; the only behavioral change is the addition of the
  `Authorization` header when a token is set, which the BFF
  ignores when the route doesn't read it.
- The release-please pipeline produced a v2.1.0 release; the
  manifest is now at `2.1.0` and the next `feat:` will bump to
  `2.2.0`.
- The plugin runtime (V2.2) can now ask "does the current user
  have permission X?" via `usePermission` or by importing
  `permissionsFor` directly, with no further V2 work.

## Revisit when

- V2.2 begins. At that point, this ADR is no longer the

  `verified` and reference this ADR in the commit.
