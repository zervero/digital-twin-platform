# ADR 0014 - V3.1 Closure (Track G: Production Platform)

## Status

Accepted (V3.1). Records the moment V3.1 ships, so V3.2
(Track H: Tauri release pipeline) has a single auditable
artifact to point to when asking "what was V3.1?".

## Context

ADR 0012 defined the V3 roadmap: 5 tracks (F real auth,
G production platform, H Tauri releases, I multi-tenant,
J plugin marketplace) across 5 releases (V3.0 -> V3.4).
V3.0 shipped Track F (real auth / OIDC); see ADR 0013.

V3.1 is Track G: replace the V2.3 compose-based deployment
story with a real production platform -- a Helm chart that
provisions BFF + web + (optionally) Ingress + cert-manager
+ OpenTelemetry on Kubernetes. The V2.3 closure ADR
explicitly lists "Kubernetes manifests / Helm chart",
"TLS termination, ACME, cert provisioning", and "Metrics
endpoint, OpenTelemetry traces" under "What's not in V2.3";
V3.1 closes all three.

The V3.1 plan covers:

- A new `@dt/otel` package that owns the OpenTelemetry
  NodeSDK lifecycle (start / shutdown / kill-switch) so the
  BFF can wire OTLP traces + metrics without pulling the
  ~30 SDK packages into the BFF's dependency surface.
- A bootstrap refactor in the BFF: `index.ts` -> `bootstrap.ts`
  -> `server.ts` so the OTel SDK registers its require-hook
  BEFORE `hono` / `ws` are loaded (auto-instrumentation
  only sees modules that load after the SDK starts).
- Two env-var contract files (`apps/bff/.env.example.otel`,
  `apps/web/.env.example.otel`) that document what the BFF
  reads at runtime. Deliberately NOT wired through
  `@dt/config`'s `AppEnv` because the OTel spec is a flat
  `OTEL_*` namespace and forcing it through the typed
  config layer would either duplicate the spec or drift
  from it.
- A Helm umbrella chart at `tooling/k8s/digital-twin-platform/`
  with two subcharts (`bff`, `web`), plus cert-manager
  resources gated by `global.certManager.enabled`. The
  chart produces 10 resources by default and 13 with the
  cert-manager overlay; both counts are asserted by
  `kubeconform -strict` in CI.
- Production / staging / OIDC value overlays so operators
  don't have to hand-write `values.yaml` from scratch.
- A CI `chart` job that installs helm 4.2.2 + kubeconform
  0.6.7 from upstream releases and runs the strict lint.
  The chart job is now a required check on `main`, alongside
  the existing `verify` job.
- A new `docs/development/production-platform.md` that
  captures the install / TLS / OTel / operations runbook.

`docs/plans/v3.1-implementation-plan.md` decomposes V3.1
into 8 bite-sized tasks with exact files, exact code, exact
tests, and exact verification commands. The plan resolves
10 open design questions (otel vs otel/sdk-node, config
shape, version compatibility, import order, secret
material, chart structure, values validation, ingress
shape, OIDC example, kubeconform CRD handling) up front
so the implementation is mechanical, not exploratory.

This ADR records that all 8 tasks shipped and that the 10
acceptance items are verified.

## Decision

V3.1 is **shipped** as `digital-twin-platform@4.1.0`
(effective version, despite the release-please artifact
documented in the CHANGELOG footnote; see "Deviations"
below). The 8 implementation tasks and the 10 acceptance
items below are verified as of the commit timeline at the
bottom of this ADR.

### Deviations from the plan

Three items in the plan were deliberately adjusted during
implementation:

1. **`@opentelemetry/resources` was pinned to `1.30.x`
   instead of the `2.x` shown in the OTel SDK docs.**
   During T1 we discovered that `@opentelemetry/sdk-node@0.57.0`
   internally bundles `@opentelemetry/resources@1.30` and that
   the 1.x `Resource#merge()` expects the 1.x `Resource`
   class. The 2.x `resourceFromAttributes()` returns an
   interface-shaped object that doesn't interoperate with
   the 1.x class, so sdk-node silently dropped the
   `service.version` and `deployment.environment.name`
   attributes that `@dt/otel` carefully builds. The fix in
   commit `e26e204` switches `buildOtelResource()` to
   `new Resource({...})` so the shape matches what sdk-node
   wants. The dependency pin (`^1.30.0`) keeps pnpm from
   resolving to the 2.x line.
2. **The cert-manager values key was renamed `certManager`
   (camelCase) instead of `cert-manager` (kebab-case).**
   Helm's template parser treats the dash in a dotted
   identifier (`.Values.cert-manager.enabled`) as an action
   modifier and fails the parse. This is the same kind of
   quirk that motivates kebab-case file names but camelCase
   values keys throughout the helm ecosystem. The example
   overlay is `values-certmanager-example.yaml`; the CRD
   group stays `cert-manager.io` (that's an HTTP URL, not a
   helm expression).
3. **Cross-chart values were placed under
   `.Values.global.{ingress, certManager}` instead of at the
   umbrella's top level.** The Ingress lives in the `web`
   subchart; the ClusterIssuer + Certificate live at the
   umbrella level. Subcharts can only see umbrella-level
   values via `global:`. Without this, `helm template` fails
   with "nil pointer evaluating interface {}.enabled".

Plus one non-deviation worth recording:

4. **Release-please emitted two consecutive release PRs
   (#12 = `3.1.0`, #13 = `4.0.0`) for the same set of V3.1
   work.** PR #12 merged with an empty body, which apparently
   didn't update release-please's "consumed commits" state.
   PR #13 then re-scanned from `3.0.0` and re-emitted the V3.0
   `BREAKING CHANGE` footers as a fresh major bump. PR #14
   (`4.1.0`) is the correct minor bump for T5-T6. A
   CHANGELOG footnote (commit `3237833`) points readers at
   `3.0.0` for the actual breaking change and at
   `3.1.0` / `4.1.0` for the actual V3.1 feature work. The
   effective V3.1 version is `4.1.0`. No tag or release
   artifact was deleted. Future work (V3.2) will go on top
   via release-please as normal; if the state-machine bug
   recurs, we'll consider switching to release-please v18+
   or manual version management.

None of items 1-3 change the public surface or the
acceptance matrix. All three are documented in code comments
in the relevant files:

- `@opentelemetry/resources` 1.x API: `packages/otel/src/index.ts`
- `certManager` rename: `tooling/k8s/digital-twin-platform/templates/cluster-issuer.yaml`
- `global:` namespace: `tooling/k8s/digital-twin-platform/values.yaml`

## Consequences

- `@dt/otel` becomes a permanent dependency of the BFF.
  Its public surface (`startOtel`, `shutdownOtel`,
  `isOtelDisabled`, `buildOtelResource`, `OtelConfig`) is
  the canonical place to evolve OTel semantics in V3.x.
- The Helm umbrella chart is the new supported production
  deployment shape. The V2.3-era `docker-compose.yml`
  contract is now considered legacy; `apps/{bff,web}`
  Dockerfiles remain for dev-loop convenience but are no
  longer the recommended path.
- TLS termination is delegated to cert-manager via the
  Ingress-shim annotation + an explicit `Certificate` CRD
  as a belt-and-suspenders fallback. Operators must install
  cert-manager themselves; the chart only consumes the CRDs.
- OIDC credentials remain outside the chart. The chart
  references an externally-created `Secret` by name; the
  operator provisions it via `kubectl create secret`,
  sealed-secrets, or external-secrets.
- OTel traces + metrics start flowing from the BFF on boot
  and are flushed as the last step of graceful shutdown.
  The V2.3 graceful-shutdown contract (drain WS, flip
  `/ready` to 503, close HTTP, exit) is preserved and
  extended with one extra step.
- The lefthook `pre-push` hook now runs the full pre-merge
  gate locally. CI remains the final authority; the local
  hook just saves the round trip when something is clearly
  broken.
- `docs/development/deployment.md` is now mostly a historical
  document about the V2.3 compose shape. The canonical
  production guide is `docs/development/production-platform.md`.

## What's next

V3.2 is Track H (Tauri release pipeline): signed `.dmg` /
`.msi` / `.AppImage`, auto-update channel, GitHub Releases
as the artifact host. Track H reuses G's signing-key
infrastructure (added in a hypothetical V3.1.x follow-up)
and G's TLS-aware download mirror. The V3.2 plan will land
in `docs/plans/v3.2-implementation-plan.md` and follow the
same 8-task decomposition that V3.0 and V3.1 used. Track
H is largely independent of F/G/I/J at the code level but
ships after G so the desktop release CI reuses G's
release-please workflow and image-signing infrastructure.
