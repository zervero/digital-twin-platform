# ADR 0018 - V3.5 Track K: Internationalization (i18n)

## Status

Accepted (V3.5 implementation starting). Records the addition
of an i18n track to the V3 era, addressing the visible UI
quality gap that became obvious during the v4-prep visual
refresh.

## Context

The V3 roadmap (ADR 0012) closed at V3.4 with five tracks
(F real auth, G production platform, H Tauri releases,
I multi-tenant, J plugin marketplace) shipped across
V3.0 -> V3.4. None of the five tracks owned UI copy
quality; they each shipped one feature, and the chrome
around it inherited whatever strings the implementing
engineer typed first.

After V3.4 closed, the v4-prep visual refresh (PR #18,
v4.4.0) made the gap loud. The page now has `设备`,
`加载中…`, `清除选择`, `退出` living next to `Install`,
`Activate`, `Uninstall`, `Loading installed plugins...`,
`Plugin marketplace` in the same panels. The mixed
copy:

- Reads as a prototype to a non-Chinese-speaking operator.
- Reads as un-translated to a Chinese-speaking operator.
- Costs us the "operator product" positioning the v4-prep
  visual refresh was meant to establish.

This is small enough not to deserve its own V4 track, but
large enough that shipping V4 on top of it would carry the
baggage forward. The decision is to add a sixth V3-era
track, **Track K: i18n**, ship it as V3.5, and only then
open V4 planning.

## Decision

Add a new package `@dt/i18n` and migrate every user-facing
string in `@dt/app-shell` to it. The plan, scope, and
non-goals are in
[`docs/plans/v3.5-implementation-plan.md`](../plans/v3.5-implementation-plan.md);
this ADR fixes the design choice that the plan executes
against.

### Why a self-hosted i18n package, not `vue-i18n`

Considered:

- **`vue-i18n`**: industry-standard, supports ICU
  MessageFormat, pluralization, lazy locale loading. Adds
  ~15KB to the app-shell bundle, requires a Vue plugin
  mount (`app.use(i18n)`), and historically needs a
  build-time transform for `<i18n-t>` blocks. The project
  has consistently refused similar "default Vue tooling"
  additions (we picked Pinia over Vuex, esbuild over
  Vite-plugin i18n, vanilla SFC over Nuxt) when the
  zero-build library surface can carry the same shape.
- **`@intlify/unplugin-vue-i18n`**: same library, adds a
  Vite plugin so `<i18n-t>` blocks compile-time-check.
  Useful for ICU plural forms; not justified for ~18
  strings across two locales.
- **Self-hosted JSON + Pinia store + composable**: ~80
  lines of glue code, no runtime dep beyond `vue` and
  `pinia` (both already present), key-based `t('xxx')`
  calls that typecheck against the dictionary. Per AGENTS.md
  "no new build-time dep direction", this is the smallest
  delta that satisfies the requirement.

The interfaces are intentionally narrow:

- `useI18n()` returns `{ t, locale, setLocale, available }`.
- `t(key, vars?)` does plain string interpolation via
  `{name}`-style placeholders. No plural forms, no date /
  number formatting. Those are V3.5.x follow-ups if a UI
  surface ever needs them.

### Locale surface

Two locales ship: `en` (default), `zh-CN`. The locale
picker exposes three states to the user:

- `system` — follow `navigator.language`, fall back to `en`
- `en`
- `zh-CN`

The chosen locale persists in `localStorage` under the key
`dt.locale.v1`. The `v1` suffix lets us rename the storage
shape later without colliding with an old value. On first
load, `system` is the default; the picker remembers the
last explicit choice forever.

### Workspace boundary

New package `@dt/i18n` has these dependencies and only these:

- `vue` (peer, for `inject()` and reactivity)
- `pinia` (peer, for the locale store)
- `@dt/contracts` (workspace, for nothing yet — present so
  future locale keys can reference `Permission` union
  values without an extra dep direction)

`@dt/i18n` does NOT depend on `@dt/app-shell`, `@dt/ui-kit`,
`@dt/api-client`, `@dt/bff`, or any `apps/*`. This keeps
the i18n package reusable in any future app (Tauri desktop,
a CLI, a server-rendered marketing page) without dragging
in the app-shell.

`@dt/app-shell` gains `@dt/i18n` as a direct dependency.
This follows the same "consumer declares direct dep"
pattern documented in `docs/architecture/ui-kit-tokens.md`
("Consumer wiring" section): the app-shell's Vite/Rollup
build resolves `@dt/i18n` consistently in dev (esbuild)
and prod (Rollup) only when both packages share the same
`node_modules/@dt/i18n` symlink target.

`@dt/ui-kit` does NOT gain `@dt/i18n`. The presentational
library stays copy-free by convention. Any future ui-kit
component that ships text (e.g. `<DtEmptyState>`) will be
evaluated as a separate ADR; the default answer is "no".

### Strings that do not get translated

Some strings are intentionally left as-is. Codifying this
list keeps future contributors from over-translating:

- The product brand "Digital Twin" (preserved as written
  in every locale; localization trademark guidance).
- The version chip (`4.4.0` and successors).
- Permission identifiers (`plugin:publish`, `plugin:install`,
  `plugin:read`) — these are API contract values, not UI
  copy. They appear inside `<code>` blocks in the
  marketplace panel.
- Status badge text that mirrors an API enum value
  (`active` in the marketplace version list) — kept in
  English to match the contract value operators see in
  logs and in the install API response.
- Field placeholders that are also valid input examples
  (`hello-plugin`, `1.0.0`) — these teach the user the
  expected input format. Replacing with translations
  would break the example.

These rules ship as a "do not translate" callout in
`docs/development/i18n.md` (T6 of the plan) and as a
gating test: a `forbidden-keys.test.ts` ensures that the
copy-side and the code-side agree on which keys exist in
the dictionary and which strings stay literal.

## Consequences

- V3.5 closes the visible V3-era UI quality gap. After
  V3.5 lands, every user-facing string in app-shell
  resolves through `@dt/i18n`. The page is either fully
  English or fully Chinese (plus the brand and code
  constants), never the current mix.
- V3.5 ships as a single release-please version bump.
  Expected effective version: `4.5.0` (`feat(i18n):` is
  a minor bump under ADR 0014's artifact-aware manifest).
- A future V4 plan does not need to plan i18n — it can
  assume `@dt/i18n` exists and call `useI18n()` from
  any new component.
- The dictionary type (`as const` JSON -> deep key union)
  costs a few hundred milliseconds of typecheck when
  growing past ~200 keys. The V3.5 plan stays well under
  that ceiling; future growth is a follow-up concern.

## Revisit when

- Locale count passes 4 (e.g. add `ja`, `ko`, `de`,
  `fr`). Consider lazy-loading the dictionary.
- Any UI surface needs plural forms, date formatting,
  number formatting, or RTL layout. Promote to
  `vue-i18n` and rewrite `useI18n` as a thin facade.
- `@dt/ui-kit` ships a component with user-visible text
  (the current zero-copy contract breaks). Revisit the
  ui-kit -> i18n dep direction.
