# ADR 0010 - V2.2 Closure

## Status

Accepted (V2.2). Records the moment V2.2 (plugin runtime) is
shipped, so V2.3 (production deployment) has a single auditable
artifact to point to when asking "what was V2.2?".

## Context

ADR 0007 defined the V2 roadmap: 5 tracks (realtime, plugin
runtime, observability, auth, production deploy) split across 4
releases (V2.0 - V2.3). `docs/plans/v2-overview.md` ordered them
A+C → D → B → E so realtime and auth land before plugin runtime,
and the plugin runtime ships before production deployment.

V2.0 (Tracks A+C, realtime + observability) shipped on 2026-07-06
as `digital-twin-platform@2.0.0` (see [ADR 0008](./0008-v2-closure.md)).
V2.1 (Track D, auth contracts) shipped on 2026-07-06 as
`digital-twin-platform@2.1.0` (see [ADR 0009](./0009-v2.1-closure.md)).

V2.2 is Track B: a plugin runtime that lets partners extend the
platform without forking the codebase. The motivation in
`docs/plans/v2-overview.md` is that the V1/V2.1 architecture had
`@dt/plugin-runtime` as an interface and a mock implementation
only; the surface is now real, with a host-agnostic runtime, a
Pinia store, composables, and a working sample plugin exercising
the extension points on every dev run.

`docs/plans/v2.2-implementation-plan.md` decomposed V2.2 into 6
bite-sized tasks with exact files, exact code, exact tests, and
exact verification commands. The plan answers 8 open design
questions (loading model, sandboxing, extension surface,
permission model, activation lifecycle, multiple plugins per
host, sample plugin, smoke scope) up front so the implementation
is mechanical, not exploratory.

This ADR records that those 6 tasks shipped and that the 9
acceptance items are verified.

## Decision

V2.2 is **shipped** as `digital-twin-platform@2.2.0`. All 9
acceptance items below are verified on the host and on CI as of
the release commit.

## What V2.2 actually contains

- **Runtime** (`@dt/plugin-runtime`): a host-agnostic, Vue-free
  plugin runtime. The package's public surface is:
    - `PluginManifest`: `{id, name, version, vendor, description?, permissions: Permission[]}` where `Permission` is the V2.1 union from `@dt/contracts`. The V1 `PluginPermission` alias is removed; the runtime re-exports `PluginPermission = Permission` for back-compat.
    - `validatePluginManifest(input: unknown): {ok:true, manifest} | {ok:false, errors}`. Pure function, no side effects, no env reads. `errors` is `PluginManifestError[]` with `code ∈ 'INVALID_ID' | 'INVALID_VERSION' | 'UNKNOWN_PERMISSION' | 'MISSING_FIELD'` and a `field` path for structured error reporting.
    - `PluginComponent`: a structural type (`render?: Function; setup?: Function; [key: string]: unknown`). Deliberately not `import type { Component } from 'vue'` — the workspace rule says no Vue in plugin-runtime. A real Vue `defineComponent` is assignable to this shape; the host's `<component :is="..." />` does the work of constructing it.
    - `PluginExtension` discriminated union: `{kind:'ui-panel', panel} | {kind:'menu-item', item} | {kind:'event-subscriber', subscriber}`. `provide:commands` is reserved in the type union but not surfaced in V2.2.
    - `PluginContext`: `{grantedPermissions: readonly Permission[], subscribe: (fn) => () => void}`. The runtime does not import `@dt/api-client` or `@dt/realtime`; the host satisfies this shape with the real implementations.
    - `createPluginRegistry()`: returns a `PluginRegistry` with `register`, `unregister`, `activateAll(ctx)`, `deactivateAll()`, `list()`, `get(id)`, `subscribe(fn)`. Per-plugin state machine: `inactive → activating → active → deactivating → errored`. Sequential activation, LIFO deactivation, failures do not block other plugins.
    - `PluginActivationError`: `{code: 'PERMISSION_DENIED' | 'ACTIVATION_FAILED', message, missingPermissions?}`. `missingPermissions` is populated when a plugin is blocked for missing grants so the host can render a meaningful error.

- **App shell integration** (`@dt/app-shell`):
    - `usePluginStore` Pinia setup store. Holds `registry` (`ShallowRef`), `entriesRaw` (`Ref`), `stream` (`ShallowRef<RealtimeStream>`). Exposes `entries`, `panels`, `menuItems` as `ComputedRef`s (Pinia auto-unwraps the inner refs). `setRegistry`, `attachRealtime`, `activateAll`, `deactivateAll` are the actions.
    - `attachRealtime(s)` wires `s.subscribe(event => ...)` to fan out to all active `event-subscriber` extensions. `activateAll(granted)` builds the `PluginContext` with `subscribe: fn => stream.value?.subscribe(fn) ?? noop`. Plugins that call `ctx.subscribe` during `activate` get a working subscription or a clean no-op, with no special-casing at the call site.
    - `bootstrapAppShell({apiClient, host, plugins?, realtime?})`. Order: `createApp(AppShell)` + `createPinia()` + `app.use(pinia)` + `provideApiClient(app, apiClient)` → `authStore.refresh()` → `createPluginRegistry()` + `register(...)` for each plugin + `setRegistry(registry)` + `attachRealtime(realtime)` + `activateAll([...authStore.permissions])` → `app.mount(host)`. Activation order matters: auth hydration must finish before plugin activation so the permission gate has current-user state; the realtime stream is attached before activation so plugins that `ctx.subscribe` during `activate` receive a working subscription.
    - `AppShell.vue` renders a sidebar `__plugins` div: `<PluginPanelHost v-for="panel in panels" :panel="panel" />` and errored plugins as `<section class="plugin-panel plugin-panel--errored">`. Toolbar renders `<button v-for="item in menu">` when `menu.length > 0`.
    - `PluginPanelHost.vue` is a small wrapper: `<component :is="panel.component" />` inside a labelled box. The host does not know what a plugin's component looks like; the runtime's structural `PluginComponent` type is all the host needs to mount it.
    - The store wraps each published panel with `markRaw(toRaw(...))` so Vue's component renderer does not log a "Component was made reactive" warning. The component is reached through a reactive array, so a naive `markRaw` would only flag the reactive proxy; unwrapping first lets the renderer see the raw target.

- **Sample plugin** (`apps/web/src/plugins/hello/index.ts`):
    - `helloPlugin: PluginRegistration` with manifest `{id:'hello', name:'Hello Plugin', version:'1.0.0', vendor:'@dt/samples', description:'Sample plugin that proves the extension-point wiring.', permissions:['auth:login']}`. Requires `auth:login` so the permission gate has something to enforce.
    - `activate(ctx)` returns one `ui-panel` extension with a `HelloPanelHost` component that renders "Hello, {displayName}!". Falls back to "friend" when the activation context says the user is permitted (always true once `auth:login` is granted).
    - `apps/web/src/main.ts` wires `helloPlugin` into the V1 sync-mount path. The T5 work avoided the dual-pinia path by reusing the host's `createPinia()` instance and calling the stores directly, then mounting `#app` (the V1 pattern) instead of `opts.host` (the `bootstrapAppShell` pattern used by tests).

- **Smoke** (`scripts/smoke-v2.sh`): inserts a `node --import tsx` block before the request-id log assertion. Runs `validatePluginManifest` against the hello plugin's manifest literal, asserts `{ok:true, manifest.id === 'hello'}`. Runs against the package's source (not its `dist`, which is a `gitkeep` placeholder) so no build step is needed. `tsx` and `@dt/plugin-runtime` are added as root devDependencies so the one-liner can resolve both the loader and the workspace package from the script's working directory.

- **Workspace boundary update** (`docs/architecture/workspace.md`): the `app-shell` boundary now lists `realtime, plugin-runtime` as allowed imports, with inline notes about why each is allowed (Pinia + Vue for app-shell, structural types only for plugin-runtime). The plugin-runtime side stays at `contracts`-only.

- **Build / CI**: no new GitHub Action. The V2.0 `ci.yml` covers the new test work via `pnpm test` and the smoke step runs the extended script as-is. The "Run V2 smoke" step is unchanged.

## What V2.2 explicitly does not contain

- **Real plugin sandboxing.** Plugins are loaded as same-origin
  modules; the trust model is "operator reviewed". V2.2 ships
  the contract surface; a future V2.x or V3 release can add
  `iframe` / `Worker` / signed-manifest isolation. Documented
  in `docs/architecture/workspace.md` so the boundary is
  visible without spelunking the plan.
- **URL-based dynamic plugin loading.** The host passes
  `readonly PluginRegistration[]` to `bootstrapAppShell`; each
  registration is a direct import (e.g. `import { helloPlugin }
  from './plugins/hello/index.js'`). No remote fetch, no
  `import()` of URLs, no `eval`. A V3 plugin marketplace
  would extend the contract.
- **Plugin commands (`provide:commands` extension).** Reserved
  in the type union but not surfaced in V2.2. The discriminated
  union only has `ui-panel`, `menu-item`, and
  `event-subscriber`.
- **Permission enforcement on existing routes.** Carried over
  from V2.1 — `/api/devices`, `/api/scene`, `/api/commands`,
  and `/api/stream` keep their open access. The plugin
  permission gate is a runtime concept (whether a plugin
  activates), not a BFF route concept. A V2.3 follow-up
  should decide whether to wire the V2.1 `requiresPermission`
  middleware into the pre-existing surfaces.
- **Multi-tenant workspace model.** V3, spec #7 in
  `digital-twin-platform-codex-dev-doc.md`.
- **Audit log events for plugin lifecycle.** V3, spec #8.
  `PluginRegistry.subscribe(fn)` emits state changes; a
  consumer could wire this to `@dt/observability` to log
  activations, but the default is silent.
- **Plugin configuration / settings UI.** Plugins ship with a
  manifest and a static `activate`; there is no admin UI to
  enable/disable, configure, or update them. The V2.2 host
  decides which registrations to pass; that's the entire
  config surface.
- **Plugin-side data fetching.** `PluginContext` is a
  structural type with `grantedPermissions` and `subscribe`.
  It does not expose the `ApiClient`. A plugin that wants to
  fetch devices must do so through its own import (e.g.
  `import { createApiClient } from '@dt/api-client'`). The
  contract is intentionally narrow; V3 can extend it.

## Known limitations at V2.2 closure

- **Plugin state is in-memory only.** `createPluginRegistry`
  holds entries in a `Map`; `usePluginStore` holds them in a
  Pinia store. There is no persistence. A page reload resets
  the registry, the activation order, and the errored
  plugins. The V1 single-page-app model is OK with this; a
  future V2.x that ships a plugin marketplace will need
  persistence.

- **Permission gate is host-side, not BFF-side.** A plugin
  that imports `@dt/api-client` directly and calls
  `client.getDevices()` does so with no permission check
  unless the BFF route enforces one (it doesn't, per V2.1's
  known limitations). The V2.2 permission gate stops a plugin
  from **activating**; it does not stop a malicious plugin
  from calling the BFF directly. The trust model is
  "operator reviewed"; document the boundary clearly in any
  third-party docs.

- **Sample plugin requires `auth:login` but the dev BFF mock
  assigns `viewer` role.** `viewer` has `auth:login` in
  `ROLE_PERMISSIONS`, so the gate is satisfied and the panel
  renders. A user with a different role would still see the
  panel because the BFF's mock store always assigns `viewer`.
  A V2.3 follow-up can add a header- or env-driven role
  override for the dev BFF.

- **`markRaw(toRaw(...))` is needed to silence Vue's "Component
  was made reactive" warning.** The component is reached
  through a reactive array (the `entriesRaw` ref's
  `extensions`), so a naive `markRaw` would only flag the
  reactive proxy; unwrapping first lets the renderer see the
  raw target. Documented in the store so the next person to
  touch the panel pipeline does not undo it.

- **`packages/app-shell/src/__tests__/bootstrap-plugins.test.ts`
  stubs `SceneViewport.vue`.** The real `SceneViewport`
  constructs a WebGL context that happy-dom does not provide.
  The stub is a `defineComponent` with a `<div />` template;
  the test asserts on the store, not the rendered DOM, so the
  stub is sufficient. A future move to jsdom-with-WebGL or
  Playwright component testing could drop the stub.

- **`scripts/smoke-v2.sh` adds `tsx` and `@dt/plugin-runtime`
  to root devDependencies.** Both are real deps of the smoke
  script; the runtime is consumed as a workspace symlink (no
  build step in the smoke). The lockfile tracks the change.

- **Options preflight on BFF routes still returns 404.**
  Carried over from V2.0 and V2.1 known limitations; the
  plugin work didn't touch the CORS shape. Still a V2.3
  follow-up. Desktop webview's GETs still succeed; the noise
  in the logs is cosmetic.

- **Smoke assumes `NODE_ENV=development`.** Same constraint as
  V2.0 and V2.1 — the mock source and the mock auth store
  are always available in dev, and the smoke forces
  `NODE_ENV=development` to keep it that way. Production
  behavior is exercised by `pnpm build`, not the smoke.

## Consequences

- `@dt/plugin-runtime` is a real, documented package. A
  consumer can import `validatePluginManifest`,
  `createPluginRegistry`, and the extension types without
  reading the source.
- `@dt/app-shell` grew a `usePluginStore` and 4 new
  composables (`usePluginPanels`, `usePluginMenu`,
  `useCurrentUser`, `usePermission`). The bootstrap helper
  now has a `plugins?` parameter; passing it is the only
  thing a host needs to do to enable the extension points.
- `apps/web` ships with `helloPlugin` wired in by default.
  Removing the `plugins:` line ships a plugin-free web
  build. The V1 sync-mount path is preserved.
- The release-please pipeline produces a v2.2.0 release; the
  manifest is now at `2.2.0` and the next `feat:` will bump
  to `2.3.0`.
- The workspace rule that "domain packages may not import
  vue or three" is unchanged. `@dt/plugin-runtime` stays
  host-agnostic; the Vue-specific work lives in
  `@dt/app-shell`.

## Revisit when

- V2.3 begins. At that point, this ADR is no longer the
  "current" boundary; the V2.3 plan becomes the source of
  truth, and this one moves to "history" alongside ADR 0008
  and ADR 0009.
- A new extension kind lands (`provide:commands`, etc.). Add
  a follow-up ADR that records the contract change and bumps
  the version.
- The verification table becomes inaccurate (e.g. a follow-up
  change regresses one of the 9 items). Promote items to
  `verified` and reference this ADR in the commit.
