# V3.5 Implementation Plan (Track K: Internationalization)

> Scope: ship a self-hosted `@dt/i18n` package, migrate
> every user-facing string in `@dt/app-shell` to use it,
> and surface a language switcher in the toolbar. Closes
> the V3-era UI copy gap that the v4-prep visual refresh
> (v4.4.0) made visible. After V3.5, every UI surface is
> either fully localized or intentionally literal (brand,
> version, permission identifiers, input examples).
>
> This is the sixth V3-era track. Track F (real auth /
> OIDC) shipped as V3.0 ([ADR 0013](../adr/0013-v3.0-closure.md)),
> Track G (production platform) shipped as V3.1
> ([ADR 0014](../adr/0014-v3.1-closure.md)), Track H
> (Tauri release pipeline) shipped as V3.2
> ([ADR 0015](../adr/0015-v3.2-closure.md)), Track I
> (multi-tenant data model) shipped as V3.3
> ([ADR 0016](../adr/0016-v3.3-closure.md)), and Track J
> (plugin marketplace + persistence) shipped as V3.4
> ([ADR 0017](../adr/0017-v3.4-closure.md)). Track K
> depends on no other V3 track — it is pure UI plumbing —
> and is the V3-era last track. V4 planning starts after
> V3.5 closes.
>
> The architectural decisions for this track (self-hosted
> i18n vs `vue-i18n`, locale surface, workspace boundary,
> what stays untranslated) live in
> [ADR 0018](../adr/0018-v3.5-i18n.md).

**`dictionary-completeness.test.ts`**: loads both locale
trees, walks every key in `en`, asserts the same key
exists in `zh-CN` with a non-empty string value, and
asserts the inverse. A mismatch fails the test. This is
the gate that prevents accidental asymmetry — the
historical cause of the visible "mix" problem.

### T3 — `useI18n()` + Pinia locale store

`packages/i18n/src/store.ts`:

```ts
import { defineStore } from 'pinia';
import { ref, computed } from 'vue';

export type LocaleCode = 'en' | 'zh-CN';
export type LocaleChoice = 'system' | LocaleCode;

const STORAGE_KEY = 'dt.locale.v1';

function detect(): LocaleChoice {
  if (typeof localStorage === 'undefined') return 'en';
  const saved = localStorage.getItem(STORAGE_KEY);
  if (saved === 'en' || saved === 'zh-CN' || saved === 'system') {
    return saved;
  }
  const nav = typeof navigator !== 'undefined'
    ? navigator.language
    : 'en';
  return nav.toLowerCase().startsWith('zh') ? 'zh-CN' : 'en';
}

export const useLocaleStore = defineStore('dt.locale', () => {
  const choice = ref<LocaleChoice>(detect());
  const resolved = computed<LocaleCode>(() => {
    if (choice.value === 'system') {
      const nav = typeof navigator !== 'undefined'
        ? navigator.language
        : 'en';
      return nav.toLowerCase().startsWith('zh') ? 'zh-CN' : 'en';
    }
    return choice.value;
  });

  function set(next: LocaleChoice): void {
    choice.value = next;
    if (typeof localStorage !== 'undefined') {
      localStorage.setItem(STORAGE_KEY, next);
    }
  }

  return { choice, resolved, set };
});
```

`packages/i18n/src/composable.ts`:

```ts
import { computed } from 'vue';
import { useLocaleStore } from './store.js';
import en from './locales/en/index.js';
import zhCN from './locales/zh-CN/index.js';

type Dictionary = Record<string, unknown>;

const TABLES: Record<'en' | 'zh-CN', Dictionary> = { en, 'zh-CN': zhCN };

function lookup(dict: Dictionary, key: string): string {
  const parts = key.split('.');
  let cur: unknown = dict;
  for (const p of parts) {
    if (cur && typeof cur === 'object' && p in (cur as Dictionary)) {
      cur = (cur as Dictionary)[p];
    } else {
      return key; // missing key: return the key itself, do not throw
    }
  }
  return typeof cur === 'string' ? cur : key;
}

function interpolate(template: string, vars?: Record<string, string | number>): string {
  if (!vars) return template;
  return template.replace(/\{(\w+)\}/g, (_, k) =>
    k in vars ? String(vars[k]) : `{${k}}`);
}

export function useI18n() {
  const store = useLocaleStore();
  const table = computed<Dictionary>(() => TABLES[store.resolved]);
  function t(key: string, vars?: Record<string, string | number>): string {
    return interpolate(lookup(table.value, key), vars);
  }
  return {
    t,
    locale: computed(() => store.resolved),
    choice: computed(() => store.choice),
    setLocale: store.set,
    available: ['en', 'zh-CN'] as const,
  };
}
```

`__tests__/store.test.ts` covers: first-load detection
(localStorage empty → navigator.language path), explicit
`set('zh-CN')` round-trip via `localStorage`, `system`
fallback when `navigator.language` is neither `en*` nor
`zh*`.

`__tests__/composable.test.ts` covers: `t('device.title')`
returns the localized string under each locale, `t('nope')`
returns `'nope'` (no throw), `t('marketplace.installHint', { permission: 'plugin:publish' })`
interpolates correctly.

### T5 — Language switcher

In `TopToolbar.vue`, add a small segmented control next to
the LiveStatusDot:

```
[ System | EN | 中文 ]
```

Three buttons, the active one styled with
`--dt-accent-primary` background. State source is
`useLocaleStore().choice`. Click handlers call
`setLocale('system' | 'en' | 'zh-CN')`.

The switcher is **not** in the marketplace or device
panel — toolbar is the natural global control surface, and
the v4-prep refresh already established it as the
"operator status" row (live dot + version chip).

Reactivity check: changing the locale updates every
`t('xxx')` call site in the same tick (Vue reactivity
through the Pinia store). No page reload required.

### T7 — Entry-point wiring

`apps/web/src/main.ts`:

```ts
import { createPinia, setActivePinia } from 'pinia';
import { createI18n } from '@dt/i18n';

// existing imports ...
const pinia = createPinia();
setActivePinia(pinia);
createI18n(pinia);
app.use(pinia);
app.mount('#app');
```

`@dt/i18n` exports `createI18n(pinia)` that simply
`app.use(pinia)` and ensures the locale store is wired
before any component reads it. The actual `useLocaleStore()`
call happens lazily inside `useI18n()`, so there is no
plugin-state surprise.

`apps/desktop/src/main.ts` (the Tauri entry) imports the
same `apps/web` bootstrap because the desktop shell runs
the web bundle inside its webview; the existing pattern is
already a single boot file. Verify by reading
`apps/desktop/src-tauri/src/lib.rs` and the web entry, and
note any divergence in the closure ADR.

## Non-goals (codified, not scope-crept later)

- **No `vue-i18n`**. The decision is in ADR 0018. If
  plural forms or date formatting ever become necessary,
  promote `@dt/i18n` to a `vue-i18n` facade in a
  follow-up ADR.
- **No locale lazy-loading**. Two locales × ~25 keys each
  ships in the initial bundle. Future growth → lazy-load
  per ADR 0018 "Revisit when".
- **No RTL layout**. Not on the roadmap.
- **No `@dt/ui-kit` i18n wiring**. ui-kit stays
  copy-free by convention. Any future component with text
  is a separate ADR.
- **No renaming of API contract fields**. `plugin:publish`
  stays `plugin:publish` in every locale. The marketplace
  panel renders it inside `<code>...</code>` literally.

## Scope additions recorded after T8

### T8.1 - CORS unblocker for web/desktop cross-origin requests

**Found during T8 verification**, not part of the original
8-task plan. While exercising the i18n work in the actual
browser (Vite at `http://localhost:5173` reaching the BFF
at `http://localhost:3001`), every fetch failed with:

  Access to fetch at 'http://localhost:3001/api/scene' from
  origin 'http://localhost:5173' has been blocked by CORS
  policy: Response to preflight request doesn't pass access
  control check: No 'Access-Control-Allow-Origin' header is
  present on the requested resource.

The BFF had **no CORS middleware at all** -- a pre-existing
gap from V3.0 (Track F) that the smoke scripts never caught
because they hit the BFF on `localhost:3001` directly
without an `Origin` header. CI runs the smoke scripts on
each PR, so this regression-shaped gap survived every
release from V3.0 through V3.4.

**Fix landed on `codex/v3.5-i18n` branch (one commit
added to the existing PR #20).**

Changes:

- `apps/bff/src/server.ts` -- mounts Hono's built-in
  `cors` middleware (`hono/cors`) before all routes, with:
  - `origin` resolved from `CORS_ALLOWED_ORIGINS` (env var,
    comma-separated), defaulting to
    `['http://localhost:5173', 'http://localhost:1420']` in
    development and an empty array (deny all) in production.
  - `credentials: true` so the OIDC session cookie rides
    along on cross-origin XHRs.
- `apps/bff/src/server.ts` -- `buildApp` helper exported
  alongside `createServer`. The helper returns the
  assembled Hono app without binding to a port, so the new
  test suite can exercise preflight + simple-response
  CORS paths in-process via `app.request` (no socket, no
  shutdown dance). Existing createServer still owns the
  `serve()` + drain path.
- `apps/bff/src/__tests__/cors.test.ts` -- 12 tests covering:
  - `resolveCorsOrigins` for the env-var / production-flag
    matrix (dev defaults, prod deny, comma parse, trim,
    whitespace-only treated as unset, env overrides dev).
  - `buildApp` preflight 204 + CORS headers for an allowed
    dev origin, header omission for a non-allowlisted origin,
    header pass-through on a real GET, no-Origin same-origin
    path, production deny-by-default, and env-driven
    production allowlist.
- `docs/development/local-dev.md` -- documents
  `CORS_ALLOWED_ORIGINS` in the BFF env-var table and
  adds a "No 'Access-Control-Allow-Origin' header"
  troubleshooting entry pointing at ADR 0018.

**Why this scope belongs on the V3.5 PR, not a hotfix
PR**: the user's established pattern is "fold follow-ups
into the active branch and let the PR absorb them" (PR
#18 in V4-prep carried `feat(ui-kit)` + `feat(app-shell)`
together; the V3.4 closure did the same). A separate
hotfix PR would have created a CORS-unblocker release that
sits between V3.4 and V3.5 in the changelog, which is
worse history than folding the unblocker into the i18n
release that revealed it.

**Re-test triggers for the future**: if the BFF ever adds
a new app origin (mobile webview, third-party embed), this
is the only place to add it. Document the test name in
`buildApp CORS preflight` in any follow-up.
