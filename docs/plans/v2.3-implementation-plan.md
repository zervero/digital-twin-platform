# V2.3 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use
> superpowers:executing-plans to implement this plan task-by-task.
> Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship V2.3 = Track E (Production deployment) from
`docs/plans/v2-overview.md`. Give `@dt/observability` /
`@dt/app-shell` / `apps/bff` / `apps/web` the production shape
that V1 and V2.0 - V2.2 explicitly deferred, so someone other
than the author can run the system in a container, behind a
reverse proxy, with structured logs and graceful shutdown.

**Why now:** V2.0 (realtime + observability), V2.1 (auth
contracts), and V2.2 (plugin runtime) all ship with dev-friendly
defaults that are wrong for production:

- `MockAuthStore` is the only `AuthStore` implementation; in
  production it must be explicitly opt-in (or rejected) so an
  operator can't accidentally ship it.
- The BFF has no signal handling — `kill <pid>` lets in-flight
  WebSocket connections drop ungracefully.
- The BFF and web have no production build / run shape; the
  only "deployment" today is `pnpm dev`.
- CI runs on Linux only; the desktop build (Tauri) is a
  cross-platform surface that has never been exercised by a
  runner.

V2.3 is the last V2 release. V3 (industrial product layer)
adds real auth, plugin marketplace, and multi-tenancy on top
of this deployment shape; the Dockerfile and the env
validation we ship now are the substrate those features
inherit.

**Architecture:** Keep the V2 boundary graph intact. Add a
production-mode env validator to `@dt/config`; add graceful
shutdown + signal handling to `apps/bff`; add `Dockerfile` to
`apps/bff` and `apps/web`; add a `docker-compose.yml` for
local prod-like orchestration; extend `.github/workflows/ci.yml`
with a Windows job (lint + typecheck + test only). No new
runtime deps. No changes to the workspace rule graph.

**Scope — IN:**
- `@dt/config`:
  - `readAppEnv` gains a `production: boolean` (derived from
    `NODE_ENV === 'production'`).
  - `readAppEnv` validates required env vars in production
    mode and throws with a structured error if any are
    missing. The V2.3 contract: in production, `AUTH_PROVIDER`
    must be set to `"mock"` (V2.3's only option) or a future
    `"oidc"` (V3). The BFF reads `AUTH_PROVIDER` and refuses
    to start if it's missing in production. Other secrets
    (e.g. `SESSION_SECRET`) are deferred to V3 when an OIDC
    provider lands.
- `apps/bff`:
  - Graceful shutdown: handle `SIGTERM` and `SIGINT`. On
    signal, stop accepting new connections, send a `close`
    frame to every active WebSocket, wait for in-flight HTTP
    responses (max 10s), then `process.exit(0)`. The dev mock
    source (if running) is stopped first so it doesn't push
    events into a draining server.
  - Production mode: when `nodeEnv === 'production'`, the
    BFF refuses to mount the dev mock source (the
    `NODE_ENV !== 'production'` gate in `src/index.ts` is
    kept, but a startup log line makes it explicit). The
    `AUTH_PROVIDER` env var is read and validated; the
    `MockAuthStore` is only mounted when `AUTH_PROVIDER` is
    `"mock"` or `nodeEnv !== 'production'`.
  - `/health` gains a `ready` companion at `/ready` that
    returns 200 once the server is listening and the WebSocket
    server is attached, 503 during graceful shutdown. The
    orchestrator can poll `/ready` to decide when to send
    SIGTERM and not to take the pod out of rotation until
    `/ready` flips to 503.
- `apps/web`:
  - Vite build is unchanged. The V2.3 work is the runtime
    image: an `nginx:1.27-alpine` base serving `dist/` with
    SPA fallback (`try_files $uri /index.html`) and a reverse
    proxy for `/api/` to the BFF service (in compose) or an
    env-configured upstream (in a real deployment).
- Docker:
  - `apps/bff/Dockerfile`: multi-stage `node:22-alpine`.
    Builder stage runs `pnpm install --frozen-lockfile` then
    `pnpm --filter @dt/bff... build` (turbo's `--filter`
    scopes the build to the BFF and its deps). Runtime stage
    copies `dist/`, installs only production deps
    (`pnpm install --prod --frozen-lockfile` on a pruned
    subset), creates a non-root `node` user, sets `tini` as
    PID 1 for signal forwarding, exposes `3001`, `HEALTHCHECK`
    hits `/health`.
  - `apps/web/Dockerfile`: multi-stage `node:22-alpine` for
    the build, `nginx:1.27-alpine` for the runtime. Copies
    `apps/web/dist/` into the nginx html root, ships a
    templated `nginx.conf` that reads `BFF_UPSTREAM` from env
    for the `/api/` proxy.
  - `docker-compose.yml` at the repo root: two services,
    `bff` and `web`, on an internal network. `web` depends_on
    `bff` with a `service_healthy` condition. Volumes for
    BFF logs to stdout. `BFF_UPSTREAM=http://bff:3001` is
    passed to the web container so nginx knows where to
    proxy `/api/`.
  - `.dockerignore` at the repo root excludes `node_modules`,
    `dist`, `.turbo`, `.git`, `coverage`, `**/*.test.ts`,
    docs, and the Tauri `src-tauri/target` (the desktop app
    is not in the deployment surface).
- CI:
  - `.github/workflows/ci.yml` gains a `windows` job on
    `windows-latest` that runs `pnpm install --frozen-lockfile`
    + `pnpm lint` + `pnpm typecheck` + `pnpm test`. The
    desktop build (`tauri build`) and the V2 smoke stay
    Linux-only for V2.3; Tauri build on Windows needs the
    Rust toolchain + WiX (MSI) / NSIS (NSIS) toolchains +
    code-signing certs, which is its own project (V3).
- Documentation:
  - `docs/development/deployment.md`: env var reference
    (every var `@dt/config` reads, plus the new
    `AUTH_PROVIDER` and `BFF_UPSTREAM`), `docker compose up`
    walkthrough, `docker run` walkthrough for BFF + web
    separately, health check / readiness probe guidance,
    graceful shutdown behavior, log volume / format notes.
- Smoke:
  - `scripts/smoke-prod.sh`: builds the two images with
    `docker compose build`, starts the stack, waits for
    `/health` to be 200 from the web container's proxy,
    hits `/api/stream` through the proxy, asserts the
    structured log line, sends `docker compose stop` (SIGTERM
    path), asserts the BFF exits 0 within 15s. Cleans up
    with `docker compose down -v`. This is a local script;
    CI does not run it (no docker-in-docker on the runner;
    documented in `deployment.md` as a "release-day
    pre-flight" step).

**Scope — OUT:**
- Tauri release build CI (signed `.dmg` / `.msi` / `.AppImage`).
  The Rust toolchain + WiX + NSIS + Apple notarization is
  its own project; V2.3 adds a Windows CI job for the
  workspace tooling (lint / typecheck / test) only.
- Kubernetes manifests, Helm chart, or any orchestrator
  beyond `docker compose`. V2.3 ships a shape that someone
  can lift into K8s, but the manifests are not in scope.
- TLS termination, reverse proxy beyond nginx's `/api/`
  proxy, or cert provisioning. Documented as "front the
  compose stack with your own reverse proxy that terminates
  TLS" in `deployment.md`.
- Real auth provider (OAuth, SAML, JWT signing /
  verification). The V2.1 `MockAuthStore` is the only
  implementation; V2.3 makes the `AUTH_PROVIDER` knob
  explicit so V3 can add `"oidc"` without breaking the
  V2.3 contract. `SESSION_SECRET` and friends are V3.
- Horizontal scaling, sticky sessions, or any stateful BFF
  surface. The BFF is stateless (the broadcaster is in-memory;
  a multi-instance deployment would need a Redis pub/sub

- Multi-region, CDN, or any geo-distribution. V2.3 ships one
  compose stack; multi-region is V3+.
- Observability beyond structured logs (Prometheus metrics

  request-id propagation is the only observability surface;
  metrics + traces are V3.
- Persistent plugin storage. V2.2's plugin registry is
  in-memory; V2.3 doesn't change that.
- Database / cache of any kind. The BFF is stateless and
  mock-backed; V2.3 doesn't add Postgres or Redis.

## Open design questions (decided in this plan)

These are the questions a reviewer would reasonably ask.
Each has a recommended default; the plan below commits to
the default. Speak up before T1 starts if you disagree.

1. **Base image for the BFF container.** Alpine, slim, or
   distroless? **Default: `node:22-alpine`.** Small (~50 MB
   delta over distroless, ~200 MB under slim), well-known
   package set for any debug session, and the BFF has no
   native deps that would force glibc. Distroless wins on
   size but loses on debuggability and is harder to add
   `curl` / `wget` for ad-hoc diagnostics.

2. **Web serving in the web container.** nginx, Caddy, or
   `vite preview`? **Default: `nginx:1.27-alpine`.** Vite
   preview is a dev convenience; the prod image should
   match what a real deployment would use. Caddy's killer
   feature (automatic HTTPS via ACME) doesn't help inside a
   container; nginx's `/api/` reverse proxy is one
   `proxy_pass` line and works the same in dev and prod.

3. **Windows CI job scope.** Lint + typecheck + test only,
   or also `pnpm build`? **Default: lint + typecheck +
   test.** `pnpm build` on Windows would trigger the full
   vite build for `apps/web` and a `tsc -p tsconfig.build.json`
   for `apps/bff`; both are cross-platform, both take 1-2
   min, but the V2.3 acceptance is "the workspace tooling
   works on Windows" and `pnpm build` doesn't add signal
   beyond what the Linux job already proves. Tauri build on
   Windows is a separate project (V3).

4. **Production-required env vars.** What must the BFF
   refuse to boot without in production? **Default:
   `AUTH_PROVIDER` only.** It must be set to `"mock"`
   (V2.3's only option) or `"oidc"` (V3, reserved).
   `SESSION_SECRET` and other JWT-signing keys are V3
   concerns; adding them now would be a placeholder that
   the smoke would have to set, which is worse than the
   honest "V2.3 has no real auth, you must opt into the
   mock explicitly".

5. **Compose persistence.** Any DB / cache / volume? **Default:
   no volumes beyond a `bff-logs` named volume for log
   capture.** The BFF is stateless; the dev mock source is
   in-memory and the broadcaster is in-memory; the web
   container is read-only static assets. A volume for logs
   is the only thing that survives `docker compose down`.

6. **Production smoke in CI.** As a CI job, or a local
   script? **Default: local script only.** GitHub-hosted
   runners don't ship with Docker-in-Docker by default;
   enabling it adds 2-3 min to the workflow and a privileged
   container, which is a poor cost / signal trade for
   "the Dockerfile works". The script is documented in
   `deployment.md` as a "release-day pre-flight" step the
   release captain runs locally before merging the
   release-please PR.

7. **Graceful shutdown timeout.** How long to wait for
   in-flight requests to drain? **Default: 10 seconds.**
   Matches the Kubernetes default `terminationGracePeriodSeconds`
   and gives most HTTP responses + one round of WebSocket
   close frames enough time. Configurable later via env if
   a real deployment needs more.

8. **BFF image size budget.** Multi-stage build, how
   aggressive? **Default: prune devDeps in the runtime
   stage.** `pnpm install --prod --frozen-lockfile` on a
   `pnpm deploy`-style pruned layout keeps the image under
   ~200 MB. DevDeps (vitest, typescript, @types/*) stay in
   the builder stage only.

9. **Web nginx config: SPA fallback + reverse proxy.**
   One config, or split? **Default: one config.** `try_files
   $uri /index.html` for SPA routing, `location /api/ {
   proxy_pass $BFF_UPSTREAM; }` for the reverse proxy.
   The `$BFF_UPSTREAM` is templated by an `envsubst` step
   in the Dockerfile's `ENTRYPOINT` so the same image can
   be used in compose (`http://bff:3001`) and in a real
   deployment (`https://bff.internal:3001`).

10. **`/ready` vs `/health`.** Two endpoints, or one with
    a query param? **Default: two endpoints, distinct
    semantics.** `/health` = "the process is alive and the
    HTTP server is responding" (liveness probe). `/ready` =
    "the process is willing to serve traffic" (readiness
    probe). During graceful shutdown, `/health` stays 200
    (the process is alive) and `/ready` flips to 503 (the
    process is draining). An orchestrator can use them
    independently: `livenessProbe: /health` (restart if
    dead), `readinessProbe: /ready` (stop sending traffic
    during drain).

**Step 1.2: Add `production` + `authProvider` to `AppEnv` and the validator**

In `packages/config/src/index.ts`:

```ts
export class EnvValidationError extends Error {
  readonly missing: readonly string[];
  readonly invalid: readonly { key: string; got: string; allowed: readonly string[] }[];
  constructor(
    missing: readonly string[],
    invalid: readonly { key: string; got: string; allowed: readonly string[] }[],
  ) {
    super(
      [
        missing.length > 0 ? `missing: ${missing.join(', ')}` : null,
        invalid.length > 0
          ? `invalid: ${invalid
              .map((i) => `${i.key}=${i.got} (allowed: ${i.allowed.join('|')})`)
              .join(', ')}`
          : null,
      ]
        .filter(Boolean)
        .join('; '),
    );
    this.name = 'EnvValidationError';
    this.missing = missing;
    this.invalid = invalid;
  }
}

export type AuthProvider = 'mock' | 'oidc';

export interface AppEnv {
  nodeEnv: 'development' | 'production' | 'test';
  production: boolean;
  logLevel: 'debug' | 'info' | 'warn' | 'error';
  port: number;
  authProvider?: AuthProvider;
}

const ALLOWED_AUTH_PROVIDERS = ['mock', 'oidc'] as const;

export function readAppEnv(env: NodeJS.ProcessEnv = process.env): AppEnv {
  const nodeEnv = readEnum(
    env['NODE_ENV'],
    ['development', 'production', 'test'] as const,
    'development',
  );
  const production = nodeEnv === 'production';

  const authProviderRaw = env['AUTH_PROVIDER'];
  let authProvider: AuthProvider | undefined;
  if (authProviderRaw !== undefined && authProviderRaw !== '') {
    if ((ALLOWED_AUTH_PROVIDERS as readonly string[]).includes(authProviderRaw)) {
      authProvider = authProviderRaw as AuthProvider;
    } else if (production) {
      throw new EnvValidationError([], [
        {
          key: 'AUTH_PROVIDER',
          got: authProviderRaw,
          allowed: ALLOWED_AUTH_PROVIDERS,
        },
      ]);
    }
  }

  if (production && !authProvider) {
    throw new EnvValidationError(['AUTH_PROVIDER'], []);
  }

  return {
    nodeEnv,
    production,
    logLevel: readEnum(env['LOG_LEVEL'], ['debug', 'info', 'warn', 'error'] as const, 'info'),
    port: readInt(env['PORT'], 3001),
    authProvider,
  };
}
```

**Step 2.2: Refactor `healthRoute` to take a shutdown flag**

In `apps/bff/src/routes/health.ts`:

```ts
import { Hono } from 'hono';

import type { ApiHealth } from '@dt/contracts';

export interface HealthRouteOpts {
  isShuttingDown: () => boolean;
}

const health: ApiHealth = {
  ok: true,
  service: 'digital-twin-bff',
  version: '0.1.0',
};

export function healthRoute(opts: HealthRouteOpts): Hono {
  const route = new Hono();

  route.get('/health', (c) => c.json(health));

  route.get('/ready', (c) => {
    if (opts.isShuttingDown()) {
      return c.json({ ok: false, reason: 'shutting down' }, 503);
    }
    return c.json({ ok: true });
  });

  return route;
}
```

```ts
const SHUTDOWN_TIMEOUT_MS = 10_000;

async function shutdown(signal: NodeJS.Signals): Promise<void> {
  if (isShuttingDown) return;
  isShuttingDown = true;
  logger.info('shutdown started', { signal });

  // 1. Stop the dev mock source so it doesn't push events
  //    into a draining server.
  if (!env.production) {
    dev?.stop();
  }

  // 2. Send a close frame to every active WebSocket.
  for (const ws of wssClients) {
    try {
      ws.close(1001, 'going away');
    } catch {
      // ignore
    }
  }

  // 3. Close the HTTP server (stops accepting new
  //    connections, waits for in-flight).
  await new Promise<void>((resolve) => {
    server.close(() => resolve());
    setTimeout(() => {
      logger.warn('shutdown timeout, forcing exit');
      resolve();
    }, SHUTDOWN_TIMEOUT_MS);
  });

  logger.info('shutdown complete');
  process.exit(0);
}

process.on('SIGTERM', () => void shutdown('SIGTERM'));
process.on('SIGINT', () => void shutdown('SIGINT'));
```

```markdown
## Graceful shutdown

The BFF handles `SIGTERM` and `SIGINT`:

1. Sets the `isShuttingDown` flag (`/ready` flips to 503).
2. Stops the dev mock source (if running).
3. Sends a WebSocket close frame (`1001 going away`) to
   every active client.
4. Closes the HTTP server (stops accepting new connections,
   waits for in-flight responses).
5. Hard-exits after a 10-second drain timeout.

The `tini` PID 1 in the BFF image forwards signals to the
node process, so `docker stop <container>` (default
`SIGTERM` then `SIGKILL` after 10s) drains cleanly.
```

```bash
#!/usr/bin/env bash
# V2.3 production smoke. Builds the BFF and web images,
# starts the compose stack, exercises /health and /api/stream
# through the web proxy, asserts the BFF logs a structured
# line, sends SIGTERM via `docker compose stop`, and asserts
# the BFF exits 0 within 15s. This is a local script; CI
# does not run it (no docker-in-docker on the runner).
# Exits 0 on success.
set -euo pipefail

COMPOSE="docker compose"
PROJECT="dt-smoke-$(date +%s)"
LOG_DIR="$(mktemp -d -t bff-prod-smoke.XXXXXX)"

cleanup() {
  $COMPOSE -p "$PROJECT" down -v --remove-orphans >/dev/null 2>&1 || true
  rm -rf "$LOG_DIR"
}
trap cleanup EXIT

echo "[smoke-prod] building images"
$COMPOSE -p "$PROJECT" build

echo "[smoke-prod] starting stack"
$COMPOSE -p "$PROJECT" up -d

echo "[smoke-prod] waiting for /health via web proxy"
for _ in $(seq 1 60); do
  if curl -sf http://localhost:8080/health >/dev/null 2>&1; then
    break
  fi
  sleep 0.5
done
if ! curl -sf http://localhost:8080/health >/dev/null 2>&1; then
  echo "[smoke-prod] /health never came up"
  $COMPOSE -p "$PROJECT" logs
  exit 1
fi
echo "[smoke-prod] /health 200 via web proxy"

# Verify /api/ is proxied.
ME_RES="$(curl -sf http://localhost:8080/api/auth/me)"
echo "$ME_RES" | grep -q '"session":null' \
  || { echo "[smoke-prod] /api/auth/me should return session:null"; $COMPOSE -p "$PROJECT" logs; exit 1; }
echo "[smoke-prod] /api/auth/me proxied to BFF"

# Verify /ready.
curl -sf http://localhost:8080/api/../ready >/dev/null \
  || { echo "[smoke-prod] /ready should be 200 when not shutting down"; $COMPOSE -p "$PROJECT" logs; exit 1; }
echo "[smoke-prod] /ready 200 when not shutting down"

# Send SIGTERM via compose stop, with a 15s deadline.
echo "[smoke-prod] sending SIGTERM (docker compose stop)"
START="$(date +%s)"
$COMPOSE -p "$PROJECT" stop -t 15
ELAPSED=$(( $(date +%s) - START ))
if [ "$ELAPSED" -gt 15 ]; then
  echo "[smoke-prod] BFF did not drain within 15s (took ${ELAPSED}s)"
  exit 1
fi
echo "[smoke-prod] BFF drained in ${ELAPSED}s"

echo "[smoke-prod] OK"
```
