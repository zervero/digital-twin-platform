# V3.2 Implementation Plan (Track H: Tauri Release Pipeline)

> Scope: graduate the V2.3 "Tauri runs locally" story into a
> signed, auto-updating desktop distribution. Adds a release
> CI that builds `.dmg` / `.msi` / `.AppImage`, signs the
> macOS + Windows artifacts, uploads them to a GitHub
> Release tagged by release-please, and ships an in-app
> update channel so installed copies detect new versions.
>
> This is the third V3 release; Track F (real auth / OIDC)
> shipped as V3.0 ([ADR 0013](../adr/0013-v3.0-closure.md))
> and Track G (production platform) shipped as V3.1
> ([ADR 0014](../adr/0014-v3.1-closure.md)). Track H is
> largely independent of F/G/I/J at the **code** level but
> sits after G in the ordering so the desktop release CI
> can reuse G's release-please tag flow and the future G
> signing-key workflow (formalized in a V3.2.x follow-up).
>
> V3.2 does **not** ship a managed mobile binary (iOS /
> Android) - the dev spec defers those to V4+. It also does
> **not** ship notarization auto-retry, delta updates, or
> staged rollouts - those are operational niceties that V3.2
> callers don't need yet.

## Open design questions (decided in this plan)

1. **Updater plugin**: Tauri's official `tauri-plugin-updater`
   (Tauri 2.x). Not a custom in-house mechanism. Rationale:
   it's the canonical path, ships signing primitives, and
   keeps the desktop app aligned with the Tauri ecosystem
   instead of diverging into a private update protocol.
2. **Update endpoint**: GitHub Releases, served via the
   `latest` redirect (`releases/latest/download/<asset>`).
   Rationale: release-please already owns the tag flow, so
   the updater reading the same Releases page stays
   single-source. A self-hosted registry would add an
   unrelated service just to host static files.
3. **Build orchestration**: `tauri-apps/tauri-action@v0`
   inside three matrix jobs (`ubuntu-latest`,
   `windows-latest`, `macos-latest`). Rationale: it's the
   upstream-maintained action, it already wires the toolchain
   (Rust, Node, pnpm), signing inputs, and release upload
   hooks, and it tracks Tauri 2.x updates. Hand-rolling the
   workflow would duplicate maintenance.
4. **Trigger shape**: tag push of `v*` (release-please's tag
   pattern) plus `workflow_dispatch` for manual re-runs.
   PRs do **not** trigger the full bundle build (Tauri
   bundling is 10-15 min per platform). PRs instead run a
   lightweight `cargo check` smoke on ubuntu + windows
   inside the existing CI to catch breakage fast.
5. **macOS shape**: a single `macos-latest` runner that
   produces a universal2 (aarch64 + x86_64) `.dmg`. Two
   parallel arch jobs would double CI minutes for a single
   user-facing artifact.
6. **Linux signing**: AppImage / .deb / .rpm are produced
   **unsigned** in V3.2. GPG signing is a follow-up; the
   audit trail comes from GitHub Release provenance, not
   artifact signatures. Rationale: GPG-signed AppImages
   are rare in practice; the cost (rotating keypair,
   documenting operator handoff) outweighs the value for a
   V3.2 first cut.
7. **Code-signing secrets**: documented in the new
   `docs/development/desktop-signing.md`. Workflow steps
   are gated on `if: ${{ secrets.APPLE_CERTIFICATE }}` etc.
   so the build still produces unsigned artifacts in
   environments without secrets (PR forks, dev VMs). The
   release cut by release-please on `main` always has
   secrets available because the runner is the repo's own.
8. **Updater keypair**: generated **once**, locally, with
   `tauri signer generate -w ~/.tauri/dtp.key`. The private
   key is stored as `TAURI_SIGNING_PRIVATE_KEY` in GH
   Actions secrets; the public key is committed to
   `tauri.conf.json`. There is no key-rotation story in
   V3.2; if the key is ever compromised, V3.2.x ships a
   rotation procedure (rebuild + push, with a 1-release
   overlap where both keys are accepted by the app).
9. **Version sync**: `tauri.conf.json`'s `version` field is
   injected at build time from `tauri-action`'s `tagName`
   input. The committed `version: "0.1.0"` in
   `tauri.conf.json` is a placeholder; CI overrides it.
   This keeps a single source of truth (the release-please
   tag).
10. **Smoke test for the updater**: V3.2 does **not** run an
    end-to-end updater smoke in CI. Tauri updater needs an
    installed app on a real desktop OS to exercise the
    download-and-install path, and the CI runners are
    headless. Manual verification (documented in T8) is the
    V3.2 contract; a CI-side smoke is a V3.2.x follow-up
    that needs a GUI runner.

## Global Constraints

- **Conventional Commits**, subject <= 72 chars, lowercase
  version names per `docs/development/contributing.md`
  §"Subject rules".
- **No new boundary violations** introduced: the updater
  plugin stays inside `apps/desktop/src-tauri/` (Tauri Rust
  side) and the frontend hook uses the official
  `@tauri-apps/plugin-updater` package. No new workspace
  package is added; this is a single-app track.
- **Tauri 2.x only**. No upgrade to a future major is in
  scope; if V3.2 lands and Tauri 3 ships mid-cycle, the
  next V3.x absorbs the upgrade.
- **AGENTS.md §5 GitHub Actions pre-flight** is followed
  for every new workflow file. The new
  `desktop-build.yml` lands with: pinned action versions,
  minimum permissions, a comment per non-obvious grant,
  `concurrency:` to cancel duplicates, and `timeout-minutes`
  per job. The first merged run is read in full (not just
  green/red).
- **Lefthook + commitlint + prettier + eslint** still
  enforced. Subject case matters; lowercase wins.
- **CI gates** stay green: `pnpm typecheck` + `pnpm lint` +
  `pnpm test` + `pnpm build` + `pnpm smoke:v2` +
  `pnpm smoke:oidc` + `pnpm chart:lint:strict` (existing V3.1
  surface). V3.2 adds `pnpm desktop:check` (a Rust-side
  `cargo check` on linux + windows) so PRs catch Tauri
  compile failures in ~2 min instead of waiting for the
  10-15 min tag-triggered bundle job.
- **Documentation discipline** per `AGENTS.md` §1:
  workflow / signing changes go into
  `docs/development/desktop-releases.md` and
  `docs/development/desktop-signing.md` (both new in T8).
  The existing `release-playbook.md` is updated with the
  desktop-release path; `architecture/overview.md` is
  updated to mention `@tauri-apps/plugin-updater` next to
  the desktop layer.
- **release-please artifact caveat** (per ADR 0014
  deviations): the next release-please run after V3.2 ships
  may again produce an extra version tag. The V3.2 closure
  ADR will record this if it happens; it does not block
  the V3.2 plan.

## Task ordering

```
T1 -> T2 -> T3 -> T4 -> T5 -> T6 -> T7 -> T8
config  code   keys  macOS  windows linux  release  docs
                                + upload
```

- T1 must land first: `tauri.conf.json` needs the updater
  block + bundle metadata before the Rust side (T2) can
  compile against it.
- T2 (Rust plugin wiring + frontend hook) depends on T1.
- T3 (keypair + secrets docs) is independent of T1 / T2 at
  the code level, but the public key is what T1 commits;
  T3 is the operator-facing companion doc.
- T4 (macOS build + signing) depends on T1, T2, T3 being
  present so the artifact can actually be signed and
  uploaded.
- T5 (Windows build + signing) is the windows sibling of
  T4; same dependencies.
- T6 (Linux build, unsigned) is the linux sibling of T4 /
  T5; same dependencies minus the signing secrets.
- T7 (release upload + tag-trigger wiring) is a thin layer
  on top of T4 / T5 / T6; it consolidates the three jobs
  into one release-please-driven flow.
- T8 (docs) lands last so it captures the actual final
  interface (the secrets shape, the workflow file, the
  release artifact layout).

## Task 2: Rust plugin wiring + frontend hook

**Files:**
- Modify: `apps/desktop/src-tauri/src/lib.rs` (register the
  updater plugin in the Tauri builder)
- Create: `apps/desktop/src/lib/updater.ts` (frontend
  wrapper around `@tauri-apps/plugin-updater` so the rest
  of the app does not import the plugin package directly)
- Modify: `apps/desktop/src/main.ts` (call the updater
  wrapper on app boot; expose a typed event the existing
  plugin-panel system can render if it wants to)
- Create: `apps/desktop/src/__tests__/updater.test.ts`
  (mock `@tauri-apps/plugin-updater`, assert the wrapper
  delegates `check()` and surfaces `available` /
  `currentVersion` / `latestVersion`)
- Modify: `apps/desktop/src-tauri/capabilities/default.json`
  (already touched in T1; ensure `updater:default` is the
  only new entry)

**What gets added:**

The Rust side registers the plugin:

```rust
use tauri_plugin_updater::UpdaterExt;

#[cfg_attr(mobile, tauri::mobile_entry_point)]
pub fn run() {
    tauri::Builder::default()
        .plugin(tauri_plugin_updater::Builder::new().build())
        .setup(|_app| Ok(()))
        .run(tauri::generate_context!())
        .expect("error while running tauri application");
}
```

(`.UpdaterExt` is the trait the builder needs to call
`app.updater()` if the app wants to programmatically
trigger a check from the Rust side. V3.2 does **not** use
it - the frontend hook drives the flow - but the import
keeps the dependency graph honest and is needed by some
Tauri 2.x minor versions.)

The frontend wrapper is the only file the rest of the app
sees:

```ts
// apps/desktop/src/lib/updater.ts
import { check } from '@tauri-apps/plugin-updater';
import { getVersion } from '@tauri-apps/api/app';

export interface UpdateState {
  status: 'idle' | 'checking' | 'available' | 'up-to-date' | 'error';
  currentVersion: string;
  latestVersion?: string;
  error?: string;
}

export async function checkForUpdate(): Promise<UpdateState> {
  try {
    const update = await check();
    if (update === null) {
      return { status: 'up-to-date', currentVersion: await getVersion() };
    }
    return {
      status: 'available',
      currentVersion: await getVersion(),
      latestVersion: update.version,
    };
  } catch (err) {
    return {
      status: 'error',
      currentVersion: await getVersion(),
      error: err instanceof Error ? err.message : String(err),
    };
  }
}
```

The boot path in `main.ts` calls `checkForUpdate()` once on
startup. The result is logged via the existing observability
hooks; a future T8 doc will describe how to surface it in
the UI.

**Step 2.1: unit test**

`updater.test.ts` mocks `@tauri-apps/plugin-updater.check`
to return a fake update object and asserts the wrapper
surfaces `status: 'available'` with the right version
fields. A second test mocks `check` to return `null` and
asserts `status: 'up-to-date'`. A third test mocks `check`
to throw and asserts `status: 'error'` with the error
message.

**Step 2.2: no Rust unit test**

The Rust side is a 2-line plugin registration; the
test surface is the `cargo check` smoke in T4-T6 plus the
full bundle build. No `#[cfg(test)]` block is added in
V3.2.

## Task 3: Updater keypair + signing secrets docs

**Files:**
- Create: `docs/development/desktop-signing.md` (operator
  handoff for the 3 secret sets + the updater keypair)
- Modify: `docs/development/local-dev.md` (add a short
  section: "Generating a Tauri updater keypair for local
  testing" - a one-liner, not a how-to)
- Modify: `apps/desktop/src-tauri/tauri.conf.json`
  (populate `plugins.updater.pubkey` with the public key
  generated from the keypair)
- Modify: `.gitignore` (exclude any local `*.key` /
  `*.key.pub` files a contributor generates; the public
  key in `tauri.conf.json` is committed, the private key
  is not)

**What gets added:**

The keypair generation step (run once, locally, by the
operator):

```bash
# Installed as part of the @tauri-apps/cli
pnpm tauri signer generate -w ~/.tauri/dtp.key

# Outputs:
#   Private key:  ~/.tauri/dtp.key  (NEVER commit; goes to GH secrets)
#   Public key:   <base64>          (goes into tauri.conf.json)
```

The public key is committed to `tauri.conf.json`:

```json
"updater": {
  "pubkey": "dW50cnVzdGVkIGNvbW1lbnQ6...=="
}
```

The private key is stored as `TAURI_SIGNING_PRIVATE_KEY`
(plaintext PEM content) and `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`
(if the key was generated with one) in the repo's GH
Actions secrets. The `desktop-build.yml` workflow reads
these in T4-T6.

`docs/development/desktop-signing.md` covers:

- **macOS secrets** (used in T4): `APPLE_CERTIFICATE`
  (base64 `.p12`), `APPLE_CERTIFICATE_PASSWORD`,
  `APPLE_ID`, `APPLE_PASSWORD` (app-specific password),
  `APPLE_TEAM_ID`. Source: Apple Developer portal.
- **Windows secrets** (used in T5): `WINDOWS_CERTIFICATE`
  (base64 `.pfx`), `WINDOWS_CERTIFICATE_PASSWORD`.
  Source: any CA-issued code-signing cert; Sectigo /
  DigiCert are the common picks.
- **Updater secrets** (used in T4-T6): the keypair from
  above. Source: `tauri signer generate` (this task).
- **Rotation procedure** (sketch): generate a new keypair,
  push a release that ships both old + new public keys
  in `tauri.conf.json`, wait one release cycle for old
  clients to update, remove the old key. Out of scope for
  V3.2.x follow-up.

**Step 3.1: ASCII key handling**

The public key is base64; ASCII-clean. The private key
file never enters the repo. No binary secrets get
checked in.

**Step 3.2: `tauri.conf.json` placeholder**

Until the operator generates the keypair and provides the
public key, `plugins.updater.pubkey` is a placeholder
(`"REPLACE_WITH_OUTPUT_OF_TAURI_SIGNER_GENERATE"`). CI
will not ship a release with this placeholder; T4's
workflow gates the macOS job on the key being present
(`if: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY }}`). The
operator step is documented in T8's quickstart.

## Task 4: macOS build job + signing/notarization

**Files:**
- Create: `.github/workflows/desktop-build.yml` (new
  workflow file, gated on tag push + workflow_dispatch;
  contains the macOS job in this task, windows in T5,
  linux in T6)
- Create: `scripts/desktop-build-mac.sh` (optional; most
  of the work is inline in the workflow, but a thin
  script helps local "build the same way CI does" runs)

**What gets added:**

A `macos` job inside the new workflow file:

```yaml
macos:
  name: Desktop / macOS (.dmg)
  runs-on: macos-latest
  timeout-minutes: 30
  if: ${{ github.event_name == 'workflow_dispatch' || startsWith(github.ref, 'refs/tags/v') }}
  steps:
    - uses: actions/checkout@v5

    - uses: pnpm/action-setup@v5

    - uses: actions/setup-node@v5
      with:
        node-version: 22.17.1
        cache: pnpm

    - name: Install
      run: pnpm install --frozen-lockfile

    - name: Build + sign + notarize (.dmg)
      uses: tauri-apps/tauri-action@v0
      env:
        # Apple notarization + signing; all from GH secrets.
        APPLE_CERTIFICATE: ${{ secrets.APPLE_CERTIFICATE }}
        APPLE_CERTIFICATE_PASSWORD: ${{ secrets.APPLE_CERTIFICATE_PASSWORD }}
        APPLE_ID: ${{ secrets.APPLE_ID }}
        APPLE_PASSWORD: ${{ secrets.APPLE_PASSWORD }}
        APPLE_TEAM_ID: ${{ secrets.APPLE_TEAM_ID }}
        # Updater keypair from T3.
        TAURI_SIGNING_PRIVATE_KEY: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY }}
        TAURI_SIGNING_PRIVATE_KEY_PASSWORD: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY_PASSWORD }}
      with:
        tagName: ${{ github.ref_name }}
        releaseName: "digital-twin-platform ${{ github.ref_name }}"
        releaseBody: "See CHANGELOG.md for details."
        releaseDraft: true
        prerelease: false
```

**Step 4.1: action version**

`tauri-action@v0` is the current major; pinned to the
major tag per AGENTS.md §5. Bumping is a separate commit
when the upstream `@v1` ships.

**Step 4.2: signing / notarization gating**

If `APPLE_CERTIFICATE` is missing, `tauri-action` fails
fast with a clear error. V3.2 does **not** have a
"build unsigned" path for macOS - the bundle needs to be
signed to run on modern macOS without scary Gatekeeper
warnings. The release cut by release-please on `main`
always has secrets available; forks running the workflow
manually without secrets fail loudly.

**Step 4.3: AGENTS.md §5 checklist applied**

The new workflow file passes the pre-flight review:
- **Permissions**: `contents: write` (for the tag-triggered
  release upload), `pull-requests: write` (T7 adds a
  comment per release). Minimal; no `id-token: write`
  (no OIDC).
- **Action versions**: `@v5` / `@v0` pinned, all current
  with the Node 24 runner.
- **Concurrency**: `group: ${{ github.workflow }}-${{ github.ref }}` + `cancel-in-progress: true`.
- **Timeout**: 30 min per platform; matches the upper
  bound of a clean tauri build.
- **First-run verification**: T8 includes "after merge,
  read the full first-run log" in the release playbook.

Same shape as T4 minus the Apple-specific env vars. The

preinstalled; `tauri-action` calls the right toolchain

**Step 5.1: signing gating**

If `WINDOWS_CERTIFICATE` is missing, `tauri-action` builds
an unsigned `.msi`. Modern Windows will still install it
(SmartScreen warns but does not block by default for
first-party). V3.2 documents this caveat in T8 but does
**not** gate the build on the secret being present - the
operator may intentionally cut an unsigned dev release.

**Step 5.2: existing CI integration**

The existing `windows` job in `ci.yml` (which only runs
lint / typecheck / test on `windows-latest`) stays. The
new `desktop-build.yml` adds the bundle build on top;
CI surface area increases by one workflow file, not by
modifying the existing one.

## Task 6: Linux build job (unsigned)

**Files:**
- Modify: `.github/workflows/desktop-build.yml` (add the
  `linux` job to the new workflow file)

**What gets added:**

A `linux` job parallel to macOS / windows:

```yaml
linux:
  name: Desktop / Linux (.AppImage + .deb + .rpm)
  runs-on: ubuntu-latest
  timeout-minutes: 30
  if: ${{ github.event_name == 'workflow_dispatch' || startsWith(github.ref, 'refs/tags/v') }}
  steps:
    - uses: actions/checkout@v5

    - uses: pnpm/action-setup@v5

    - uses: actions/setup-node@v5
      with:
        node-version: 22.17.1
        cache: pnpm

    - name: Install system deps
      # Tauri Linux needs webkit2gtk + libsoup + a few others.
      # The pinned versions match Tauri's official Linux guide.
      run: |
        sudo apt-get update
        sudo apt-get install -y \
          libwebkit2gtk-4.1-dev \
          build-essential \
          curl \
          wget \
          file \
          libxdo-dev \
          libssl-dev \
          libayatana-appindicator3-dev \
          librsvg2-dev

    - name: Install
      run: pnpm install --frozen-lockfile

    - name: Build (.AppImage + .deb + .rpm)
      uses: tauri-apps/tauri-action@v0
      env:
        # No code-signing secrets needed for V3.2 (decision 6).
        TAURI_SIGNING_PRIVATE_KEY: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY }}
        TAURI_SIGNING_PRIVATE_KEY_PASSWORD: ${{ secrets.TAURI_SIGNING_PRIVATE_KEY_PASSWORD }}
      with:
        tagName: ${{ github.ref_name }}
        releaseName: "digital-twin-platform ${{ github.ref_name }}"
        releaseBody: "See CHANGELOG.md for details."
        releaseDraft: true
        prerelease: false
```

The Tauri updater still signs the update payload even
though the AppImage / .deb / .rpm artifacts themselves are
unsigned. That's a separate signing layer (the
`tauri signer sign` step inside `tauri-action`) and is
what makes the in-app update channel trustworthy.

**Step 6.1: GitHub Releases storage**

`tauri-action` uploads the `.AppImage`, `.deb`, `.rpm`,
plus the `latest.json` index + `.sig` signature to the
GitHub Release that `tagName` points at. T7's release job
is a thin wrapper that promotes the draft release to
"latest" + non-draft.

```yaml
  name: Desktop / Publish GitHub Release
  runs-on: ubuntu-latest
  needs: [macos, windows, linux]
  if: ${{ always() && needs.macos.result == 'success' && needs.windows.result == 'success' && needs.linux.result == 'success' }}
  steps:
    - name: Verify draft release exists
      env:
        GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
      run: |
        gh release view "${{ github.ref_name }}" \
          --repo "${{ github.repository }}" \
          --json isDraft,name,tagName
```

(`tauri-action` already created the draft release during
T4 / T5 / T6; this job is a verification + finalization
step. The release becomes "latest" because it points at the
newest tag.)

**Step 7.1: partial-failure policy**

If any of macos / windows / linux fails, the `release` job
skips (because of the `if` chain). The draft release on
GitHub stays as a draft so the operator can decide
whether to delete it, retry the failing job, or cut a
new release once the underlying issue is fixed.

**Step 7.2: AGENTS.md §5 follow-up**

The new workflow's first merged run is read in full by the
operator (per AGENTS.md §5's first-run verification step).
T8 makes this explicit in the release playbook.

## Known caveats (deliberate, recorded for the closure ADR)

- **Linux artifacts are unsigned**. GPG signing is a
  follow-up; the trust story relies on GitHub Release
  provenance, not artifact signatures.
- **Updater keypair has no rotation story**. If the key
  is ever compromised, a V3.2.x ships the rotation
  procedure documented in T8.
- **No CI smoke for the updater**. The end-to-end
  download-and-install path needs a real GUI runner; V3.2
  documents the manual verification path.
- **release-please artifact caveat**: per ADR 0014
  deviations, the release-please run after V3.2 may
  produce an extra version tag. The V3.2 closure ADR
  records this if it happens.
- **Code-signing secrets are operator-managed**. V3.2
  ships the workflow expecting them; if the repo owner
  has not yet provisioned them, macOS / Windows releases
  fail loudly. The first time the workflow is run on a
  fresh fork without secrets, the failure mode is
  documented in desktop-signing.md.
