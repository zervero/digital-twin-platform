# V3.1 Implementation Plan (Track G: Production Platform)

> Scope: graduate the V2.3 Docker Compose story into a
> Kubernetes-native production platform. Adds a Helm chart,
> cert-manager-managed TLS, OpenTelemetry instrumentation in
> the BFF, and the production-side CI that lints / templates
> / kubeconform-validates the manifests before merge.
>
> This is the second V3 release; Track F (real auth / OIDC)
> shipped as V3.0.0 ([ADR 0013](../adr/0013-v3.0-closure.md)).
> Track G depends on the V3.0 OIDC code path (the ingress +
> TLS contract terminates a real `Authorization` flow).
>
> V3.1 does **not** ship an observability stack (Prometheus,
> Grafana, Jaeger) — it ships the SDK + exporter side only,
> so a hosted Grafana Cloud / Datadog / Honeycomb can consume
> the telemetry without platform code changes. HA / multi-
> region stays in V4 per [`docs/plans/v3-overview.md`](./v3-overview.md).

## Open design questions (decided in this plan)

1. **Helm chart structure**: a single umbrella chart at
   `tooling/k8s/digital-twin-platform/` with subcharts for
   `bff` and `web`. Not a monorepo of independent charts.
   The umbrella is what `helm install` consumes; the
   subcharts are what `helm dependency update` resolves.
   Rationale: keeps the deployable surface to one chart per
   environment, while letting each app evolve independently.
2. **Image registry**: images are pulled from a configurable
   `registry/repository` value; the default is
   `ghcr.io/zervero/digital-twin-platform-{bff,web}`. CI
   will be extended in a follow-up release (out of V3.1
   scope) to push images on tag; for now the chart assumes
   the images are already present.
3. **TLS**: cert-manager + Let's Encrypt (`prod` ClusterIssuer
   with an opt-in `staging` for dev clusters). Ingress is
   nginx ingress controller; the chart does not pin a
   specific ingress class — `ingressClassName: nginx` is the
   default, overridable per environment.
4. **OpenTelemetry protocol**: OTLP/HTTP for traces and
   metrics. Logs continue to flow through stdout + the
   `json-file` driver (the V2.3 story), with a documented
   sidecar pattern for clusters that want OTel-native log
   shipping. The chart ships a `Pod` topology that keeps the
   BFF as a single container with OTel env vars; no sidecar.
5. **Auth integration**: the V3.0 OIDC env vars
   (`OIDC_ISSUER_URL`, `OIDC_AUDIENCE`, ...) become Helm
   values; the chart ships a `values-oidc-example.yaml`
   that points at a generic Auth0 / Keycloak issuer.
6. **Multi-tenancy**: still deferred to V3.3 (Track I). The
   chart supports `namespace`-per-tenant via the umbrella's
   `values` namespace pattern but does not enforce it.
7. **Resource sizing**: defaults sized for a small dev /
   staging cluster (BFF: 100m / 256Mi requests, 500m /
   512Mi limits; web: 50m / 64Mi requests, 200m / 128Mi
   limits). Production values are operator-tunable; no
   VPA / HPA in V3.1 (those depend on metrics from OTel,
   which is a follow-up).
8. **Local K8s validation**: V3.1 does **not** require a
   running cluster. CI gates the chart with `helm lint` +
   `helm template` + `kubeconform` against the rendered
   output. This matches the V2.3 stance ("don't gate on
   Docker daemon availability") — the static guarantees are
   the contract.

## Global Constraints

- **Conventional Commits**, subject ≤ 72 chars, lowercase
  version names per `docs/development/contributing.md`
  §"Subject rules".
- **No new boundary violations** introduced by the OTel
  work: the new `@dt/otel` package is pure logic (no Vue,
  no Three, no BFF runtime). It depends on
  `@opentelemetry/*` and `@dt/observability`. Consumers are
  `apps/bff` (server SDK), and a future `apps/web` browser
  RUM integration (deferred to V3.x).
- **AUTH_PROVIDER gate from V2.3 + V3.0 stays**. The chart
  defaults `AUTH_PROVIDER=oidc` for prod values, but the
  gate logic is unchanged.
- **Lefthook + commitlint + prettier + eslint** still
  enforced. Subject case matters; lowercase wins.
- **CI** (`pnpm typecheck` + `pnpm lint` + `pnpm test` +
  `pnpm build` + `pnpm smoke:v2` + `pnpm smoke:oidc`) stays
  green. V3.1 adds `pnpm chart:lint` + `pnpm chart:render`
  to the local script surface so contributors can run the
  same gates CI runs.
- **Documentation discipline** per `AGENTS.md` §1: every
  chart change is documented in `docs/development/production-platform.md`
  (new in T8) or in the chart's own `NOTES.txt`.

## Task ordering

```
T1 → T2 → T3 → T4 → T5 → T6 → T7 → T8
otel pkg   bff wiring  bff/web  helm       ci      prod
                                     chart    gates   docs
                                     + tls
```

- T1 must land first: the OTel SDK lives in a new package
  `@dt/otel` so the BFF and any future consumer can wire it
  uniformly.
- T2 (BFF OTel wiring) depends on T1 — the BFF imports the
  package and starts the SDK before importing anything that
  needs auto-instrumentation.
- T3 (BFF + web OTel env-var config) is the chart-side twin
  of T2: the chart supplies the OTLP endpoint and resource
  attributes; T2's code reads them.
- T4 (Helm chart skeleton + BFF + web subcharts) is the
  bulk of the work. It depends on T3 because the rendered
  manifests need to include the OTel env vars.
- T5 (cert-manager + TLS ingress) is a layer on top of T4.
- T6 (production values + values-oidc-example) is a layer
  on top of T4 + T5.
- T7 (CI gates: helm lint, helm template, kubeconform) is
  independent of source code in the chart sense but depends
  on T4 having rendered outputs to validate.
- T8 (docs) lands last so it captures the actual final
  interface.

## Task 1: New package `@dt/otel`

**Files:**
- Create: `packages/otel/` (new workspace package)
- Modify: `pnpm-workspace.yaml` (already globs `packages/*`)
- Modify: `package.json` (root devDeps: `@opentelemetry/api`,
  `@opentelemetry/sdk-node`,
  `@opentelemetry/auto-instrumentations-node`,
  `@opentelemetry/exporter-trace-otlp-http`,
  `@opentelemetry/exporter-metrics-otlp-http`,
  `@opentelemetry/sdk-metrics`,
  `@opentelemetry/resources`,
  `@opentelemetry/semantic-conventions`)
- Create: `packages/otel/src/index.ts`
- Create: `packages/otel/src/__tests__/start-otel.test.ts`

**What gets added:** A thin facade that owns the OTel Node
SDK lifecycle. Consumers (`apps/bff`) call `startOtel({...})`
once at boot; the package sets up:

- `NodeSDK` with `getNodeAutoInstrumentations` (http, hono,
  ws, dns, net).
- OTLP/HTTP trace + metric exporters; logs remain on stdout
  via `@dt/observability` (the V3.0 story does not change).
- Resource attributes populated from env: `service.name`,
  `service.version` (read from the root `package.json` via
  the build), `deployment.environment`, `service.instance.id`.
- Graceful shutdown: `await sdk.shutdown()` on SIGTERM/SIGINT.

The package is **types-only** like the rest of the workspace
(no real `dist/`). The BFF image layers OTel at runtime the
same way it layers `tsx`.

**Step 1.1: SDK wrapper**

```ts
// packages/otel/src/index.ts
import { NodeSDK } from '@opentelemetry/sdk-node';
import { getNodeAutoInstrumentations } from '@opentelemetry/auto-instrumentations-node';
import { OTLPTraceExporter } from '@opentelemetry/exporter-trace-otlp-http';
import { OTLPMetricExporter } from '@opentelemetry/exporter-metrics-otlp-http';
import { PeriodicExportingMetricReader } from '@opentelemetry/sdk-metrics';
import { resourceFromAttributes } from '@opentelemetry/resources';
import {
  ATTR_SERVICE_NAME,
  ATTR_SERVICE_VERSION,
} from '@opentelemetry/semantic-conventions';

export interface OtelConfig {
  serviceName: string;
  serviceVersion: string;
  environment: string;
  otlpEndpoint: string;       // e.g. https://otel.example.com
  metricsIntervalMs?: number;
}

export async function startOtel(config: OtelConfig): Promise<NodeSDK> { ... }
export async function shutdownOtel(sdk: NodeSDK): Promise<void> { ... }
```

**Step 1.2: env resolution**

`startOtel()` reads `OTEL_EXPORTER_OTLP_ENDPOINT`,
`OTEL_SERVICE_NAME`, `DEPLOYMENT_ENVIRONMENT` from
`process.env`, with the `OtelConfig` argument as overrides.
The BFF can pass `serviceVersion: readAppEnv().version`.

**Step 1.3: tests**

- `start-otel.test.ts` mocks `NodeSDK` and asserts that
  `shutdownOtel` calls `sdk.shutdown()` exactly once.
- A second test confirms the package is a no-op when
  `OTEL_SDK_DISABLED=true` (the OTel standard kill switch).

## Task 2: BFF OTel wiring

**Files:**
- Modify: `apps/bff/src/index.ts` (call `startOtel()` before
  importing `hono` or anything else that needs auto-
  instrumentation)
- Modify: `apps/bff/src/__tests__/graceful-shutdown.test.ts`
  (assert `shutdownOtel` runs in the shutdown path)
- Modify: `apps/bff/package.json` (add `@dt/otel` as a
  dependency)
- Modify: `apps/bff/Dockerfile` (no changes — OTel layers
  the same way `tsx` does; the `tsx` install pattern in
  `Dockerfile` is replicated for the OTel SDK in a follow-up
  if needed; for V3.1 the SDK is bundled by the BFF build)

**What gets added:** A `bootstrap.ts` (or inline `await startOtel()`)
that fires before `import('./http/server.js')`. The OTel SDK
must be initialized **before** any module that the auto-
instrumentations should patch is required.

**Step 2.1: import-order rule**

`apps/bff/src/index.ts` is restructured so the file
**only** does `await startOtel({...})` at the top, then
`await import('./http/server.js')` (dynamic). Static
`import { Hono }` etc. move into `http/server.ts`. This
matches the auto-instrumentation requirement that the SDK
register its require-hook before the instrumented modules
are loaded.

**Step 2.2: shutdown**

The existing `gracefulShutdown()` is extended to call
`shutdownOtel(sdk)` after the WS drain but before the HTTP
server close. The OTel SDK flushes its pending spans /
metrics on shutdown so the SIGTERM telemetry isn't lost.

**Step 2.3: tests**

- New test: SIGTERM causes `shutdownOtel` to be invoked
  (mocked via the `@dt/otel` re-export).
- New test: `startOtel` is called before `createServer`.

## Task 4: Helm chart skeleton + BFF + web subcharts

**Files:**
- Create: `tooling/k8s/digital-twin-platform/Chart.yaml`
- Create: `tooling/k8s/digital-twin-platform/values.yaml`
- Create: `tooling/k8s/digital-twin-platform/values.schema.json`
- Create: `tooling/k8s/digital-twin-platform/templates/_helpers.tpl`
- Create: `tooling/k8s/digital-twin-platform/templates/namespace.yaml`
- Create: `tooling/k8s/digital-twin-platform/charts/bff/...`
- Create: `tooling/k8s/digital-twin-platform/charts/web/...`
- Modify: `package.json` (root scripts: `chart:lint`,
  `chart:render`, `chart:lint:strict`)

**What gets added:** A single umbrella chart
`digital-twin-platform` (chart version `0.1.0`) with two
subcharts `bff` and `web`. The umbrella owns the namespace,
the secret material, and the ingress; the subcharts own the
workload.

**Step 4.1: chart shape**

```
tooling/k8s/digital-twin-platform/
├── Chart.yaml                # umbrella + subcharts (bff, web)
├── values.yaml               # defaults
├── values.schema.json        # JSON schema for values validation
├── templates/
│   ├── _helpers.tpl          # name/label/fullname helpers
│   ├── namespace.yaml
│   └── ingress.yaml          # in T5
└── charts/
    ├── bff/
    │   ├── Chart.yaml
    │   ├── values.yaml
    │   └── templates/
    │       ├── deployment.yaml
    │       ├── service.yaml
    │       ├── configmap.yaml
    │       ├── secret.yaml
    │       └── serviceaccount.yaml
    └── web/
        ├── Chart.yaml
        ├── values.yaml
        └── templates/
            ├── deployment.yaml
            ├── service.yaml
            ├── configmap.yaml
            └── serviceaccount.yaml
```

**Step 4.2: BFF subchart**

- 1 replica by default; `replicaCount` overridable.
- Deployment with `livenessProbe` / `readinessProbe` wired
  to `/health` and `/ready` from V2.3.
- Resources sized for staging (see design question 7).
- `ServiceAccount` per workload; no cluster-wide role.
- `ConfigMap` for non-secret env vars (`LOG_LEVEL`,
  `NODE_ENV`, `AUTH_PROVIDER`); `Secret` for OIDC env vars
  (`OIDC_ISSUER_URL`, `OIDC_AUDIENCE`,
  `OIDC_CLIENT_ID`) — the chart does not generate the secret;
  it references an existing `Secret` by name (`bffOidcSecret`
  value).
- OTel env vars from T3 layered in via `ConfigMap`.
- `PodDisruptionBudget` (minAvailable: 1) so `helm uninstall`
  in dev clusters does not wedge the node.

**Step 4.3: web subchart**

- Same shape as BFF, plus an `Ingress` (in T5).
- nginx-rendered SPA + `/api/` reverse proxy preserved from
  V2.3's nginx.conf; the chart only injects
  `BFF_UPSTREAM` via `envsubst`.

**Step 4.4: scripts**

```json
{
  "scripts": {
    "chart:lint": "helm lint tooling/k8s/digital-twin-platform",
    "chart:render": "helm template dtp tooling/k8s/digital-twin-platform --include-crds",
    "chart:lint:strict": "helm lint --strict tooling/k8s/digital-twin-platform && kubeconform -strict -summary -verbose <(pnpm chart:render)"
  }
}
```

`kubeconform` is added as a root devDep.

## Task 5: cert-manager + TLS ingress

**Files:**
- Modify: `tooling/k8s/digital-twin-platform/charts/web/templates/ingress.yaml`
- Create: `tooling/k8s/digital-twin-platform/templates/cluster-issuer.yaml`
- Create: `tooling/k8s/digital-twin-platform/templates/certificate.yaml`
- Create: `tooling/k8s/digital-twin-platform/values-cert-manager-example.yaml`
- Modify: `tooling/k8s/digital-twin-platform/values.yaml` (add
  ingress + cert-manager knobs)

**What gets added:** nginx Ingress with TLS termination
delegated to cert-manager. Two layers:

1. **ClusterIssuer** (template at the umbrella level,
   guarded by `cert-manager.enabled: true`): a single
   `letsencrypt-prod` issuer using the HTTP-01 challenge.
   A `letsencrypt-staging` issuer is opt-in via
   `cert-manager.staging: true` for dev clusters.
2. **Certificate** (also umbrella-level): the
   `digital-twin-platform-tls` certificate referencing the
   issuer, with `dnsNames` populated from
   `ingress.hosts[]`.

The Ingress lives in the `web` subchart because it
terminates the public-facing path; the chart structure keeps
the BFF Ingress (if added later) in the `bff` subchart.

**Step 5.1: ingress defaults**

```yaml
ingress:
  enabled: true
  className: nginx
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-prod
    nginx.ingress.kubernetes.io/proxy-body-size: 16m
  hosts:
    - host: digital-twin.example.com
      paths:
        - path: /
          pathType: Prefix
  tls:
    - hosts:
        - digital-twin.example.com
      secretName: digital-twin-platform-tls
```

**Step 5.2: kubeconform schema**

`values.schema.json` rejects `ingress.hosts[*].host` values
that don't resolve to a valid DNS name. The
`chart:lint:strict` script fails on schema violations.

## Task 6: Production values + OIDC example

**Files:**
- Create: `tooling/k8s/digital-twin-platform/values-prod.yaml`
- Create: `tooling/k8s/digital-twin-platform/values-staging.yaml`
- Create: `tooling/k8s/digital-twin-platform/values-oidc-example.yaml`
- Create: `tooling/k8s/digital-twin-platform/secrets.example.yaml`
  (placeholder; **not** a real Secret — it's a template
  showing the keys the chart expects)

**What gets added:** Three value overlays:

1. **`values-prod.yaml`** — `replicaCount: 2`, larger
   resource limits, OIDC env vars, ingress enabled.
2. **`values-staging.yaml`** — `replicaCount: 1`, dev
   resource limits, ingress disabled (port-forward access).
3. **`values-oidc-example.yaml`** — OIDC env vars set to
   generic Auth0 placeholders, with a comment block showing
   how to substitute for Keycloak / Okta.

The chart's `templates/secret.yaml` references a Secret the
operator pre-creates (the chart does not own secret
material). `secrets.example.yaml` documents the keys and is
suffixed `.example` so `kubectl apply -f secrets.yaml` is a
conscious copy-paste.

installs via the existing `pnpm install` step (no new
action needed).

## Task 8: Production platform docs

**Files:**
- Create: `docs/development/production-platform.md`
- Modify: `docs/development/deployment.md` (cross-link;

**What gets added:** A single-page guide that mirrors

- **Quickstart (local cluster)**: install `kind` /
  `minikube`, `helm install dtp tooling/k8s/digital-twin-platform`,
  `kubectl port-forward svc/dtp-web 8080:80`,
  `curl http://localhost:8080/health`.
- **Production install**: `helm repo add`, `helm install`,
  link to the values overlays.

  ClusterIssuer (or rely on the chart-supplied one), how
  to verify the certificate issued, how to renew.

  a hosted OTLP endpoint; the env vars the chart sets; how
  to bring your own endpoint.
- **Operations**: `helm upgrade`, `helm rollback`, log
  retrieval via `kubectl logs`, pod scale via
  `kubectl scale`.
- **Troubleshooting**: common failure modes (cert pending,
  image pull backoff, OTel exporter unreachable) and the
  chart / kubectl knobs that fix them.
- **What's not in V3.1**: HA / multi-region, autoscaling,
  VPA, sidecar log shipping, browser RUM.

to move "Kubernetes manifests / Helm chart" and "Metrics
endpoint, OpenTelemetry traces" into a "Shipped in V3.1"
list, with cross-links to the new doc and the future
closure ADR.
