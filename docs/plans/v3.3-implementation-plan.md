# V3.3 Implementation Plan (Track I: Multi-Tenant Data Model)

> Scope: graduate the V3.0 single-tenant data model into a
> tenant-scoped one. Every DTO that holds tenant data gets
> a required `tenantId` field. The BFF extracts the
> tenant from an OIDC JWT claim and scopes every
> authenticated request to that tenant. The realtime
> broadcaster filters events by tenant. A new dev IdP
> minting ceremony produces per-tenant JWTs so the
> `pnpm smoke:tenant` end-to-end test can verify
> isolation across 3 dev tenants.
>
> This is the fourth V3 release. Track F (real auth / OIDC)
> shipped as V3.0 ([ADR 0013](../adr/0013-v3.0-closure.md)),
> Track G (production platform) shipped as V3.1
> ([ADR 0014](../adr/0014-v3.1-closure.md)), and Track H
> (Tauri release pipeline) shipped as V3.2
> ([ADR 0015](../adr/0015-v3.2-closure.md)). Track I depends
> on F (auth) for tenant identity and on G (production
> platform) for tenant-aware routing; H is independent at
> the code level but ships before I so the operator has
> stable release mechanics to lean on.
>
> V3.3 does **not** ship database-per-tenant isolation,
> cross-tenant impersonation, or per-tenant rate limiting
> -- those are V3.3.x follow-ups documented at the end
> of this plan. V3.3 ships the **shared DB with
> `tenant_id` column** model, the JWT claim extraction,
> and the BFF enforcement layer. Adding a real database
> with row-level security is a V3.3.x concern (one
> follow-up task per backend store).

## Open design questions (decided in this plan)

1. **Tenancy model**: shared DB with `tenant_id` column on
   every tenant-scoped table; row-level security at the
   storage layer is a V3.3.x follow-up. Rationale: V1-V3.2
   have no persistent storage (mock data only), so the
   "migration" is updating the mock shape to be
   multi-tenant. A real database lands when the platform
   team picks one; the `tenant_id` column is the contract
   the BFF already understands. Database-per-tenant is a
   V3.3.x follow-up if a tenant needs hard isolation.
2. **Tenant identity source**: same OIDC issuer as Track F,
   sourced from a namespaced JWT claim
   (`https://api.digital-twin-platform.local/tenant_id` by
   default, overridable via `OIDC_TENANT_CLAIM` env var).
   Rationale: the IdP already authenticates the user; adding
   a tenant claim is the smallest delta. The namespaced
   claim convention is the OIDC standard (Auth0, Keycloak,
   Okta all support it).
3. **Tenant resolution at request time**: implicit from
   the JWT. Cross-tenant request shapes (e.g. an admin
   acting on behalf of tenant B) are out of scope for V3.3;
   the BFF returns 403 on any attempt to access a tenant
   the caller is not part of. A future "tenant
   impersonation" feature for admins is a V3.3.x follow-up.
4. **Realtime fan-out**: in-process subscription filter
   keyed on `event.tenantId`. The `RealtimeBroadcaster`
   now filters events at the broadcast step so subscribers
   only see events for their tenant. A multi-tenant broker
   (Redis pub/sub, NATS) is a V3.3.x concern when the
   BFF is deployed with multiple replicas.
5. **Permission model**: the V3.0 `Permission` union stays
   stable. V3.3 does not add a new permission; tenant
   isolation is enforced at the BFF layer, not the
   permission layer. A future `tenant:admin` permission for
   the "manage tenant" endpoint is a V3.3.x follow-up.
6. **`@dt/tenant` ownership**: pure types + helpers, like
   `@dt/auth-oidc`. No I/O, no runtime deps beyond
   `@dt/contracts`. The package is the single source of
   truth for the `Tenant` shape, the claim constant, and
   the claim extraction helper.
7. **WebSocket auth**: `/api/stream` is now auth-gated.
   V3.0-V3.2 left it open (any client could subscribe);
   V3.3 makes the WS route use the same `requiresTenantScope`
   middleware so the server can read `tenantId` from the
   session and filter events accordingly.
8. **Dev IdP shape**: the existing `dev-oidc-idp.mjs` is
   extended with a `--tenant` flag so a single BFF smoke
   can mint tokens for 3 different dev tenants. The
   `pnpm smoke:tenant` script mints the tokens, hits each
   tenant's `/api/devices`, and asserts that no tenant
   sees another tenant's data.

## Global Constraints

- **Conventional Commits**, subject <= 72 chars, lowercase
  version names per `docs/development/contributing.md`
  -- "Subject rules".
- **No new boundary violations** introduced: `@dt/tenant`
  imports only from `@dt/contracts` (the existing
  rule "no new package may import a non-shared package"
  is preserved). The BFF keeps its existing dependency
  surface -- `requiresTenantScope` is a new middleware
  in the BFF, not a new package dependency. `@dt/auth-oidc`
  is extended but does not gain a dependency on
  `@dt/tenant`; the claim constant is defined in
  `@dt/tenant` and re-exported through `@dt/auth-oidc` so
  consumers can pick either entry point.
- **AGENTS.md § 1 documentation discipline**: every
  public type addition (the `tenantId` fields, the
  `Tenant` type, the new error codes) is mirrored in the
  relevant doc. `@dt/contracts` device/scene/command/event
  JSDoc blocks get a one-line note that `tenantId` is
  required. The new `docs/development/multi-tenant.md`
  (T8) is the operator handoff.
- **AGENTS.md § 5 GitHub Actions pre-flight** is
  followed for the new `pnpm smoke:tenant` CI job (added
  in T8). The job lands with: pinned action versions,
  minimum permissions, a comment per non-obvious grant,
  `concurrency:` to cancel duplicates, and `timeout-minutes`.
  The first merged run is read in full.
- **Lefthook + commitlint + prettier + eslint** still
  enforced. Subject case matters; lowercase wins.
- **CI gates** stay green: `pnpm typecheck` + `pnpm lint` +
  `pnpm test` + `pnpm build` + `pnpm smoke:v2` +
  `pnpm smoke:oidc` + `pnpm chart:lint:strict` (existing
  V3.1 + V3.2 surface). V3.3 adds `pnpm smoke:tenant` (a
  3-tenant isolation smoke that runs in ~10 seconds and
  is cheap enough to put on every PR).
- **Existing smoke:v2 + smoke:oidc stay green**. V3.3
  keeps the legacy single-tenant mock path (the `mock`
  auth provider mints sessions with a default dev tenant
  ID) so the existing dev loop and CI smokes do not
  require new setup. The new `smoke:tenant` covers the
  multi-tenant case explicitly.
- **Backwards compatibility for the public `Permission`
  union**: V3.3 does not add a new permission value. The
  multi-tenancy contract is enforced at the BFF layer,
  not the permission layer; this keeps the
  V3.0-compatible `Permission` union stable so older auth
  contracts from V2.1 stay valid.
- **Release-please artifact caveat** (per ADR 0014
  deviations): the next release-please run after V3.3
  ships may again produce an extra version tag. The V3.3
  closure ADR will record this if it happens; it does not
  block the V3.3 plan.

## Task ordering

```
T1 -> T2 -> T3 -> T4 -> T5 -> T6 -> T7 -> T8
contracts  @dt/tenant  JWT claim  middleware  mock  routes  WS  smoke+docs
```

- T1 must land first: every T2-T7 task constructs or
  consumes a `tenantId` field on a DTO. The DTOs are the
  shared contract; changing them after the fact means
  rewriting test fixtures and downstream code.
- T2 (`@dt/tenant` package) depends on T1: the `Tenant`
  type re-exports `Permission`-adjacent helpers and
  assumes `tenantId: string` is on every scoped DTO.
- T3 (`@dt/auth-oidc` claim extraction) depends on T2:
  the `tenantId` field on `VerifiedSession` is a
  re-export of `@dt/tenant`'s `getTenantIdFromClaims`.
- T4 (BFF `requiresTenantScope` middleware) depends on
  T3: the middleware reads the claim-extracted
  `tenantId` and puts a `TenantContext` on the Hono
  context.
- T5 (mock data split) depends on T2: the per-tenant
  mock factories use the `Tenant` type from `@dt/tenant`.
- T6 (tenant-scoped routes) depends on T4 + T5: the
  route handlers consume the `c.var.tenant` set by
  T4 and the `getDevicesForTenant` / `getSceneForTenant`
  factories from T5.
- T7 (tenant-scoped realtime broadcaster) depends on T4
  + T6: the WS route uses `requiresTenantScope` to gate
  the upgrade, and the broadcaster filters by
  `event.tenantId`.
- T8 (dev IdP + smoke + docs) lands last so it captures
  the actual final interface (the claim shape, the
  middleware signature, the route error codes).

```ts
// packages/tenant/src/tenant.ts

/**
 * Canonical OIDC claim name that carries the tenant
 * identifier. Namespaced per OIDC convention (Auth0,
 * Keycloak, Okta all support namespaced claims).
 *
 * Override at runtime by setting the `OIDC_TENANT_CLAIM`
 * env var to a different claim name. The default is
 * `https://api.digital-twin-platform.local/tenant_id`;
 * a deployment that uses Auth0 might prefer
 * `https://your-app.auth0.com/tenant_id`.
 */
export const TENANT_ID_CLAIM =
  'https://api.digital-twin-platform.local/tenant_id';

export type TenantPlan = 'free' | 'pro' | 'enterprise';

export interface Tenant {
  /** Unique opaque tenant ID (e.g. `acme-corp`). */
  id: string;
  /** Human-readable name (e.g. `Acme Corporation`). */
  name: string;
  /** URL-safe slug; same as `id` for V3.3. */
  slug: string;
  /** Billing tier. V3.3 placeholder; real billing is V4+. */
  plan: TenantPlan;
  /** ISO-8601 timestamp. */
  createdAt: string;
}

/**
 * What the BFF puts on `c.var.tenant` after
 * `requiresTenantScope` runs. Carries the full Tenant
 * record so route handlers can read plan, name, etc.
 * without a separate lookup.
 */
export interface TenantContext {
  tenant: Tenant;
}

/**
 * Extract the tenant ID from a verified JWT's claims.
 * Returns `null` if the claim is missing or not a
 * non-empty string. The BFF treats `null` as a
 * 401 `AUTH_NO_TENANT` response.
 */
export function getTenantIdFromClaims(
  claims: Record<string, unknown>,
  claimName: string = TENANT_ID_CLAIM,
): string | null {
  const value = claims[claimName];
  if (typeof value !== 'string') return null;
  if (value.length === 0) return null;
  return value;
}
```

The `__tests__/claims.test.ts` covers:

- a JWT-shaped object with a custom claim name

- a JWT-shaped object with the claim set to a non-string

**What gets added:**

The new middleware combines the V3.0 auth flow with
the V3.3 tenant extraction:

```ts
// apps/bff/src/middleware/requires-tenant.ts

import type { MiddlewareHandler } from 'hono';
import { permissionsFor, type Permission } from '@dt/contracts';
import { getTenantIdFromClaims, type TenantContext } from '@dt/tenant';
import type { AuthStore } from '../auth/store.js';
import { resolveTenant } from '../tenants/resolve.js';

export function requiresTenantScope(
  store: AuthStore,
  required: Permission,
): MiddlewareHandler {
  return async (c, next) => {
    const me = await store.getMe(c.req.raw.headers);
    if (!me.session) {
      return c.json(
        { error: 'AUTH_SESSION_EXPIRED', message: 'Session not found' },
        401,
      );
    }

    // Tenant is required for every scoped route.
    const tenantId = me.session.tenantId ?? null;
    if (!tenantId) {
      return c.json(
        { error: 'AUTH_NO_TENANT', message: 'Session has no tenant' },
        401,
      );
    }
    const tenant = resolveTenant(tenantId);
    if (!tenant) {
      return c.json(
        { error: 'AUTH_NO_TENANT', message: `Unknown tenant: ${tenantId}` },
        401,
      );
    }

    const permissions =
      me.session.permissions ?? permissionsFor(me.session.user.roles);
    if (!permissions.includes(required)) {
      return c.json(
        { error: 'AUTH_FORBIDDEN', message: `Missing permission: ${required}` },
        403,
      );
    }

    c.set('user', me.session.user);
    c.set('permissions', [...permissions]);
    c.set('tenant', { tenant } satisfies TenantContext);
    return next();
  };
}
```

```ts
          status: n.status ?? 'offline',
          sceneNodeId: n.id,
          updatedAt: new Date('2026-01-01T08:00:00.000Z').toISOString(),
        })),
    ]),
  );

export function getDevicesForTenant(tenantId: string): readonly Device[] {
  return DEMO_DEVICES_BY_TENANT[tenantId] ?? [];
}
```

```ts
// apps/bff/src/realtime/broadcaster.ts (V3.3)

export class RealtimeBroadcaster {
  // ... existing fields

  /**
   * V3.3: subscribe to events for a specific tenant.
   * The internal stream is single-tenant, so the
   * broadcaster now owns one subscription per tenant
   * and fans events out to the right subscribers.
   */
  subscribeClient(
    tenantId: string,
    onEvent: (event: DigitalTwinEvent) => void,
  ): () => void {
    return this.stream.subscribe((event) => {
      if (event.tenantId === tenantId) onEvent(event);
    });
  }
}
```

```bash
  sleep 0.5

TOKEN_A=$(node scripts/dev-oidc-idp.mjs mint --tenant acme-corp)
TOKEN_B=$(node scripts/dev-oidc-idp.mjs mint --tenant globex-ind)
TOKEN_C=$(node scripts/dev-oidc-idp.mjs mint --tenant initech-llc)

# Each tenant sees only its own devices
DEVICES_A=$(curl -s -H "Authorization: Bearer $TOKEN_A" \
  http://localhost:3001/api/devices)
test "$(echo "$DEVICES_A" | jq '.[0].tenantId')" = '"acme-corp"'

DEVICES_B=$(curl -s -H "Authorization: Bearer $TOKEN_B" \
  http://localhost:3001/api/devices)
test "$(echo "$DEVICES_B" | jq '.[0].tenantId')" = '"globex-ind"'

# tenantId: 'globex-ind' -> 403

  http://localhost:3001/api/devices)
```

## Known caveats (deliberate, recorded for the closure ADR)

- **Mock tenant registry in production code path**. V3.3
  uses the in-process `DEMO_TENANTS` as the tenant
  resolver. A real database with row-level security is a
  V3.3.x follow-up; the `resolveTenant` function is the
  single point that needs to change.
- **In-process broadcaster fan-out**. The V3.3
  `RealtimeBroadcaster` is single-process; a multi-replica
  deployment would need a Redis / NATS pub/sub layer to
  filter events across replicas. V3.3 ships the
  per-tenant filter at the subscription step; the
  multi-replica case is a V3.3.x concern.
- **No admin cross-tenant impersonation**. V3.3 returns
  403 on any attempt to access a tenant the caller is
  not part of. A future "tenant impersonation" feature
  for admins is a V3.3.x follow-up (it would be a
  permission-gated "act-as-tenant" header + a new
  permission value).
- **No per-tenant rate limiting**. V3.3 enforces
  isolation but not fairness; a noisy tenant can
  saturate the BFF for everyone. Per-tenant rate
  limiting is a V3.3.x follow-up.
- **release-please artifact caveat**: per ADR 0014
  deviations, the release-please run after V3.3 may
  produce an extra version tag. The V3.3 closure ADR
  records this if it happens.
- **Permission union is unchanged**. V3.3 does not add a
  `tenant:admin` permission; multi-tenancy is enforced at
  the BFF layer, not the permission layer. This keeps
  the V3.0 permission contract stable.
