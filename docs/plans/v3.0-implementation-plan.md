# V3.0 Implementation Plan (Track F: Real Auth, OIDC)

> Scope: replace the V2 `AUTH_PROVIDER=mock` placeholder with a
> real OIDC implementation. `AUTH_PROVIDER=oidc` becomes
> functional end-to-end (BFF verifies JWTs, `requiresPermission`
> middleware enforces scopes, web + desktop do the redirect
> flow). `AUTH_PROVIDER=mock` stays for the simplest dev loop.
>
> This is the first release of the V3 era
> ([`docs/plans/v3-overview.md`](./v3-overview.md); [ADR 0012](../adr/0012-v3-roadmap.md)).
> K8s + TLS + OTel (Track G), signed Tauri installers (Track
> H), multi-tenant data model (Track I), and the plugin
> marketplace (Track J) live in later V3.x releases.

## Open design questions (decided in this plan)

1. **OIDC provider**: generic, not vendor-locked. The BFF uses
   [`jose`](https://github.com/panva/jose) for JWT verification
   + JWKS fetching. The browser uses
   [`oidc-client-ts`](https://github.com/authts/oidc-client-ts)
   for the redirect flow. Works against Auth0, Keycloak, Okta,
   Google, GitHub OAuth; any provider that speaks standard OIDC
   and signs JWTs with RSA or EC keys. **No vendor SDK is
   introduced** as a project dependency.
2. **JWKS caching**: in-memory LRU keyed by issuer URL, 5-min
   TTL, refreshed on `kid` cache miss (handles key rotation).
   Bounded to 16 issuers so a misconfigured client doesn't
   OOM the process.
3. **Permission claim shape**: support both `scope` (space-
   separated string, OIDC standard) and `permissions` (JSON
   array, Auth0-style). The `requirePermission` helper accepts
   both. The `Permission` union from `@dt/contracts` (V2.1) is
   still the single source of truth.
4. **Token transport**: OIDC: HttpOnly + SameSite=Lax + Secure
   cookie set by `/api/auth/oidc/callback`. Web reads it via
   `/api/auth/me`. No `localStorage` token storage — XSS
   resilient by default.
5. **Dev OIDC IdP**: a small Node script at
   `scripts/dev-oidc-idp.mjs` that listens on `:9999`, exposes
   a discovery endpoint, and issues self-signed RS256 JWTs.
   Not a real IdP; a development convenience. CI uses
   `AUTH_PROVIDER=oidc` against this script via
   `pnpm smoke:oidc`. Production runs against a real IdP
   configured via the env-var shape this plan establishes.
6. **`requiresPermission` middleware semantics**: 401 if no
   session, 403 if session lacks the permission. Distinct so
   consumers can tell "log in" from "you can't do that".
7. **Dev BFF toggle**: `AUTH_PROVIDER=mock` (default in
   `NODE_ENV=development`, controlled via `OIDC_ALLOW_DEV_BYPASS=true`)
   lets the dev BFF skip JWT verification and use a built-in
   mock user, matching the V2 behavior. Production OIDC never
   has this flag set.

## Global Constraints

- **Conventional Commits**, subject ≤ 72 chars, lowercase
  version names (`v3`, `v2.3`, never `V3` / `V2.3`) per
  `docs/development/contributing.md` §"Subject rules".
- **No new boundary violations**: `@dt/auth-oidc` is a pure-
  logic package (no Vue, no Three, no BFF). `apps/bff` already
  imports from `@dt/config` + `@dt/contracts`; adding
  `@dt/auth-oidc` is in the same vein. `@dt/app-shell` stays
  Vue + Pinia; the OIDC browser composable lives there.
- **`AUTH_PROVIDER` gate** from V2.3 stays. `mock` is always
  valid; `oidc` is valid only when the OIDC env vars are set.
- **Lefthook + commitlint + prettier + eslint** still
  enforced. Subject case matters; lowercase wins.
- **CI** (`pnpm typecheck` + `pnpm lint` + `pnpm test` +
  `pnpm build` + `pnpm smoke:v2`) stays green. V3.0 adds
  `pnpm smoke:oidc` as a CI runnable smoke; if real OIDC config
  isn't available in CI, falls back to the dev IdP script.

## Task ordering

```
T1 → T2 → T3 → T4 → T5 → T6 → T7 → T8
auth-oidc pkg  config    routes / middleware   web/   docs
                                enforcement     desk   dev
                                                compos-  IdP
                                                able
```

- T1 must land first: the new `@dt/auth-oidc` package is
  imported by T3 (BFF auth provider) and T4 (BFF routes).
- T2 (config) can land alongside T1 — it's an additive change
  to `@dt/config`. Independent of T1's runtime behavior.
- T3-T5 (BFF auth provider + routes + permission middleware)
  are a chain. T3's interface feeds T4's routes; T4's
  session shape feeds T5's middleware.
- T6 (web + desktop login composable) depends on T4's session
  shape and the OIDC redirect URL convention.
- T7 (dev IdP script) is independent of T1-T6 in source code;
  it can land anytime but is most useful after T5.
- T8 (docs) lands last so it documents the actual final
  interface, not the planned interface.

**Step 1.2: verify.ts**

`packages/auth-oidc/src/verify.ts`:

```ts
/**
 * @dt/auth-oidc — JWT verification + permission extraction.
 *
 * V3 boundary. Host-agnostic and runtime-agnostic: no Vue,
 * no Three, no BFF. Consumers are expected to be the BFF
 * (apps/bff) and any future server-side plugin that needs
 * to verify a user's identity.
 *
 * The `verifyJwt` function does the heavy lifting: it pulls
 * the JWKS from the issuer (cached), verifies the JWT against
 * it, and returns either the parsed claims or a typed error.
 * `extractPermissions` parses the `scope` (space-separated
 * string) and `permissions` (array) claims into the V2.1
 * `Permission` union.
 */

import {
  createRemoteJWKSet,
  jwtVerify,
  type JWTPayload,
  type JWTVerifyGetKey,
  type JWTVerifyResult,
} from 'jose';

import { ALL_PERMISSIONS, type Permission } from '@dt/contracts';

export interface OidcVerifyConfig {
  /** The issuer URL — must match the JWT `iss` claim exactly. */
  issuerUrl: string;
  /** The audience claim — must match the JWT `aud` claim. */
  audience: string;
  /** Optional JWKS URI override; defaults to the issuer's discovery endpoint. */
  jwksUri?: string;
  /** Cache TTL for the JWKS, in milliseconds. Default 5 minutes. */
  jwksTtlMs?: number;
}

export interface VerifiedSession {
  subject: string;
  permissions: readonly Permission[];
  claims: JWTPayload;
}

export type VerifyResult =
  | { ok: true; session: VerifiedSession }
  | { ok: false; code: 'EXPIRED' | 'BAD_SIGNATURE' | 'BAD_AUDIENCE' | 'BAD_ISSUER' | 'JWKS_ERROR' | 'UNKNOWN'; message: string };

const PERMISSION_SET = new Set<string>(ALL_PERMISSIONS);

export async function verifyJwt(
  token: string,
  config: OidcVerifyConfig,
): Promise<VerifyResult> {
  let jwksResolver: JWTVerifyGetKey;
  try {
    const jwksEndpoint =
      config.jwksUri ?? new URL('/.well-known/jwks.json', config.issuerUrl).href;
    jwksResolver = createRemoteJWKSet(new URL(jwksEndpoint), {
      cacheMaxAge: config.jwksTtlMs ?? 5 * 60 * 1000,
      cooldownDuration: 30 * 1000,
    });
  } catch (e) {
    return {
      ok: false,
      code: 'JWKS_ERROR',
      message: e instanceof Error ? e.message : String(e),
    };
  }

  let result: JWTVerifyResult;
  try {
    result = await jwtVerify(token, jwksResolver, {
      issuer: config.issuerUrl,
      audience: config.audience,
    });
  } catch (e) {
    const err = e as { code?: string; message?: string };
    const message = err.message ?? 'JWT verification failed';
    if (err.code === 'ERR_JWT_EXPIRED') {
      return { ok: false, code: 'EXPIRED', message };
    }
    if (err.code === 'ERR_JWS_SIGNATURE_VERIFICATION_FAILED') {
      return { ok: false, code: 'BAD_SIGNATURE', message };
    }
    if (err.code === 'ERR_JWT_CLAIM_VALIDATION_FAILED') {
      // jose reports aud/iss claim mismatches under this code.
      // We disambiguate by re-reading the token header (no signature
      // check needed at this point because the error implies the
      // signature was fine but a claim failed).
      if (/aud/i.test(message)) {
        return { ok: false, code: 'BAD_AUDIENCE', message };
      }
      if (/iss/i.test(message)) {
        return { ok: false, code: 'BAD_ISSUER', message };
      }
    }
    return { ok: false, code: 'UNKNOWN', message };
  }

  const permissions = extractPermissions(result.payload);
  const subject =
    typeof result.payload.sub === 'string' ? result.payload.sub : '';
  return {
    ok: true,
    session: {
      subject,
      permissions,
      claims: result.payload,
    },
  };
}

export function extractPermissions(payload: JWTPayload): readonly Permission[] {
  const out: Permission[] = [];
  // `scope` is the OIDC-standard space-separated permission string.
  if (typeof payload.scope === 'string') {
    for (const p of payload.scope.split(/\s+/).filter(Boolean)) {
      if (PERMISSION_SET.has(p)) {
        out.push(p as Permission);
      }
    }
  }
  // `permissions` is the Auth0-style JSON-array claim. Some
  // providers emit this in addition to or instead of `scope`.
  if (Array.isArray(payload.permissions)) {
    for (const p of payload.permissions) {
      if (typeof p === 'string' && PERMISSION_SET.has(p)) {
        if (!out.includes(p as Permission)) {
          out.push(p as Permission);
        }
      }
    }
  }
  return out;
}
```

**Step 1.3: tests**

`packages/auth-oidc/src/__tests__/verify.test.ts` covers:
- A valid RS256-signed JWT with `iss`, `aud`, `sub`, and
  `scope: "auth:login devices:read"` is accepted; permissions
  are extracted from the scope claim.
- The same JWT with no `aud` claim (or a wrong audience)
  returns `{ ok: false, code: 'BAD_AUDIENCE' }`.
- The same JWT with a wrong issuer returns `{ ok: false,
  code: 'BAD_ISSUER' }`.
- An expired JWT returns `{ ok: false, code: 'EXPIRED' }`.
- A JWT with `permissions: ['auth:login']` array claim works
  equivalently.
- Tokens with `scope: 'foo:bar auth:login'` drop the unknown
  `foo:bar` and keep the valid `auth:login`.
- The test signs JWTs locally with `generateKeyPair` from
  `jose`; no real IdP needed.

`packages/auth-oidc/src/__tests__/jwks.test.ts` covers:
- A successful first fetch populates the cache and a second
  call within the TTL window does not re-fetch.
- A `kid` mismatch in the token header clears the cache and
  re-fetches.
- A network error during JWKS fetch returns
  `{ ok: false, code: 'JWKS_ERROR' }`.

This implementation deliberately avoids `fetch`-based JWKS
resolution so the package stays Node-native + bun-compatible.
The `createRemoteJWKSet` from jose handles HTTP via the global
`fetch` (Node 18+), so the dependency footprint stays
`jose` only.

**Step 3.3: oidc-provider.ts**

```ts
import { verifyJwt } from '@dt/auth-oidc';
import type { AuthProvider, Session } from './provider.js';
import type { OidcConfig } from '@dt/config';
import type { Permission } from '@dt/contracts';

const SESSION_COOKIE = 'dt_oidc_session';

interface OidcProviderOptions {
  config: OidcConfig;
  /** Dev-mode toggle: skip JWT verification and return a built-in mock user. */
  devBypass: boolean;
}

export class OidcAuthProvider implements AuthProvider {
  constructor(private readonly opts: OidcProviderOptions) {}

  async getSession(headers: Headers): Promise<Session | null> {
    if (this.opts.devBypass) return DEMO_SESSION;

    const token = this.readCookie(headers, SESSION_COOKIE);
    if (!token) return null;

    const result = await verifyJwt(token, {
      issuerUrl: this.opts.config.issuerUrl,
      audience: this.opts.config.audience,
      jwksUri: this.opts.config.jwksUri,
    });
    if (!result.ok) return null;
    return {
      subject: result.session.subject,
      displayName:
        (result.session.claims['name'] as string | undefined) ??
        result.session.subject,
      permissions: result.session.permissions,
    };
  }

  // `login` redirects to /api/auth/oidc/start in T4; this is the
  // fallback for callers that hit /api/auth/login directly. When
  // devBypass is on, it returns a 200 with a dev session.
  async login(req: Request): Promise<Response> {
    if (this.opts.devBypass) {
      return new Response(JSON.stringify({ session: DEMO_SESSION }), {
        status: 200,
        headers: {
          'content-type': 'application/json',
          'set-cookie': this.writeCookie(
            SESSION_COOKIE,
            // dev token; not a real JWT, just a marker for the
            // dev-only code path. Real auth never sets this.
            'dev-bypass',
            60 * 60 * 24,
          ),
        },
      });
    }
    return Response.redirect(new URL('/api/auth/oidc/start', req.url), 302);
  }

  async logout(req: Request): Promise<Response> {
    const headers = new Headers({ 'content-type': 'application/json' });
    headers.append(
      'set-cookie',
      this.writeCookie(SESSION_COOKIE, '', 0),
    );
    return new Response(JSON.stringify({ ok: true }), { status: 200, headers });
  }

  requirePermission(session: Session | null, perm: Permission): boolean {
    if (!session) return false;
    return session.permissions.includes(perm);
  }

  private readCookie(headers: Headers, name: string): string | null {
    const raw = headers.get('cookie');
    if (!raw) return null;
    for (const piece of raw.split(';')) {
      const [k, ...v] = piece.trim().split('=');
      if (k === name) return decodeURIComponent(v.join('='));
    }
    return null;
  }

  private writeCookie(name: string, value: string, maxAgeSec: number): string {
    return [
      `${name}=${encodeURIComponent(value)}`,
      'HttpOnly',
      'SameSite=Lax',
      'Path=/',
      `Max-Age=${maxAgeSec}`,
    ].join('; ');
  }
}

const DEMO_SESSION: Session = {
  subject: 'demo',
  displayName: 'Demo User',
  permissions: ['auth:login', 'devices:read', 'devices:write',
                'scene:read', 'scene:write', 'commands:read',
                'commands:write'],
};
```

**Step 3.4: factory**

```ts
// apps/bff/src/auth/index.ts
import type { AppEnv } from '@dt/config';
import { MockAuthProvider } from './mock-provider.js';
import { OidcAuthProvider } from './oidc-provider.js';
import type { AuthProvider } from './provider.js';

export type { AuthProvider, Session } from './provider.js';

export function createAuthProvider(env: AppEnv): AuthProvider {
  if (env.AUTH_PROVIDER === 'oidc') {
    if (!env.oidc) {
      // Handled by readAppEnv validation in T2, but defense-in-depth.
      throw new Error('[bff] AUTH_PROVIDER=oidc requires oidc config');
    }
    const devBypass =
      env.NODE_ENV !== 'production' && process.env['OIDC_ALLOW_DEV_BYPASS'] === 'true';
    return new OidcAuthProvider({ config: env.oidc, devBypass });
  }
  return new MockAuthProvider();
}
```

## Task 4: OIDC routes (BFF)

**Files:**
- Modify: `apps/bff/src/routes/auth.ts`
- Create: `apps/bff/src/routes/oidc.ts`
- Modify: `apps/bff/src/index.ts` (mount the new routes)

**What gets added:** Two OIDC-specific routes
(`/api/auth/oidc/start` and `/api/auth/oidc/callback`) plus
refactored `/api/auth/me`, `/api/auth/login`, and
`/api/auth/logout` that delegate to `authProvider`.

**Step 4.1: PKCE flow**

`/api/auth/oidc/start` generates `code_verifier` + `state`,
stores them in short-lived HttpOnly cookies, and redirects to
the issuer's authorize endpoint. The URL is built from the
issuer's `/.well-known/openid-configuration`'s `authorization_endpoint`.

`/api/auth/oidc/callback`:
1. Validates the `state` cookie against the query `state`
   param (CSRF).
2. Exchanges the `code` + `code_verifier` at the issuer's
   `token_endpoint` (POST).
3. Sets the `id_token` (or `access_token`, configurable) as
   the `dt_oidc_session` cookie (HTTPOnly + SameSite=Lax).
4. Redirects to the SPA's post-login path (default `/`).

Both routes fall back to a 503 + JSON error if the dev IdP
isn't reachable; the deployment doc records the failure mode
clearly so the operator doesn't mistake it for an authn bug.

**Step 4.2: route rewires**

`apps/bff/src/routes/auth.ts`:

```ts
import { Hono } from 'hono';
import type { AuthProvider } from '../auth/index.js';

export function authRoute(auth: AuthProvider) {
  return new Hono()
    .get('/me', async (c) => {
      const session = await auth.getSession(c.req.header());
      return c.json({ session });
    })
    .post('/login', (c) => auth.login(c.req.raw))
    .post('/logout', (c) => auth.logout(c.req.raw));
}
```

## Task 5: `requiresPermission` middleware on existing routes

**Files:**

**What gets added:** A `requiresPermission(perm)` Hono
middleware that pulls the current session via
`auth.getSession(headers)`, calls
`auth.requirePermission(session, perm)`, and returns:

- `401` if `!session`
- `403` if `session && !session.permissions.includes(perm)`
- `next()` otherwise

The middleware is applied per-route in the route files:

| Route | Permission | Reason |
| --- | --- | --- |
| `GET /api/devices` | `devices:read` | V2.1 plan section "BFF route shape" |
| `GET /api/devices/:id` | `devices:read` | same |
| `POST /api/devices/:id/commands` | `commands:write` | V2.1 audit log |
| `GET /api/scene` | `scene:read` | V2.1 plan |
| `POST /api/scene/updates` | `scene:write` | V2.1 plan |

`scripts/smoke-v2.sh` is extended to verify each protected
route returns 401 without a session and 403 with the wrong
permission. The existing happy-path tests pass because the
mock provider's `DEMO_SESSION` carries all permissions.

```ts
// apps/bff/src/middleware/permission.ts
import type { MiddlewareHandler } from 'hono';
import type { AuthProvider, Session } from '../auth/index.js';
import type { Permission } from '@dt/contracts';

export function requiresPermission(
  auth: AuthProvider,
  perm: Permission,
): MiddlewareHandler {
  return async (c, next) => {
    const session = await auth.getSession(c.req.header());
    if (!session) return c.json({ error: 'unauthenticated' }, 401);
    if (!auth.requirePermission(session, perm)) {
      return c.json({ error: 'forbidden', permission: perm }, 403);
    }
    c.set('session', session);
    return next();
  };
}
```

`Session` is stored on `c.var` via `declare module 'hono'`;
the routes that need it read it via `c.get('session')`.

```ts
// packages/app-shell/src/composables/useOIDCStart.ts
import { computed } from 'vue';

export interface OidcStartOptions {
  /** Pulled from a build-time env or runtime config; see T8 docs. */
  authMode: 'mock' | 'oidc';
  bffBaseUrl: string;
}

export function useOIDCStart(opts: OidcStartOptions) {

    opts.authMode === 'oidc'
```

## Task 7: Dev OIDC IdP script

**Files:**
- Create: `scripts/dev-oidc-idp.mjs`
- Modify: `package.json` (add `dev:oidc` and `smoke:oidc`
  scripts)

**What gets added:** A Node 22 script that simulates an OIDC
IdP for development and CI. It is **not** a production
component; production uses a real IdP. CI gates V3.0 by
running `pnpm smoke:oidc` against this script.

**Step 7.1: script outline**

```js
#!/usr/bin/env node
// Dev OIDC IdP. Listens on $DEV_OIDC_PORT (default 9999).
// Exposes:
//   GET  /.well-known/openid-configuration
//   GET  /.well-known/jwks.json
//   GET  /authorize?client_id=...&redirect_uri=...&state=...&code_challenge=...
//   POST /token (application/x-www-form-urlencoded)
//   GET  /userinfo
// Issues RS256-signed JWTs. The matching public JWK is served
// at /jwks.json. The "user" is selected via a query param
// (`?as=user@example.com`) on /authorize; the script maps it
// to a fixed permission set.
```

The script is self-contained (no npm deps beyond `jose` for
signing). Signing keys are ephemeral — generated at startup
and held in process memory.

**Step 7.2: package.json scripts**

```json
{
  "scripts": {
    "dev:oidc": "node scripts/dev-oidc-idp.mjs",
    "smoke:oidc": "bash scripts/smoke-oidc.sh"
  }
}
```

**Step 7.3: smoke**

`scripts/smoke-oidc.sh` (paired with `dev-oidc-idp.mjs`):
boots the dev IdP, sets `OIDC_ALLOW_DEV_BYPASS=false`,
`AUTH_PROVIDER=oidc`, `OIDC_ISSUER_URL=http://localhost:9999`,
hits `/authorize?as=viewer`, extracts the JWT from the
callback, calls `/api/auth/me`, asserts the session carries
the right permissions, calls `GET /api/devices` with the JWT
cookie, asserts 200.
