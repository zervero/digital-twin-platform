# V2.2 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use
> superpowers:executing-plans to implement this plan task-by-task.
> Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship V2.2 = Track B (Plugin runtime) from
`docs/plans/v2-overview.md`. Give `@dt/plugin-runtime` a real
manifest validator and a plugin registry, and let `@dt/app-shell`
host extension points that a third-party plugin can opt into
without forking the platform.

**Why now:** V2.1 just shipped the auth + permission model
(`@dt/contracts` `Role` / `Permission` / `usePermission`), which is
exactly the gating mechanism plugins need. Shipping plugins on
top of an ad-hoc auth model would have forced a redo of the
permission union in V3. The realtime and observability surfaces
from V2.0 give plugins a sane place to subscribe to device
updates and emit structured logs.

**Architecture:** Keep the V1/V2 boundary graph intact. The
runtime stays in `@dt/plugin-runtime`; the app shell hosts the
registry; a host app passes a list of plugin registrations into
`bootstrapAppShell` and the shell activates them after auth
hydration. Plugins declare which `Permission`s they need in
their manifest; activation is rejected if the current user
lacks any of them.

**Scope — IN:**
- `@dt/plugin-runtime`:
  - `PluginManifest` validator (manual, zero new deps) that
    catches malformed ids, non-semver versions, and unknown
    permissions.
  - `PluginRegistry` class with a per-plugin state machine
    (`inactive → activating → active → deactivating → errored`),
    `activateAll()` / `deactivateAll()`, and a state-change
    stream.
  - A `PluginExtension` shape: a plugin declares one or more of
    `ui-panel` (renders into a slot in `AppShell`),
    `menu-item` (renders into a top-toolbar menu), and
    `event-subscriber` (subscribes to `@dt/realtime`).
  - Re-export `Permission` from `@dt/contracts` so the manifest's
    `permissions` field narrows against the same union the BFF
    enforces. The V1 hardcoded `PluginPermission` type is
    removed.
- `@dt/app-shell`:
  - `usePluginStore` Pinia store wrapping the registry; reads
    the registered UI panels as a reactive list.
  - `usePluginMenu` composable returning the menu items.
  - An `extension-point` slot in `AppShell.vue` (sidebar bottom
    for panels, top toolbar for menu items) so plugins can
    render without touching the shell template.
  - `bootstrapAppShell` accepts an optional `plugins: readonly
    PluginRegistration[]` argument; activates them after auth
    hydration so the permission gate has current-user state.
- Sample: `apps/web/src/plugins/hello/` ships a "Hello, <user>"
  panel that requires `auth:login` and proves the wiring
  end-to-end.

**Scope — OUT:**
- Marketplace data model, search, install, and versioning
  (V3, spec #6 in the V1 dev doc §9).
- Tenant / workspace scoping of plugins (V3, spec #7).
- Audit log of plugin activations (V3, spec #8).
- Real sandboxing (iframe, worker, signed modules, CSP).
  V2.2 uses same-origin static imports and a contract-based
  trust model: the operator reviewed the plugin before adding
  it to the host's static list. A follow-up ADR (V3) decides
  whether to bolt on a sandbox.
- Hot reload, dev-time plugin re-activation, plugin settings UI.
- Permission enforcement on existing BFF routes
  (`/api/devices`, `/api/scene`, etc.) — still a V2.1
  known limitation; the plugin's own BFF endpoints (if any) are
  a future task.
- AI-agent plugin surface (`provide:commands`). The intent
  union is V3; V2.2 ships the structural slots but no AI
  consumer.

**Tech Stack:** Same as V2.1. No new runtime deps. Vitest 2,
Vue 3 + Pinia, TypeScript 5.6, Hono 4, Node `>=22.17.1`.

## Open design questions (decided in this plan)

1. **Loading model.** Static (build-time) vs dynamic (URL-based
   `import()`) vs npm package convention. **Decision: static.**
   The host app passes a `readonly PluginRegistration[]` into
   `bootstrapAppShell`; each registration is already imported
   at the call site (a `import { helloPlugin } from
   './plugins/hello.js'`). Dynamic URL loading is a V3 problem
   because it pulls in signed modules, CSP, and a registry
   storage story that the platform doesn't have yet.

2. **Sandboxing.** Real sandboxing requires an iframe + `postMessage`
   or a Web Worker + `comlink`. **Decision: same-origin module
   load, contract-based trust.** A plugin is JS that runs in
   the same realm as the app. The trust model is "the operator
   reviewed this code and added it to the static list." A V3
   ADR will revisit if and when the marketplace ships.

3. **Extension surface.** What can a plugin actually do?
   **Decision: 3 hooks, all opt-in.**
   - `ui-panel`: renders a Vue component into a sidebar slot.
   - `menu-item`: appears in the top toolbar.
   - `event-subscriber`: subscribes to `@dt/realtime` with a
     callback for typed events.
   The `provide:commands` hook (for the V3 AI agent) is
   reserved in the type but not surfaced in V2.2.

4. **Permission model.** Reuse the V2.1 `Permission` union from
   contracts. **Decision: yes.** Drop the V1
   `PluginPermission` type in `@dt/plugin-runtime`; the manifest
   imports `Permission` from `@dt/contracts`. The validator
   rejects manifests that list a permission the union doesn't
   include. Activation also checks the current user via
   `permissionsFor(user.roles)`; if any required permission is
   missing, the plugin goes to `errored` with a `PERMISSION_DENIED`
   reason and the host UI shows a "plugin blocked by permission"
   notice in its slot.

5. **Activation lifecycle state machine.** A single plugin has
   `inactive | activating | active | deactivating | errored`.
   **Decision: yes.** This matches how a typical app plugin
   model works (VS Code, Obsidian, etc.) and gives the host UI
   something concrete to render while activation runs
   (loading / errored / ok). `activateAll()` runs activations
   sequentially; `deactivateAll()` reverses. Failures during
   activation do not block the other plugins.

6. **Multiple plugins per host.** A registry holds many
   plugins. **Decision: yes.** `usePluginStore` exposes
   `panels: ComputedRef<PluginPanelEntry[]>` so the host
   template can render them in declaration order with a stable
   key. A disable toggle is a host-level concern, not a V2.2
   surface.

7. **Demo plugin.** Ship a "hello" panel to prove the wiring
   end-to-end. **Decision: yes.** Lives in
   `apps/web/src/plugins/hello/` so the integration is
   visible in the host's `main.ts`. The panel says "Hello,
   {user.displayName}" and shows the plugin's declared
   permissions; the host can turn it off by removing one
   line.

8. **Smoke test scope.** What does the V2.2 smoke assert?
   **Decision: extend `scripts/smoke-v2.sh` with a check that
   the sample plugin's manifest validates against
   `validatePluginManifest`** (a node one-liner that imports
   the runtime via the workspace). The full plugin activation
   path is exercised by the `app-shell` unit tests
   (T5) and by manual `pnpm dev`; the smoke stays at the
   "the wiring typechecks and the manifest is valid" level
   because the BFF is not in the plugin loop in V2.2.

**Step 1.2: Implement the validator**

Create `packages/plugin-runtime/src/manifest.ts`:

```ts
/**
 * Plugin manifest validation.
 *
 * The validator is a pure function over `unknown`. It never
 * throws and never reads the network / filesystem / env. The
 * BFF and the app shell both call it; the registry calls it
 * before admitting a registration.
 *
 * The permission check uses the V2.1 `Permission` union from
 * `@dt/contracts` as the source of truth. Adding a permission
 * to the union is enough to make it acceptable in a manifest.
 */

import type { Permission } from '@dt/contracts';

const ID_RE = /^[a-z][a-z0-9-]*$/;
const SEMVER_RE = /^\d+\.\d+\.\d+(?:-[0-9A-Za-z-.]+)?(?:\+[0-9A-Za-z-.]+)?$/;
const KNOWN_PERMISSIONS: ReadonlySet<Permission> = new Set<Permission>([
  'device:read',
  'device:write',
  'scene:read',
  'scene:write',
  'command:send',
  'auth:login',
]);

export interface PluginManifestError {
  code:
    | 'INVALID_ID'
    | 'INVALID_VERSION'
    | 'UNKNOWN_PERMISSION'
    | 'MISSING_FIELD'
    | 'INVALID_TYPE';
  message: string;
  field?: string;
}

export interface PluginManifest {
  id: string;
  name: string;
  version: string;
  vendor: string;
  description?: string;
  entry?: string;
  permissions: readonly Permission[];
}

function asObject(v: unknown): Record<string, unknown> | null {
  return typeof v === 'object' && v !== null ? (v as Record<string, unknown>) : null;
}

function isString(v: unknown): v is string {
  return typeof v === 'string';
}

export type ValidateResult =
  | { ok: true; manifest: PluginManifest }
  | { ok: false; errors: PluginManifestError[] };

export function validatePluginManifest(input: unknown): ValidateResult {
  const errors: PluginManifestError[] = [];
  const obj = asObject(input);
  if (!obj) {
    return { ok: false, errors: [{ code: 'INVALID_TYPE', message: 'manifest must be an object' }] };
  }

  const id = obj.id;
  if (!isString(id) || id.length === 0) {
    errors.push({ code: 'MISSING_FIELD', message: 'id is required', field: 'id' });
  } else if (!ID_RE.test(id)) {
    errors.push({
      code: 'INVALID_ID',
      message: 'id must start with a-z and contain only a-z, 0-9, and `-`',
      field: 'id',
    });
  }

  const name = obj.name;
  if (!isString(name) || name.length === 0) {
    errors.push({ code: 'MISSING_FIELD', message: 'name is required', field: 'name' });
  }

  const version = obj.version;
  if (!isString(version) || version.length === 0) {
    errors.push({ code: 'MISSING_FIELD', message: 'version is required', field: 'version' });
  } else if (!SEMVER_RE.test(version)) {
    errors.push({
      code: 'INVALID_VERSION',
      message: 'version must be a semver string (e.g. 1.2.3 or 1.2.3-beta.1)',
      field: 'version',
    });
  }

  const vendor = obj.vendor;
  if (!isString(vendor) || vendor.length === 0) {
    errors.push({ code: 'MISSING_FIELD', message: 'vendor is required', field: 'vendor' });
  }

  const description = obj.description;
  if (description !== undefined && !isString(description)) {
    errors.push({
      code: 'INVALID_TYPE',
      message: 'description must be a string when present',
      field: 'description',
    });
  }

  const entry = obj.entry;
  if (entry !== undefined && !isString(entry)) {
    errors.push({
      code: 'INVALID_TYPE',
      message: 'entry must be a string when present',
      field: 'entry',
    });
  }

  const permissions = obj.permissions;
  const perms: Permission[] = [];
  if (!Array.isArray(permissions)) {
    errors.push({
      code: 'INVALID_TYPE',
      message: 'permissions must be an array of strings',
      field: 'permissions',
    });
  } else {
    permissions.forEach((p, i) => {
      if (!isString(p)) {
        errors.push({
          code: 'INVALID_TYPE',
          message: 'permission must be a string',
          field: `permissions[${i}]`,
        });
        return;
      }
      if (!KNOWN_PERMISSIONS.has(p as Permission)) {
        errors.push({
          code: 'UNKNOWN_PERMISSION',
          message: `unknown permission: ${p}`,
          field: `permissions[${i}]`,
        });
        return;
      }
      perms.push(p as Permission);
    });
  }

  if (errors.length > 0) return { ok: false, errors };

  return {
    ok: true,
    manifest: {
      id: id as string,
      name: name as string,
      version: version as string,
      vendor: vendor as string,
      permissions: perms,
      ...(description !== undefined ? { description: description as string } : {}),
      ...(entry !== undefined ? { entry: entry as string } : {}),
    },
  };
}
```

**Files:**

**Interfaces produced:**
- `PluginPanel`:
  `{ id: string; title: string; component: Component }`.
- `PluginMenuItem`:
  `{ id: string; label: string; onSelect: () => void }`.
- `PluginEventSubscriber`:
  `{ eventTypes?: readonly DigitalTwinEventType[];
  handle: (event: DigitalTwinEvent) => void | Promise<void> }`.
- `PluginExtension`: discriminated union
  `{ kind: 'ui-panel'; panel: PluginPanel }
  | { kind: 'menu-item'; item: PluginMenuItem }
  | { kind: 'event-subscriber'; subscriber: PluginEventSubscriber }`.
- `PluginRegistration`:
  `{ manifest: PluginManifest; activate: (ctx) => Promise<PluginExtension[]>;
  deactivate?: (ctx) => Promise<void> }`.
- `PluginState`:
  `'inactive' | 'activating' | 'active' | 'deactivating' | 'errored'`.
- `PluginStateChange`:
  `{ pluginId: string; prev: PluginState; next: PluginState;
  error?: PluginActivationError }`.
- `PluginActivationError`:
  `{ code: 'PERMISSION_DENIED' | 'ACTIVATION_FAILED';
  message: string; missingPermissions?: readonly Permission[] }`.
- `PluginRegistry` interface:
  `register(reg)`, `unregister(id)`, `activateAll(granted)`,
  `deactivateAll()`, `list()`, `get(id)`, `subscribe(fn)`.
- `createPluginRegistry(): PluginRegistry` — factory.

The activation context `ctx` provides:
- `permissions: readonly Permission[]` — the user's current
  effective permissions (already filtered by `useAuthStore`).
- `api: ApiClient` — the V2.1 client, so a plugin can call
  `api.getDevices()` etc.
- `events: RealtimeStream` — the V2.0 stream, so a plugin can
  `events.subscribe(...)` for typed events.

Wait — this is a contract shift. The runtime currently only
imports `@dt/contracts`. The registry's `ctx` shape needs
`ApiClient` (from `@dt/api-client`) and `RealtimeStream`
(from `@dt/realtime`). Adding those imports to
`@dt/plugin-runtime` would change the workspace boundary graph
(plugin-runtime → api-client / realtime), which is not in the
allowed graph.

**Resolution:** the activation context is a structural
interface owned by the runtime; the host app constructs a
concrete `PluginContext` and passes it in. The runtime
declares the types it needs (e.g. a `RealtimeStreamLike` with
just `subscribe`), the host app satisfies them with the
concrete class. The runtime never imports `api-client` or
`realtime`.

```ts
 *
 * An extension is the *thing* a plugin contributes to the host:
 * a panel that renders into a slot, a menu item that appears
 * in the toolbar, or an event subscriber that runs alongside
 * the realtime stream. The discriminated union is the
 * extension point surface; adding a new kind is a breaking
 * change to plugins that return the old shape.
 *
 * The shapes are intentionally minimal. Plugins that need
 * richer state should keep it in their own module and pass
 * callbacks in via the activation context, not via
 * extension fields.
 */

import type { Component } from 'vue';

import type { DigitalTwinEvent, DigitalTwinEventType } from '@dt/contracts';

export interface PluginPanel {
  /** Stable id; collisions in the host render as a warning. */
  id: string;
  /** Human-readable title shown in the panel header. */
  title: string;
  /** The Vue component to render. */
  component: Component;
}

export interface PluginMenuItem {
  id: string;
  label: string;
  onSelect: () => void;
}

export interface PluginEventSubscriber {
  /** When omitted, the subscriber receives every event. */
  eventTypes?: readonly DigitalTwinEventType[];
  handle: (event: DigitalTwinEvent) => void | Promise<void>;
}

export type PluginExtension =
  | { kind: 'ui-panel'; panel: PluginPanel }
  | { kind: 'menu-item'; item: PluginMenuItem }
  | { kind: 'event-subscriber'; subscriber: PluginEventSubscriber };
```

Create `packages/plugin-runtime/src/registry.ts`:

```ts
/**
 * Plugin registry.
 *
 * Owns the per-plugin state machine and the activation flow.
 * `activateAll` runs sequentially: one plugin's failure does
 * not block the next. `deactivateAll` reverses in the
 * reverse-order they were activated (LIFO), which is what
 * plugins with a teardown expectation want.
 *
 * The activation context is a structural type. The host
 * constructs the concrete `ApiClient` and `RealtimeStream`
 * bindings; the runtime only declares the shape it needs.
 */

import type { DigitalTwinEvent, Permission } from '@dt/contracts';

import type { PluginExtension } from './extensions.js';
import type { PluginManifest } from './manifest.js';

export type PluginState =
  | 'inactive'
  | 'activating'
  | 'active'
  | 'deactivating'
  | 'errored';

export interface PluginActivationError {
  code: 'PERMISSION_DENIED' | 'ACTIVATION_FAILED';
  message: string;
  missingPermissions?: readonly Permission[];
}

export interface PluginRegistration {
  manifest: PluginManifest;
  activate: (ctx: PluginContext) => Promise<PluginExtension[]>;
  deactivate?: (ctx: PluginContext) => Promise<void>;
}

/**
 * Structural activation context. The runtime does not import
 * `@dt/api-client` or `@dt/realtime`; the host satisfies this
 * shape with the real implementations.
 */
export interface PluginContext {
  grantedPermissions: readonly Permission[];
  subscribe: (fn: (event: DigitalTwinEvent) => void) => () => void;
}

export interface PluginStateChange {
  pluginId: string;
  prev: PluginState;
  next: PluginState;
  error?: PluginActivationError;
}

export interface PluginRegistryEntry {
  manifest: PluginManifest;
  state: PluginState;
  extensions: readonly PluginExtension[];
  error?: PluginActivationError;
}

export interface PluginRegistry {
  register(reg: PluginRegistration): void;
  unregister(id: string): void;
  activateAll(ctx: PluginContext): Promise<void>;
  deactivateAll(): Promise<void>;
  list(): readonly PluginRegistryEntry[];
  get(id: string): PluginRegistryEntry | undefined;
  subscribe(fn: (change: PluginStateChange) => void): () => void;
}

function missingPermissions(
  required: readonly Permission[],
  granted: readonly Permission[],
): Permission[] {
  const set = new Set<Permission>(granted);
  return required.filter((p) => !set.has(p));
}

export function createPluginRegistry(): PluginRegistry {
  const entries = new Map<string, PluginRegistryEntry & { reg: PluginRegistration }>();
  const listeners = new Set<(c: PluginStateChange) => void>();

  function emit(change: PluginStateChange): void {
    for (const fn of listeners) fn(change);
  }

  function setState(
    id: string,
    next: PluginState,
    patch: Partial<PluginRegistryEntry> = {},
  ): void {
    const cur = entries.get(id);
    if (!cur) return;
    const prev = cur.state;
    const merged = { ...cur, ...patch, state: next };
    entries.set(id, merged);
    emit({ pluginId: id, prev, next, error: merged.error });
  }

  return {
    register(reg) {
      if (entries.has(reg.manifest.id)) {
        throw new Error(`plugin already registered: ${reg.manifest.id}`);
      }
      entries.set(reg.manifest.id, {
        manifest: reg.manifest,
        state: 'inactive',
        extensions: [],
        reg,
      });
    },

    unregister(id) {
      const cur = entries.get(id);
      if (!cur) return;
      if (cur.state === 'active' || cur.state === 'activating') {
        // Best-effort: a non-awaited deactivation so unregister
        // stays synchronous. The host is expected to call
        // deactivateAll() first if it cares.
        void cur.reg.deactivate?.({
          grantedPermissions: [],
          subscribe: () => () => undefined,
        });
      }
      entries.delete(id);
    },

    async activateAll(ctx) {
      for (const [id, cur] of entries) {
        if (cur.state !== 'inactive') continue;
        const missing = missingPermissions(cur.manifest.permissions, ctx.grantedPermissions);
        if (missing.length > 0) {
          setState(id, 'errored', {
            error: {
              code: 'PERMISSION_DENIED',
              message: `plugin requires missing permission(s): ${missing.join(', ')}`,
              missingPermissions: missing,
            },
          });
          continue;
        }
        setState(id, 'activating');
        try {
          const exts = await cur.reg.activate(ctx);
          setState(id, 'active', { extensions: exts });
        } catch (err) {
          setState(id, 'errored', {
            error: {
              code: 'ACTIVATION_FAILED',
              message: err instanceof Error ? err.message : String(err),
            },
          });
        }
      }
    },

    async deactivateAll() {
      const ordered = [...entries.values()].filter(
        (e) => e.state === 'active',
      );
      // LIFO teardown.
      for (const cur of [...ordered].reverse()) {
        setState(cur.manifest.id, 'deactivating');
        try {
          await cur.reg.deactivate?.({
            grantedPermissions: [],
            subscribe: () => () => undefined,
          });
          setState(cur.manifest.id, 'inactive', { extensions: [] });
        } catch (err) {
          setState(cur.manifest.id, 'errored', {
            error: {
              code: 'ACTIVATION_FAILED',
              message: err instanceof Error ? err.message : String(err),
            },
          });
        }
      }
    },

    list() {
      return [...entries.values()].map((e) => ({
        manifest: e.manifest,
        state: e.state,
        extensions: e.extensions,
        error: e.error,
      }));
    },

    get(id) {
      const e = entries.get(id);
      if (!e) return undefined;
      return {
        manifest: e.manifest,
        state: e.state,
        extensions: e.extensions,
        error: e.error,
      };
    },

    subscribe(fn) {
      listeners.add(fn);
      return () => listeners.delete(fn);
    },
  };
}
```

```ts
/**
 * usePluginStore — Pinia store wrapping the plugin registry.
 *
 * The registry is held as a `ShallowRef`; the entries, panels,
 * and menu items are derived `ComputedRef`s that re-run when
 * the registry's `subscribe` callback fires.
 */

import { defineStore } from 'pinia';
import { computed, ref, shallowRef } from 'vue';

import type { Permission } from '@dt/contracts';
import {
  type PluginContext,
  type PluginRegistry,
  type PluginRegistryEntry,
  createPluginRegistry,
  type PluginPanel,
  type PluginMenuItem,
} from '@dt/plugin-runtime';

import { ApiClientKey } from './api-store.js';

export const usePluginStore = defineStore('dt:plugins', () => {
  // The host injects the api client at bootstrap. The store
  // reads it lazily so the registry's `activate` callbacks
  // can call `api.getDevices()` etc.
  const api = inject(ApiClientKey);

  const registry = shallowRef<PluginRegistry>(createPluginRegistry());
  const entriesRaw = ref<PluginRegistryEntry[]>([]);

  function setRegistry(r: PluginRegistry): void {
    registry.value = r;
    // Replay current snapshot.
    entriesRaw.value = [...r.list()];
  }

  function rebuild(): void {
    entriesRaw.value = [...registry.value.list()];
  }

  // Track the subscription so hot-reloads don't leak.
  let unsubscribe: (() => void) | null = null;
  function track(): void {
    unsubscribe?.();
    unsubscribe = registry.value.subscribe(() => rebuild());
  }
  track();

  const entries = computed(() => entriesRaw.value);

  const panels = computed<PluginPanel[]>(() => {
    const out: PluginPanel[] = [];
    for (const e of entriesRaw.value) {
      if (e.state !== 'active') continue;
      for (const ext of e.extensions) {
        if (ext.kind === 'ui-panel') out.push(ext.panel);
      }
    }
    return out;
  });

  const menuItems = computed<PluginMenuItem[]>(() => {
    const out: PluginMenuItem[] = [];
    for (const e of entriesRaw.value) {
      if (e.state !== 'active') continue;
      for (const ext of e.extensions) {
        if (ext.kind === 'menu-item') out.push(ext.item);
      }
    }
    return out;
  });

  async function activateAll(granted: readonly Permission[]): Promise<void> {
    if (!api) {
      throw new Error('[plugin-store] ApiClient not provided. Call provideApiClient() first.');
    }
    const ctx: PluginContext = {
      grantedPermissions: granted,
      subscribe: (fn) => {
        // The host wires this to the realtime stream in T4. In
        // the meantime, the helper is a no-op so plugins that
        // don't need events still typecheck.
        void fn;
        return () => undefined;
      },
    };
    await registry.value.activateAll(ctx);
    rebuild();
  }

  async function deactivateAll(): Promise<void> {
    await registry.value.deactivateAll();
    rebuild();
  }

  return {
    entries,
    panels,
    menuItems,
    setRegistry,
    activateAll,
    deactivateAll,
  };
});
```

### Task 6: Extend the V2 smoke with manifest validation

**Files:**

**What gets added:** a small node one-liner that imports
`@dt/plugin-runtime`'s `validatePluginManifest` and runs it

to re-export something" regressions.

```bash
# Exercise the V2.2 plugin manifest validator. The runtime
# lives in the workspace's `packages/plugin-runtime`. We
# import it via tsx (already a workspace dep) so we don't
# need a build step in the smoke.

import { validatePluginManifest } from '@dt/plugin-runtime';
const r = validatePluginManifest({
  id: 'hello',
  name: 'Hello Plugin',
  version: '1.0.0',

"
```
